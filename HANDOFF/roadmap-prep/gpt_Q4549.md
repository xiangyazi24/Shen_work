# What `SourceFromSolutionEnvelopePass` gives, and what it does not give

`SourceFromSolutionEnvelopePass S` is the nonlinear spatial-envelope pass:

```lean
def SourceFromSolutionEnvelopePass
    {p : CM2Params} {u₀ : intervalDomainPoint → ℝ}
    (S : ConjugateMildSolutionData p u₀) : Type :=
  ∀ {m : ℕ} {c T' : ℝ}, 0 < c → T' < S.T → 1 ≤ m →
    WindowCoefficientEnvelope m c T'
      (fun σ n => cosineCoeffs (intervalDomainLift (S.u σ)) n) →
    WindowSourceEnvelope (m - 1) c T'
      (bFormSourceCoeffs p S.u)
```

That pass is a major ingredient for `hsrcB`, because it supplies the source coefficient envelopes that feed the spectral ladder / pass-2 summability / time-regularity chain. But it is **not itself** a `DuhamelSourceTimeC1` and it does not include the restart representation or resolver witness.

The shortest dependency picture is:

```lean
SourceFromSolutionEnvelopePass S
  + base WCE/WSE and heat-gain machinery
  + source coefficient time-continuity / derivative bounds
  ──> hsrcB : DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u)
```

Then `hsrcB` is only the **first** field of `BFormMildSpectralLeaves`.

# Remaining leaves after `SourceFromSolutionEnvelopePass`

## 1. `hsrcB` — produced, but not by the pass alone

Needed type:

```lean
DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u)
```

Status: produced once the source ladder plus time-C¹ coefficient machinery is fully wired. The `SourceFromSolutionEnvelopePass` is the spatial/decay engine for this, but the final `DuhamelSourceTimeC1` also needs the modewise `HasDerivAt`, `hadotcont`, summable envelope, and derivative-bound fields.

If Codex has already assembled the “source-time-C¹” object, this leaf is done. If it only has `SourceFromSolutionEnvelopePass`, this leaf is still one wrapper/assembler away.

## 2. `hB_restart` — still a separate leaf

Needed type:

```lean
RestartRepresentation S
```

Expanded:

```lean
∀ t₀, 0 < t₀ → t₀ < S.T →
  ∀ᶠ s in 𝓝 t₀, ∀ y : intervalDomainPoint,
    S.u s y =
      ∑' n,
        localRestartCoeff
          (cosineCoeffs (intervalDomainLift (S.u (t₀ / 2))))
          (fun σ n => bFormSourceCoeffs p S.u (t₀ / 2 + σ) n)
          (s - t₀ / 2) n * cosineMode n y.1
```

Status: **not produced by `SourceFromSolutionEnvelopePass`**. It is the restart cosine representation / Duhamel representation leaf. The file uses it twice:

1. To build `hPdeAgreement` via `hasBFormSpectralPdeAgreement_of_restart`.
2. To derive per-slice cosine realization through `restartSliceCoeff_realization` in the shared positive-time regularity layer.

So if Codex has a separate restart-representation producer, this is done; otherwise it remains open.

## 3. `hlogData` — separate Fourier-data leaf

Needed type:

```lean
∀ t, 0 < t → t < S.T → LogisticCosineFourierData p S.u t
```

Status: separate. It should be produced by the logistic weak-H² / coefficient-decay adapter, not by `SourceFromSolutionEnvelopePass` as such. It is consumed by `hasBFormSpectralPdeAgreement_of_restart`.

If the current fork has the logistic weak-H²/time-C¹ producer wired, this is done. Otherwise it is still a required field in `BFormMildSpectralLeaves`.

## 4. `hchemData` — separate chem-div Fourier-data leaf

Needed type:

```lean
∀ t, 0 < t → t < S.T →
  ChemDivCosineFourierData p (S.u t)
    (coupledChemicalConcentration p S.u t)
```

Status: separate. This is where the chem-div source weak-H² witness / quadratic decay / zero-mode data feed the Fourier-data structure. It is not automatically present just because `SourceFromSolutionEnvelopePass` exists.

If the fork’s weak-H² witness plus decay producer has been packaged into `ChemDivCosineFourierData`, this leaf is done. If only `WindowSourceEnvelope 2` exists, this still needs the Fourier-data constructor.

## 5. `hResolverWitness` — still required

Needed type:

```lean
ResolverSourceWitnessFromMild p S
```

Status: still a leaf of `BFormMildSpectralLeaves`. The final assembler converts it internally using:

```lean
hResolverData := hResolverData_of_sourceWitness H.hResolverWitness
```

This is the `hResolverWitness` object the question calls out. It is **not** discharged by `SourceFromSolutionEnvelopePass`. It must be produced separately from the resolver/mild-source witness machinery.

The good news: `hResolverPos` is not a remaining user-supplied leaf in this reduced bundle. The file comments say resolver strict positivity is “free” from the conjugate mild solution fields, while the actual reduced bundle asks only for `hResolverWitness`; `spectralFrontier_of_leaves` uses the witness to build `hResolverData`.

# Bottom line

`bFormMildSpectralBootstrapData_of_leaves` is the final direct closer **after** the five-field leaf bundle exists. The remaining obligations between `SourceFromSolutionEnvelopePass` and the final `HSpectral` field are:

1. Produce `hsrcB` from the source ladder/time-C¹ machinery.
2. Produce `hB_restart` from the B-form restart/Duhamel representation.
3. Produce `hlogData` from the logistic Fourier-data adapter.
4. Produce `hchemData` from the chem-div Fourier-data adapter.
5. Produce `hResolverWitness` from the resolver source witness machinery.

No extra leaf remains after `BFormMildSpectralLeaves` is built; at that point the HSpectral field is a pure one-line application of `bFormMildSpectralBootstrapData_of_leaves`.
