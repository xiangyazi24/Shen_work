## The real cycle risk is layer direction, not these files

The dangerous pattern is a new PDE-layer file importing a Paper2-layer file, followed later by Paper2 importing that PDE file. There is already precedent for this kind of layer tension in the codebase: IntervalChemDivTimeDerivative.lean lives under ShenWork/PDE but imports Paper2 files such as IntervalPicardLimitK1.

That can compile if the import graph never returns to it, but it is fragile. For the new files, keep this discipline:

```plain text
Mathlib / PDE primitives
  ↓
low Paper2 atoms: parity, cosine IBP, multiplier algebra, flux C2/C3 bounds
  ↓
producer files: weak-H2, source pass, adot bound
  ↓
HSpectral / V6 assembly / headline
```

Do not let a low primitive file import IntervalMildPositiveTimeRegularityV6, IntervalChiNegV6Assembly, IntervalTruncatedStrictPositivityProducerV6, or any final assembly file. That would create a real cycle once the final assembly imports the primitive.

# 3. Import-closure risk

This is the biggest concrete hazard I found.

Exact module-name searches for visible new files returned only their own files, not downstream imports:

```plain text
ChemFluxCosineDivergence        -> only IntervalChemFluxCosineDivergence.lean
TruncatedFluxC3Bounds           -> only IntervalTruncatedFluxC3Bounds.lean
LogisticNemytskiiC2             -> only IntervalLogisticNemytskiiC2.lean
EllipticMultiplierBound         -> only IntervalEllipticMultiplierBound.lean
```

That means these modules appear to be landed but not wired into the source-pass / HSpectral / V6 headline closure yet. They may compile individually, but they will not affect the final theorem unless a downstream file imports them.

The missing-or-not-visible files are an even stronger import-closure risk. If these are intended to be part of the final closure, the target branch must contain them under exact module paths matching their import names.

The final cold gate should not be “build the new files.” It must be:

```bash
lake clean
lake build ShenWork.Paper2.IntervalDomainChiNonposHeadline
```

or whatever module owns the final χ₀≤0 headline theorem.

Before that, run a module-name import closure check conceptually equivalent to:

```bash
rg -n 'IntervalChemFluxCosineDivergence|IntervalEvenDerivParity|IntervalSolutionEvenRepresentative|IntervalChemDivSourceWeakH2Producer|IntervalChemDivAdotUniformBound|IntervalUniformTruncatedMapCertificateDatum|IntervalTruncatedFluxC3Bounds|IntervalTruncatedLogisticLipschitz|IntervalLogisticNemytskiiC2|IntervalEllipticMultiplierBound' ShenWork --glob '*.lean'
```

Expected result:

- each file appears at least once in an import ... line from a downstream producer or assembly module;

- absent files do not appear as imports anywhere;

- no file appears only in its own declaration comments.

If a module appears only in its own file, it is outside the final theorem’s import closure and may silently remain uncompiled by the headline build.

## IntervalTruncatedFluxC3Bounds.lean

Looks directionally safe: it imports C2 and adds C3 estimates.

Hazard: this file is only a pointwise algebra bound. It does not by itself prove the derivative tower for the actual Q; the weak-H2/source-pass producer still has to supply the actual Q''' expression and the hypotheses u ∈ C³, R ∈ C⁴, etc. That is expected, but it must be in the wiring.

## IntervalLogisticNemytskiiC2.lean

Safe standalone atom. It imports only Mathlib and states the δ-free logistic second-derivative bound.

Hazard: it takes the second-derivative expression as input. It does not prove the calculus chain rule for u ↦ u^(α+1). That is fine if the source producer already has the expression, but not if the final proof expects this file to derive it.

## IntervalEllipticMultiplierBound.lean

No cycle risk, but it carries a mathematical integration warning in its own header: the two-parameter denominator μ + νλ does not match the currently compiled resolver denominator μ + λ; the header says the compiled resolver uses p.μ + λ_k with p.ν as a source prefactor.

This is a high-value warning. Do not wire the two-parameter lemma into the current resolver unless the resolver operator is actually generalized to μ - ν∂ₓₓ. For current code, prefer the existing one-parameter multiplier route.

What would make the gate fail:

1. A landed file is on a different branch/ref. The build cannot resolve the import, or the file silently never enters the target closure.

1. A new source file imports a higher-level Paper2 assembly module and creates a cycle only after the final assembly imports it.

1. A file contains sorry/admit/custom axiom; then #print axioms exposes sorryAx or a custom axiom only if that declaration is actually used by the final theorem.

Recommended gate sequence:

lake build ShenWork.Paper2.IntervalDomainChiNonposHeadline
