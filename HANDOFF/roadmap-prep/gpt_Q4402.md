# 1. The coordinate ring and the Dedekind-domain question

## 1.1 Lean model

Use nested polynomial rings. The outer variable is x; the variable of the AdjoinRoot polynomial is y:

```javascript
open Polynomial
open scoped Polynomial

variable (K : Type*) [Field K]

-- f : K[x]
variable (f : K[X])

-- g(Y) = Y^2 - f(x), a polynomial over K[x]
def hyperellipticPoly : K[X][X] :=
  X ^ 2 - C f

abbrev AffineRing : Type _ :=
  AdjoinRoot (hyperellipticPoly K f)
```

Here AdjoinRoot means the quotient of K[X][Y] by the principal ideal generated by Y²-f(X).

## 1.2 Mathematical statement

Assume:

```plain text
char K != 2,
f.natDegree = 6,
f.Squarefree.
```

Then A is a Dedekind domain.

The affine plane curve is smooth because the partial derivatives of

There is an important base-ring distinction:

A is smooth as a K-algebra, but the finite map K[X] -> A is not smooth or unramified at the roots of f.

It is a ramified double cover of the x-line. Thus a hoped-for path

```plain text
AdjoinRoot smooth over K[X]
  -> Dedekind
```

is mathematically the wrong route. The Jacobian-criterion smoothness is for the two-variable presentation over K.

## 1.3 What current Mathlib gives automatically

The following pieces are present.

### Free and finite over K[X]

For a monic polynomial g, Mathlib provides:

```javascript
g.Monic.powerBasis'       -- more precisely AdjoinRoot.powerBasis' hg
g.Monic.free_adjoinRoot
g.Monic.finite_adjoinRoot
```

The declarations in Mathlib/RingTheory/AdjoinRoot.lean are:

```javascript
AdjoinRoot.powerBasis' (hg : g.Monic)
Polynomial.Monic.free_adjoinRoot
Polynomial.Monic.finite_adjoinRoot
```

For g=Y²-C f, this gives a rank-two basis {1,y} over K[X].

### Domain

There is no domain instance merely from Squarefree f; one proves that Y²-f is prime/irreducible and then uses

```javascript
AdjoinRoot.isDomain_of_prime
```

This is the pattern used by the elliptic affine-coordinate file. Since f is squarefree and nonconstant, it is not a square in K[X]; consequently Y²-f is irreducible over K(X) and primitive, hence prime in K[X][Y] by Gauss. That irreducibility/prime certificate is a project theorem.

### Noetherian

Once the rank-two finite-module instance is installed, Noetherianity follows from the standard finite-algebra-over-Noetherian infrastructure. K[X] is a PID and hence Noetherian.

### Integral and dimension at most one

The monic equation makes A integral over K[X]. Current Mathlib has

```javascript
Ring.DimensionLEOne.of_isIntegral
```

which transports dimension at most one through an integral domain extension:

```javascript
haveI : Algebra.IsIntegral K[X] A := inferInstance
haveI : Ring.DimensionLEOne A :=
  Ring.DimensionLEOne.of_isIntegral K[X] A
```

The base K[X] has Ring.DimensionLEOne because it is a principal ideal domain.

## 1.4 The missing piece: IsIntegrallyClosed A

Current Mathlib does not contain a theorem whose endpoint is schematically

```javascript
(f.Squarefree) -> IsIntegrallyClosed (AdjoinRoot (Y^2 - C f))
```

nor a general ready-to-apply chain

```plain text
smooth finite-type one-dimensional K-algebra
  -> regular ring
  -> normal ring
  -> IsDedekindDomain.
```

The exact project-level gap is therefore:

```javascript
instance affineRing_isIntegrallyClosed :
    IsIntegrallyClosed (AffineRing K f) := ...
```

Once that exists, the remaining Dedekind packaging is routine. IsDedekindDomain is defined as domain + Noetherian + integrally closed + dimension at most one, and isDedekindDomain_iff exposes those fields.

A compile-shaped endpoint is:

```javascript
noncomputable instance affineRing_isDedekindDomain
    [Fact (Polynomial.Squarefree f)]
    [NeZero (2 : K)] :
    IsDedekindDomain (AffineRing K f) := by
  -- domain, Noetherian, and DimensionLEOne are already installed
  refine (isDedekindDomain_iff
    (AffineRing K f) (FractionRing (AffineRing K f))).2 ?_
  refine ⟨inferInstance, inferInstance, inferInstance, ?_⟩
  intro z hz
  -- this is exactly the normality theorem
  exact (isIntegrallyClosed_iff _).mp
    (inferInstance : IsIntegrallyClosed (AffineRing K f)) hz
```

The exact binder order may need minor adjustment, but the mathematical dependency is accurate.

## 1.5 Two realistic proofs of normality

### Route N1: local DVR proof

Use IsDedekindDomainDvr from

Mathlib/RingTheory/DedekindDomain/Dvr.lean.

It asks for:

```javascript
class IsDedekindDomainDvr A extends IsNoetherian A A where
  is_dvr_at_nonzero_prime :
    ∀ P ≠ ⊥, ∀ _ : P.IsPrime,
      IsDiscreteValuationRing (Localization.AtPrime P)
```

For this quadratic ring, split primes by whether the contraction to K[X] divides f.

- Away from f, the discriminant 4f is a unit and the localization is finite étale over a DVR; its local factors are DVRs.

- At a prime dividing f, squarefreeness says the zero is simple. Locally f=p·u, and the relation y²=p·u makes y a uniformizer. The localization is again a DVR.

This is mathematically elementary but substantial Lean plumbing.

### Route N2: identify the normalization

Let F be the function field of the curve and define

```javascript
abbrev Normalization := FunctionField.ringOfIntegers K F
```

Current Mathlib defines this as

```javascript
integralClosure K[X] F
```

and provides, under the function-field and separability instances:

```javascript
FunctionField.ringOfIntegers.instIsDomain
FunctionField.ringOfIntegers.instIsFractionRing
FunctionField.ringOfIntegers.instIsIntegrallyClosed
FunctionField.ringOfIntegers.instIsDedekindDomain
```

The key current source is Mathlib/NumberTheory/FunctionField.lean; its IsDedekindDomain instance is obtained from

```javascript
IsIntegralClosure.isDedekindDomain K[X] K⟮X⟯ F _
```

This gives a Dedekind ring immediately. But it gives the normalization Abar, not automatically the concrete quotient A.

There is a canonical algebra map

```plain text
A -> Abar
```

because A is integral over K[X]. Proving that map surjective is exactly the assertion that A is normal. Thus the constructor solves the existence of a Dedekind model, but it does not remove the need to identify it with the coordinate ring used by the explicit formulas.

# 3. The orientation correction

## 3.1 Why ClassGroup A × ℤ is not the correct raw quotient

Suppose one first passes to ClassGroup A. For every nonzero function alpha,

```plain text
class((alpha)) = 0 in ClassGroup A.
```

Therefore the proposed relation

becomes merely

inside ClassGroup A × ℤ. This forgets which affine principal divisor was paired with the infinity order. In general it does not reconstruct Pic^0(C).

A noncanonical splitting could turn the resulting extension into a product in special cases, but no such splitting should be built into the definition.

## 3.2 Correct definition before the principal quotient

Let:

```javascript
abbrev F := FractionRing A
abbrev InvFrac :=
  Additive ((FractionalIdeal A⁰ F)ˣ)

abbrev OrientedRaw := InvFrac A × ℤ
```

Construct the infinity valuation as an additive homomorphism on nonzero functions:

```javascript
def ordInfPlus : Additive Fˣ →+ ℤ := ...
```

Convert the principal-ideal monoid homomorphism to additive notation:

```javascript
noncomputable def principalIdealAdd :
    Additive Fˣ →+ InvFrac A :=
  (toPrincipalIdeal A F).toAdditive
```

Then take the product homomorphism:

```javascript
noncomputable def principalOriented :
    Additive Fˣ →+ OrientedRaw A :=
  (principalIdealAdd A).prod ordInfPlus
```

and define:

```javascript
abbrev OrientedClassGroup :=
  OrientedRaw A ⧸ (principalOriented A ordInfPlus).range
```

The quotient automatically inherits an AddCommGroup instance from QuotientAddGroup.

The exact spelling of the multiplicative-to-additive conversion may require a small coercion annotation at your pinned revision, but the stable ingredients are:

```plain text
Additive
MonoidHom.toAdditive
AddMonoidHom.prod
AddMonoidHom.range
QuotientAddGroup
```

The relevant current file is

## 4.1 Coordinate ring

```javascript
  FractionRing W.CoordinateRing
```


There is no IsDedekindDomain instance for this coordinate ring in that file.

## 4.2 Point ideal

For an affine nonsingular point (x,y), it defines the integral ideal

and then constructs a nonzero invertible fractional ideal:

```javascript
CoordinateRing.XYIdeal'
```

The crucial implementation is explicit:

```javascript
Units.mkOfMulEqOne
```

is given a candidate inverse, and an ideal multiplication identity proves the product is one. Thus the file never needs the theorem “every nonzero ideal is invertible.”

## 4.3 Map into the class group

The exact group homomorphism is:

```javascript
WeierstrassCurve.Affine.Point.toClass :
  W.Point →+ Additive (ClassGroup W.CoordinateRing)
```

It sends:

```plain text
O       -> 0,
(x,y)   -> ClassGroup.mk W.FunctionField (CoordinateRing.XYIdeal' h).
```

The addition proof is reduced to explicit ideal identities:

```javascript
CoordinateRing.mk_XYIdeal'_neg_mul
CoordinateRing.mk_XYIdeal'_mul_mk_XYIdeal'
```

## 4.4 How the group laws are obtained

The file first defines the geometric addition formula on points. It then proves:

```javascript
Point.toClass_eq_zero
Point.toClass_injective
```

Injectivity uses the norm and a degree obstruction. Finally it proves commutativity and associativity by applying toClass_injective and using the already associative/commutative class group:

```javascript
instance : AddCommSemigroup W.Point
instance : AddCommGroup W.Point
```

So it does not transport a group structure from a class-group equivalence. It uses an injective homomorphism into an existing abelian group as a proof device.

# 6. Bottom line by question

## (1) Is A Dedekind?

Yes mathematically for K=Q and K=L. No, current Mathlib does not infer it from squarefreeness.

Available path:

```plain text
monic AdjoinRoot
 -> finite/free over K[X]
 -> Noetherian and integral
 -> DimensionLEOne.of_isIntegral
 + project theorem IsIntegrallyClosed A
 -> IsDedekindDomain A.
```

The exact obstruction is proving normality. FunctionField.ringOfIntegers K F is the available normalization constructor and already has a Dedekind instance; identifying it with A is the same normality problem.

## (2) Does ClassGroup work directly?

Yes. ClassGroup A is a commutative group for every domain and is the quotient of invertible fractional ideals by principal ideals. Use:

```plain text
toPrincipalIdeal
ClassGroup.mk
ClassGroup.equiv
```

With a Dedekind instance add:

```plain text
FractionalIdeal.mk0
ClassGroup.mk0.
```

## (3) How should the orientation be defined?

Use

```plain text
(Additive InvFrac(A) × Z) / range(principalIdealAdd, ordInfPlus).
```

Do not start from ClassGroup A × Z. Proving the resulting quotient canonically equals Pic^0(C) requires a project-local divisor/product-formula theorem; Mathlib has no ready-made general Picard/Jacobian identification.

## (4) Can the elliptic pattern be followed?

Yes, structurally, and it is probably the shortest path. The exact precedent is:

```plain text
CoordinateRing.XYIdeal'
ClassGroup.mk
Point.toClass
Point.toClass_eq_zero
Point.toClass_injective
```

followed by proving the point group axioms through the injective class-group map. For genus two, replace XYIdeal' by the Mumford ideal and replace ClassGroup by the oriented quotient.

# Final recommendation

For the N18 Route-C build, start with the oriented quotient over the group of invertible fractional ideals and prove explicit invertibility only for the Mumford ideals you use. This matches current Mathlib’s strongest successful precedent and avoids making the entire formalization depend on a missing generic smooth-curve normality theorem.
