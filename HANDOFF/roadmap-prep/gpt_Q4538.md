# 3. Where SourceFromSolutionEnvelopePass actually sits

Its current exact type is:

```javascript
def SourceFromSolutionEnvelopePass
    {p : CM2Params} {u₀ : intervalDomainPoint → ℝ}
    (S : ConjugateMildSolutionData p u₀) : Type :=
  ∀ {m : ℕ} {c T' : ℝ},
    0 < c → T' < S.T → 1 ≤ m →
    WindowCoefficientEnvelope m c T'
      (fun σ n => cosineCoeffs (intervalDomainLift (S.u σ)) n) →
    WindowSourceEnvelope (m - 1) c T'
      (bFormSourceCoeffs p S.u)
```

This is the nonlinear spatial R-engine:

```plain text
solution coefficient order m
  ⟶ source coefficient order m−1.
```

It is upstream of hsrcB; it is not equal to hsrcB, and there is currently no coercion or one-line constructor

```javascript
SourceFromSolutionEnvelopePass S →
  DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u).
```

The same positive-time regularity file explicitly identifies another independent engine:

```javascript
def MildSolutionSliceHasDerivAtTime
    (S : ConjugateMildSolutionData p u₀) : Prop :=
  ∃ udot,
    ... HasDerivAt in time ... ∧
    ... spatial continuity of udot ... ∧
    ... compact-window uniform bound ...
```

That is the temporal L-engine. The intended construction is:

```plain text
bounded-slice base
  + heat/Duhamel envelope gain
  + finite alternating use of SourceFromSolutionEnvelopePass
  ⟶ pass-2 source envelope / eigenvalue-weighted coefficient control
  + MildSolutionSliceHasDerivAtTime
  + logistic and chem-div chain-rule packagers
  ⟶ hsrcB : DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u).
```

The base

```javascript
windowCoefficientEnvelope_zero_of_bound
```

is already proved from S.hbound and S.hcont, and the heat gain is existing infrastructure. The repository roadmap says the atomic derivative/IBP/multiplier estimates for SourceFromSolutionEnvelopePass are landed, but the finite ladder and its final wiring into hsrcB remain the active assembly step.

Accordingly, the honest implication is:

```plain text
SourceFromSolutionEnvelopePass
+ MildSolutionSliceHasDerivAtTime
+ finite ladder/termination wiring
+ source-time-C¹ packagers
⇒ hsrcB.
```

It is not:

```plain text
SourceFromSolutionEnvelopePass ⇒ hsrcB.
```

# 4. Status of every actual leaf

## hsrcB

```javascript
DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u)
```

Status: not yet universally produced.

The major ingredients are isolated:

- spatial engine SourceFromSolutionEnvelopePass;

- temporal engine MildSolutionSliceHasDerivAtTime;

- order-zero coefficient base;

- parabolic gain and restart-coefficient summability engines;

- logistic/chem-div source packagers.

But the current generic-S file still labels the spatial and temporal engines as open leaves, and no theorem in the audited route constructs hsrcB from them end-to-end. Older ...SourceTimeC1On combiners only combine already supplied logistic and chem-div time-regularity packages; they do not create those packages from bare S.

## hB_restart

```javascript
RestartRepresentation S
```

Status: conditionally producible, not automatic from bare S.

The repository has the restart algebra:

```javascript
bForm_restart_of_global_cosine
conjugatePicardLimit_B_restart_of_global_cosine
```

These derive the local restart representation from:

- coefficient continuity in time;

- a global from-zero B-form cosine representation;

- absolute summability of the global restart coefficients.

Thus the restart step itself is proved. What is still needed for the generic V6 S is a producer of those global representation/summability premises, or a direct generic-S restart theorem. In the current reduced record, hB_restart remains an explicit leaf.

## hlogData

```javascript
∀ t, 0 < t → t < S.T →
  LogisticCosineFourierData p S.u t
```

Status: analytic producers exist, but no universal generic-S record filler is wired here.

The existing route supplies:

- continuity of the logistic source slice from spatial slice continuity;

- quadratic coefficient decay/Fourier ℓ¹ from the restart representation and positive-time regularity;

- logisticCosineFourierData_constExtend to package the result.

This leaf should be routine once hsrcB + hB_restart have produced the interior C² slice and the required source decay. It is nevertheless still a separate field of BFormMildSpectralLeaves; bFormMildSpectralBootstrapData_of_leaves does not derive it automatically.

## hchemData

```javascript
∀ t, 0 < t → t < S.T →
  ChemDivCosineFourierData p (S.u t)
    (coupledChemicalConcentration p S.u t)
```

Status: not closed; current type has a known endpoint-interface defect.

The old ChemDivCosineFourierData requires a globally continuous representative agreeing with the physical chemDivLift on the closed interval Icc 0 1. The repository's own IntervalBankSourceSliceLeaves audit shows that the natural closed representative is generically discontinuous at 0 and 1: the zero extension forces the ambient derivative value at the endpoint to differ from the interior limit. Therefore the old closed-agreement package can be uninhabited for a genuine chemotaxis-divergence slice.

The repository has already built the mathematically correct replacement:

```javascript
ChemDivCosineFourierDataIoo
```

in IntervalBankChemSliceFix.lean. It requires agreement only on Ioo 0 1, and endpoint-insensitive cosine-coefficient/inversion consumers have been reproved. But the current BFormMildSpectralLeaves.hchemData field still asks for the old ChemDivCosineFourierData type.

So this is not merely missing wiring. Before claiming all leaves are produced, the HSpectral route should be refactored to consume the Ioo package, or the old package should be replaced by an endpoint-insensitive equivalent.

## hResolverWitness

```javascript
ResolverSourceWitnessFromMild p S
```

Status: lowered to producer-facing frontiers, but not produced from bare S in this assembler.

Once supplied, it is converted immediately by:

```javascript
hResolverData_of_sourceWitness H.hResolverWitness.
```

Earlier resolver work lowers the witness to windowed source data, joint-continuity data, and coefficient-envelope inputs. Those are useful, satisfiable producer surfaces, but they remain analytic inputs; the audited tree does not export a theorem

```javascript
(S : ConjugateMildSolutionData p u₀) → ResolverSourceWitnessFromMild p S
```

from the generic mild fields alone. It is therefore a genuine fifth leaf, separate from the B-form SourceFromSolutionEnvelopePass.

# Bottom line

- Assembler question: yes, direct.

- Source-pass question: SourceFromSolutionEnvelopePass is upstream of hsrcB, not a direct assembler leaf.

- Other exact leaves: hB_restart, hlogData, hchemData, and hResolverWitness, plus the hsrcB that must be built from the source pass and the time engine.

- Are all produced? no. The restart and logistic machinery are substantially available but not universally wired; the resolver witness is still a lowered frontier; hsrcB still needs the finite spatial/time assembly; and the current closed-endpoint hchemData type must be replaced by the already-built Ioo interface.
