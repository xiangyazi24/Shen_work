# 2. The representative product rule is correct

Define the smooth global representatives

```javascript
let Urep : ℝ → ℝ := intervalSolutionLiftU ...
let Rrep : ℝ → ℝ := intervalResolverLiftR ...

let Qrep : ℝ → ℝ := fun x =>
  Urep x * deriv Rrep x * (1 + Rrep x) ^ (-p.β)

let Srep : ℝ → ℝ := deriv Qrep
```

Assuming the relevant differentiability and 0 < 1 + Rrep x, ordinary calculus gives

```plain text
Qrep' = Urep' Rrep' (1+Rrep)^(-β)
      + Urep Rrep'' (1+Rrep)^(-β)
      - β Urep (Rrep')² (1+Rrep)^(-β-1).
```

After the elliptic identity

```plain text
Rrep'' = μ Rrep - ν Urep^γ,
```

this is exactly the physical product-rule representative already named

```javascript
liftChemotaxisDivPhysicalRep
```

in IntervalChiNegH1ChemDivRepresentative.lean.

There is one non-definitional normalization to remember. The literal flux uses division by (1+R)^β, whereas the proposed representative multiplies by (1+R)^(-β). The local equality proof must use

```javascript
Real.rpow_neg hbase.le p.β
```

plus div_eq_mul_inv. It will not close by rfl alone.

More importantly, the cleanest proof does not expand this derivative. Prove local equality of the two flux functions and invoke

```javascript
Filter.EventuallyEq.deriv_eq
```

at an interior point. That automatically transports the whole product/quotient derivative correctly.

# 4. Why exact endpoint agreement generally fails

Agreement of two functions on Icc 0 1 does not imply agreement of their ordinary derivatives at 0 or 1. At an endpoint, Icc supplies only a one-sided neighborhood.

This matters because intervalDomainLift is zero outside [0,1]. The literal flux is therefore zero outside the interval. It is also zero at the endpoints because the Neumann resolver derivative vanishes there. But its interior derivative at an endpoint need not be zero.

For the smooth physical flux

```plain text
Q = u R_x (1+R)^(-β),
```

Neumann data give u_x = R_x = 0 at an endpoint, so

```plain text
Q_x = u R_xx (1+R)^(-β)
```

there. Using the elliptic equation,

```plain text
Q_x = u (μR - νu^γ) (1+R)^(-β),
```

which is generally nonzero.

The zero-extended literal flux has exterior derivative zero. If the interior derivative above is nonzero, the two-sided derivative at the endpoint does not exist. Mathlib then assigns

```javascript
deriv literalFlux 0 = 0
```

through deriv_zero_of_not_differentiableAt. The smooth doubly-reflected representative instead has its genuine derivative Qrep' 0, which can be nonzero.

A concrete calculus counterexample is:

```plain text
β = 0,
Urep(x) = 1,
Rrep(x) = 2 + cos(πx).
```

Then

```plain text
Qrep(x) = -π sin(πx),
Srep(0) = Qrep'(0) = -π².
```

The literal flux obtained by keeping Qrep on [0,1] and setting it to zero to the left has left derivative 0 and right derivative -π². It is not differentiable at 0, so Mathlib's deriv value is 0, not -π².

This is exactly why the repository currently proves only:

```javascript
coupledChemDivSourceLift_eq_physicalRep_Ioo_of_classical
```

and explicitly describes the physical representative as “endpoint-insensitive.” The file IntervalChiNegH1ChemDivRepresentative.lean also states that its closed-slab physical representative is not a claim of endpoint equality with the literal lifted chemotaxisDiv.

Thus:

```plain text
Qrep = literal flux on Icc        may be true,
deriv Qrep = deriv literal flux   is guaranteed only on Ioo.
```

# 7. Final answers to the three questions

1. Does the product derivative match the actual source?  Yes on Ioo 0 1. The product/rpow calculus is correct, and the strongest Lean route is local flux equality followed by EventuallyEq.deriv_eq. It generally does not match the literal source at both endpoints.

1. Is there an endpoint subtlety?  Yes, and it is real. The smooth doubly-even source representative carries the physical one-sided endpoint value. The literal source uses ordinary deriv of a zero-extended flux and can receive Mathlib's junk value 0. Do not claim EqOn ... Icc without a separate endpoint theorem; such a theorem is generally false.

1. Does the measure-zero gap break the adapter?  It breaks only the current syntactic hypothesis of intervalWeakH2Neumann_of_doublyEven_agree. It does not break the mathematics, cosine coefficients, weak H²_N, or Duhamel spectral estimates. Use the existing chemDivSource_weakH2_of_cosineRep or add an Ioo/a.e. congruence adapter.
