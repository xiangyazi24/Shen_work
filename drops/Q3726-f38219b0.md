## Recommendation

The next smallest honest reduction is **not** deleting `Hu` from the Task270
frontier using only `hsrc0`.  The next useful reduction is to **produce `hsrc0`**
from the existing iterate-side bounded-source bootstrap interface

```lean
ShenWork.IntervalPicardLimitBddBootstrap.duhamelSourceBddOn_of_iterates
```

possibly through a thin `Hsrc0BootstrapInputs` structure and adapter into the
Task270 hsrc0-only surface.

Reason: after Task270, the resolver-source side has been reduced to exactly one
source residual (`hsrc0`).  The remaining sibling `Hu` is not a resolver-source
residual; it is the u-side time-regularity package used by the regularity frontier
wiring.  It cannot be derived from `hsrc0 : DuhamelSourceBddOn` alone, because
`Hu` requires local restart charts with **`DuhamelSourceTimeC1`**, not just bounded
source envelopes.

## Option audit

### Option 1: delete `Hu` from Task270 using hsrc0-only data

Verdict: **not viable**.

`HasTimeNeighborhoodSpectralAgreement` is defined in
`PDE/IntervalMildTimeDerivContinuity.lean` as an existential local restart chart:

```lean
structure HasTimeNeighborhoodSpectralAgreement
```

Task270's `hsrc0` is only:


That package provides boundedness, per-compact coefficient envelopes, and
coefficient continuity on closed horizons.  It does **not** provide
`DuhamelSourceTimeC1`, and it does not contain the coefficient derivative data
needed by `restartCosineSeries_hasDerivAt_time` / `HasTimeNeighborhoodSpectralAgreement`.

## Option 4: other small non-circular wiring

The sibling `HuCoeff`/`HuCoeffNoK1` surfaces already exist and are useful only if
`Hu` remains carried.  After Task270, choosing `bc := limitCoeff` and deriving
`henv` from `hsrc0` is stronger for the χ0 route than the HuCoeff route because it
removes both `bc/hagree` and the compact envelope from the resolver-source input.

A small wrapper combining `Hsrc0BootstrapInputs` with the HuCoeff surfaces would
not remove a new analytic residual; it would mostly duplicate Task270's stronger
hsrc0-only path.  I would not spend a commit there unless another pane specifically
needs the Hu-chosen coefficient family for compatibility.

## Red flags and import cycles

* Do **not** try to derive `Hu` from `DuhamelSourceBddOn`; `Hu` requires
  `DuhamelSourceTimeC1` local restart data.
* Do **not** feed downstream classical data (`GradientMildClassicalFrontierCoreData`,
  `IsPaper2ClassicalSolution`, or `hpde_u`) into the source producers.
* Keep the hsrc0 bootstrap core file below `IntervalPicardLimitBddBootstrap` and
  `IntervalResolverSourceWindowEnvelopeOnlyNoJointHsrc0Inputs`, but above any
  route-facing chi-zero frontier wrapper.
* Do not import `IntervalDomainChiZeroResolverSourceEnvelopeOnlyNoJointHsrc0InputsFrontier`
  into the core bootstrap file; that would risk a route/core cycle.  Put headline
  wrappers in a separate downstream file.
* `S : GradientMildHalfStepLogisticSourceData D` does not by itself produce
  `hsrc0`.  It is a restart/source regularity package for the logistic half-step
  route, not the bounded patched-source package used by Task270.
