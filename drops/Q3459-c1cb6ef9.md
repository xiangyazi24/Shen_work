# Q3459 route audit: producing the H¹ initial-window RHS integrability frontier

## Verdict

No. At main commit `6b788c71`, I do not see an existing honest theorem chain that produces

```lean
H1IdentityRHSInitialWindowIntegrableBefore p u T taxisX uvxx reactX
```

for the physical H¹ identity RHS from initial trace, classical regularity, the landed strict-component route, mild/Picard restart data, or existing P3/Moser initial-window packages.

The current state is cleanly factored:

- `IntervalChiNegH1StrictRHSIntegrability.lean` defines the exact remaining input:
  `H1IdentityRHSInitialWindowIntegrableBefore`.
- `H1IdentityRHSIntegrableBefore_of_componentsStrictBefore_initialWindow` correctly combines strict positive-time component continuity with that initial-window assembled-RHS integrability input.
- The bounded-before wrapper
  `boundedBefore_of_H1supBoundDI_classicalChemRep_strictRHS_initialWindow_before`
  then reuses the existing H¹ bounded-before route.

But no current file produces the initial-window input itself without carrying something at least as strong as it. The closest true analogue is the P3/Moser theorem

```lean
intervalDomain_lpPDECombinedInitialWindowIntegrability_of_moserDerivativeInitial
```

in `P3MoserPDECombinedInitialProducer.lean`. That theorem does **not** prove initial-window integrability from classical data; it takes the separate input

```lean
IntegratedMoserEnergyDerivativeInitialWindowIntegrability intervalDomain u T p0
```

and then converts derivative initial-window integrability into the corresponding combined PDE scalar initial-window integrability. The H¹ route should copy that architecture.

## Existing chains that do not close this gap

### 1. Closed-window component continuity route

Existing theorem:

```lean
H1IdentityRHSIntegrableBefore_of_component_continuousOn_Icc
```

and the bridge version

```lean
H1IdentityRHSIntegrableBefore_of_componentsContinuousBefore
```

can produce `H1IdentityRHSIntegrableBefore`, but only from closed-window component continuity for all `0 ≤ a`, including

```lean
ContinuousOn (fun τ => lapL2sq u τ) (Set.Icc 0 b)
```

That is exactly the zero-start lap/H² trace route that Task78/79 isolated as an honest frontier. It is not route C.

### 2. Strict positive-time component continuity route

Existing theorem:

```lean
H1IdentityRHSIntegrableBefore_of_componentsStrictBefore_initialWindow
```

is the correct adapter, but it still requires

```lean
H1IdentityRHSInitialWindowIntegrableBefore p u T taxisX uvxx reactX
```

for the zero-start windows. Strict slab continuity only gives `[a,b]` with `0 < a`; it gives no integrability near `0`.

### 3. H¹ initial trace / scalar endpoint continuity

Existing H¹ initial endpoint theorems, such as

```lean
H1energy_continuousWithinAt_zero_of_initialTraceEnergy
H1energy_continuousOn_before_of_uxxL1Cont_initialTraceEnergy
```

produce right-continuity or closed-window continuity of the scalar H¹ energy. They do not imply integrability of

```lean
H1IdentityRHSValue p u taxisX uvxx reactX
```

near zero. H¹ energy convergence does not control the lap/H² term or the full derivative RHS.

### 4. Sup-bound packages

The package

```lean
H1SupBoundSqrtRHSIntegrableBefore
```

already contains full interval integrability of the explicit RHS. From it one can get `H1IdentityRHSIntegrableBefore` via

```lean
H1IdentityRHSIntegrableBefore_of_supBoundSqrtRHSData
```

and then restrict to `[0,b]`. But that is not a producer of the bottleneck; it assumes a stronger version of it.

Likewise,

```lean
H1SupBoundDIDataBefore
H1SupBoundSqrtDIDataBefore
```

only provide pointwise differential-inequality data. They do not provide initial-window integrability for a fixed physical split.

### 5. P3/Moser initial-window packages

The P3 theorem

```lean
intervalDomain_lpPDECombinedInitialWindowIntegrability_of_moserDerivativeInitial
```

is the correct model, not a reusable H¹ proof. It is about `integratedMoserEnergy`, `intervalDomainPowerEnergyDerivIntegral`, and Lp/Moser PDE combined scalars, not `H1energy` or `H1IdentityRHSValue`.

## What would be overstrong or circular

1. **Strict slabs imply zero-window integrability.** False. A function can be continuous on every `[a,b]` with `0 < a` and fail to be integrable at zero, e.g. `1 / t`.

2. **Initial H¹ energy convergence implies RHS integrability.** False. It gives scalar endpoint continuity, not integrability of the derivative or the lap/H² component.

3. **Use `H1IdentityRHSComponentsContinuousBefore`.** This re-enters the zero-start lap/H² trace route because it includes `ContinuousOn (fun τ => lapL2sq u τ) (Icc 0 b)`.

4. **Use `H1LapComponentZeroRightContinuous`, `H1LiftDeriv2ZeroSlabRepresentativeBefore`, or `H1ZeroStartPhysicalRHSDataBefore`.** Those are the route-B endpoint-H² atoms, not route C.

5. **Use `IsPaper2BoundedBefore` or the final H¹ bound to produce `hDerivInit` or `hInit`.** That is circular: those estimates are being used to prove bounded-before.

6. **Use `H1SupBoundDIDataBefore` as an integrability producer.** It is pointwise and existential; it does not provide interval-integrability of a fixed split `taxisX uvxx reactX`.

7. **Use `H1SupBoundSqrtRHSIntegrableBefore` as though it were produced.** It already contains full RHS interval-integrability, hence contains the bottleneck. It is fine as a stronger hypothesis, but it does not close the current frontier.
