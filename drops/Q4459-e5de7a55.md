# Definitive verdict

There is **no new analytic truncation-removal limit** after the negative-part energy argument proves pointwise nonnegativity of the faithful truncated Picard limit.

The exact distinction is:

1. **Recognition as an untruncated mild solution:** purely algebraic after `u ≥ 0`. The same truncated trajectory satisfies the full B-form Duhamel identity because every occurrence of `positivePart` becomes the identity.
2. **Identification with the separately named full Picard limit `conjugatePicardLimit`:** not definitional, but already handled by the explicit repository bridge using full-map uniqueness. This requires matching horizons and membership of the truncated trajectory in the full Picard ball. It does **not** require constructing a second limit or proving a new fixed-point theorem.
3. **Strict positivity is not used anywhere in truncation removal.** Pointwise `0 ≤ u` is exactly sufficient. Jensen's `0 < u` output is needed later for the `hpos` field and positive-time regularity/source work, not for existence of the physical untruncated solution.

# 4. Does removal require strict positivity?

No.

Every load-bearing truncation-removal theorem assumes only nonnegativity:

```lean
∀ x, 0 ≤ w x
```

or

```lean
∀ t x, 0 ≤ u t x.
```

In particular:

```lean
truncatedChemFluxLifted_eq_chemFluxLifted_of_nonneg
truncatedLogisticLifted_eq_logisticLifted_of_nonneg
truncatedConjugateDuhamelMap_eq_intervalConjugateDuhamelMap_of_nonneg
truncatedConjugateLimitBridge_of_faithful_truncation
```

have no `0 < u` hypothesis.

Jensen enters only after energy:

```text
energy → hnonneg
       → truncation removal / physical hmild
       → Jensen barrier → hpos
       → positive-time source/spectral/classical regularity.
```

Thus strict positivity is **not** a prerequisite for existence of the actual solution. It is a stronger downstream property of that already-physical solution.

The theorem `truncatedJensenStrictPosDataFor_of_strictPos` should not be confused with truncation removal: it is a Jensen witness-packaging theorem that assumes `hpos`; it has no role in proving `positivePart u = u`.

# 5. What could still go wrong in final wiring?

There is no hidden truncation-removal analysis, but four concrete wiring mistakes remain possible.

## A. Rewriting only the flux and forgetting the logistic source

The physical mild map contains both nonlinear legs. Use

```lean
truncatedConjugateDuhamelMap_eq_intervalConjugateDuhamelMap_of_nonneg
```

rather than manually rewriting only `truncatedChemFluxLifted`.

## B. Supplying only active-window nonnegativity to an all-time map lemma

First pass through

```lean
truncatedConjugatePicardLimit_nonneg_global
```

or prove the same zero-outside-window split locally.

## C. Setting `S.u := conjugatePicardLimit` without supplying the equality bridge

If the record uses the named full Picard limit rather than the truncated limit itself, invoke

```lean
truncatedConjugateLimitBridgeProducerData_of_cores
truncatedConjugateLimitBridge_of_faithful_truncation
```

and check `DT.T = DB.T` and `DT.M ≤ DB.M`.

## D. Treating `hpos` as needed for `hmild`

It is not. The correct dependency is

```text
Henergy → hnonneg → hmild_physical
Henergy + Jensen → hpos.
```

This matters for acyclicity: HSpectral and the strict-positive lower floor must not be imported back into the truncation-removal proof.
