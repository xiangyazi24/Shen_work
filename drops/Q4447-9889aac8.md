## But the claimed critical-path implication is false

`Paper3/Statements.lean` explicitly describes the paper's main results as Theorems 2.1–2.5. The source architecture also separates the target bundles:

The interval assembly makes the separation even more explicit. The Proposition 1.x portion occupies its own data structures and theorem wrappers; afterward it starts a new section:

No Proposition 1.3/1.4 field appears in that core target.

Therefore the correct conclusion is:

* Propositions 1.3/1.4 are genuine paper deliverables.
* They enlarge the class of parameter regimes for which one can instantiate a global bounded orbit.
* They are **not formal prerequisites for developing or proving the conditional dynamical theorems**.
* They become prerequisites only for corollaries of the form:

  ```text

  → existence of a positive global bounded orbit
  → apply Theorem 2.x to that orbit.
  ```

So the roadmap sentence “`Lᵖ` before dynamics because it unlocks Props 1.3/1.4” is a sound **coverage/milestone** argument, but not a sound **dependency/critical-path** argument.

# 3. Should the engine be front-loaded?

## Yes

The engine is the highest-uncertainty and longest-duration component:

```text
linearized-PDE correctness
fractional/Sobolev phase-space choice
nonlinear remainder estimate
Duhamel contraction/orbit control
uniform-in-tail regularization
compactness
Lyapunov PDE identity
moment-to-uniform upgrade
entry into the local basin
```

By contrast, Paper 2 `Lᵖ` is substantial but comparatively isolated and has a well-defined mathematical script: weighted energy, branch-specific absorption, finite exponent bootstrap, then Proposition 2.5.

Front-loading the engine has three advantages:

1. It discovers any missing Mathlib phase-space or semigroup primitive early.
2. It forces the physical-PDE-to-modal correctness interface to stabilize before more downstream files depend on an abstract `sigma` formula.
3. It prevents a long period of apparent progress on independent `Lᵖ` results while the actual trilogy critical path remains untouched.

The cheap linear part should be first:

```text
3A. physical linearization → cosine mode block → sigma → spectral gap.
```

This already yields the linear side of Theorem 2.2 and validates all constants and signs. It is small enough that there is no strategic justification for placing the entire `Lᵖ` mountain before it.

Then begin the true long pole:

```text
3B. stable linear semigroup + nonlinear remainder/orbit estimate.
```

Paper 2 `Lᵖ` can proceed concurrently.

# 4. Hidden cross-dependencies and gates

## Gate A — model faithfulness is shared by steps 2 and 3

This is the most serious hidden issue.

The current concrete interval definitions are

```lean
def intervalDomainChemotaxisDiv (p : CM2Params)
    (u v : intervalDomainPoint → ℝ) ... :=
  deriv (fun y =>
    intervalDomainLift u y * deriv (intervalDomainLift v) y /
      (1 + intervalDomainLift v y) ^ p.β) ...
```

and

```lean
def intervalDomainCrossDiffusionEnergyTerm ... :=
  ∫ x in 0..1,
    u(x)^(pExp-1) * |u_x| * |v_x| / (1+v)^β
```

They use `u` to the first power and do not use `p.m`.

But the Paper 3 modal growth rate is

```lean
def sigma ... :=
  -lambdaN +
    p.χ₀ * p.ν * p.γ *
      (uStar ^ (p.m + p.γ - 1) * lambdaN) /
        ((1 + vStar) ^ p.β * (p.μ + lambdaN)) -
    p.a * p.α
```

and Paper 2 Theorem 1.2 explicitly distinguishes `0 < m < 1` from `m = 1`.

Thus both mountains currently share an upstream model-consistency gate:

```text
Does the concrete interval PDE represent u^m v_x/(1+v)^β,
or is the interval implementation intentionally specialized to m=1?
```

If the target is the full trilogy model with arbitrary `m`, step 0 must repair or parameterize:

```lean
intervalDomainChemotaxisDiv
intervalDomainCrossDiffusionEnergyTerm
all physical-PDE/IBP bridges that unfold them
```

before either the general-`m` `Lᵖ` proof or the physical linearization-to-`sigma` theorem is trusted.

This is not a dependency from step 2 to step 3. It is a **common predecessor** of both.

Accordingly, step 0 should be renamed:

```text
0*. Freeze the physical model and theorem API.
```

Its gate should include:

```text
- exact flux exponent m appears in the concrete PDE;
- Paper2 and Paper3 use the same μ,ν,γ,m,β conventions;
- global-solution API exports trace, positivity, boundedness and regularity;
- no downstream theorem unfolds a stale m=1-only definition unintentionally.
```

A finite-`p` or even `L∞` bound is useful input, but it does not by itself prove time-translate compactness, the Lyapunov identity, or convergence into the local stable basin.

Therefore placing `Lᵖ` first does not substantially shorten the engine's hard part.

## Gate D — do not bake Props 1.3/1.4 into the engine interface

The correct engine theorem should look like


It should not call a specific existence theorem internally. Otherwise the code creates an artificial circular-looking dependency and prevents reuse in parameter regimes supplied by another existence theorem.

## Gate E — P3.1 may be cheap but regime-limited

The P3.1 reuse theorem should state exactly which parameter branch it covers. A result inherited from the completed `χ₀≤0` Paper 2 core cannot silently serve as the global trajectory producer for positive-sensitivity stability regimes.

This affects final applications, not the engine construction.

# Final answer to the four questions

1. **Does step 3 depend on step 2?**  No. Neither the linear modal engine nor the abstract nonlinear orbit-control engine depends on Paper 2 `Lᵖ`. Step 2 is needed only to supply global bounded trajectories in additional parameter regimes.

2. **Are Props 1.3/1.4 on the critical path?**  No, not to the conditional Theorems 2.1–2.5 engine. They are separate paper deliverables and later orbit suppliers. The “unlock” rationale is true, but it is a coverage argument rather than a dependency argument.

3. **Is `0→1→3→2→4–8` better?**  Yes, with the refinement that step 3 should be split into `3A` and `3B`, and step 2 should run in parallel with them whenever possible. At minimum, `3A` must move before step 2.

4. **Any hidden dependency making the roadmap wrong?**  The main hidden issue is not `Lᵖ→engine`; it is the shared physical-model gate. The current concrete interval chemotaxis operator/cross-energy omits `p.m`, while both the general `Lᵖ` proof and Paper 3 `sigma` use it. Fix or explicitly specialize that model before either branch. Also ensure step 1 exports boundedness/positivity if it is to serve as an orbit supplier.

# Bottom line

The committed order is **safe but strategically conservative**. It is not the optimal critical-path schedule.

Replace the strict chain

```text
0 → 1 → 2 → 3 → 4–8 → 9
```

by the dependency-aware plan

```text
0* → 1,
0* → 3A → 3B → 4,
0* → 2 → Props 1.3/1.4,
then merge → 5–8,
then 9.
```

The spectral-dynamical engine should be front-loaded immediately. Paper 2 `Lᵖ` is an important parallel mountain, not its prerequisite.
