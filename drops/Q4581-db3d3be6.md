# `shen1 --help`

`shen1` is the repo-aware Lean/PDE audit and implementation interface used by these numbered Shen tasks. It accepts a free-form request, inspects the specified repository ref, and returns an exact source-grounded report or a requested GitHub edit.

No indexed repository command definition named `shen1` surfaced, so `--help` is interpreted as help for this Q-task interface, not as a flag to a compiled Lean executable.

## Usage

```text
Q<task-number> (shen1): <request>
```

Recommended full form:

```text
Q4582 (shen1):
Repository: xiangyazi24/Shen_work
Ref: chatgpt-scratch          # branch, tag, or commit SHA
Targets:
  - ShenWork/Paper1/Foo.lean
  - ShenWork/Paper1/Bar.lean
Ask:
  - list the exact undischarged hypotheses of theorem X;
  - classify each as mechanical or genuine analysis;
  - name all existing definitions and lemmas that wire into it;
  - give the minimal lemma DAG and the single hard core.
Constraints:
  - repo-only;
  - no Minty;
  - preserve the lower-pinned route;
  - do not treat comments as proof.
Delivery:
  - write the report to <path> on <branch> and report the commit SHA.
```

## Core request types

### 1. Exact frontier audit

Use this when a theorem is a conditional wrapper and you want to know what is truly still assumed.

```text
List the EXACT carried hypotheses of
Theorem_1_1.of_assumed_frozenStationaryProfile_branches.
For every field, say whether it is:
- already proved and directly wireable;
- a mechanical adapter;
- a genuine missing analytic lemma;
- false or over-strong as currently typed.
```

A good audit distinguishes:

- the literal theorem arguments from lower-level obligations hidden inside structures;
- a statement-layer bridge from a producer of its hypotheses;
- current executable Lean from stale file-header commentary;
- a proof of a conclusion from a predicate that merely re-assumes that conclusion;
- branch-specific work from sign-agnostic infrastructure.

### 2. Lemma-DAG / minimal Lean route

```text
Give the shortest acyclic Lean DAG from the existing Green-kernel lemmas to
RotheContinuousDependence. Name every intermediate definition and theorem.
Mark new nodes explicitly.
```

Expected shape:

```text
proved atom A
proved atom B
      \ /
new lemma C
    |
existing adapter D
    |
target theorem E
```

The report should identify the smallest useful new theorem signature rather than merely restating the final headline theorem.

### 3. Hard-analysis versus wiring classification

Useful wording:

```text
For each frontier, label it:
- mechanical Lean packaging;
- standard but nontrivial formal analysis;
- genuine mathematical hard core;
- statement defect / impossible endpoint.
Explain the reason in one paragraph and identify the consumer theorem.
```

Typical mechanical work includes tuple reassociation, specialization, `simpa`, trap projections, and feeding an existing theorem into a wrapper. Typical hard work includes a new compactness/closed-graph theorem, a uniform tail estimate, a strong maximum principle, a nonlinear asymptotic linearization, or a genuinely new PDE energy estimate.

### 4. Source-accuracy / stale-comment audit

```text
Compare the file headers with the actual declarations at HEAD. Report every stale
claim such as “UNCOMMITTED” when a theorem is now proved elsewhere.
```

For exact work, specify a ref. Code search commonly reflects the default branch; non-default-branch conclusions should be checked by directly fetching the named file at the requested ref.

### 5. Lean patch or report file

```text
Implement the smallest patch that closes frontier X.
Touch only Paper1.
No `sorry`, `admit`, `axiom`, or `native_decide`.
Write the change to branch <branch> and report the commit SHA.
```

Or, for analysis only:

```text
REPORT ONLY. Do not modify Lean source. Write the complete audit to drops/<file>.md.
```

A connector write is not a compilation result. Build or CI success is claimed only when it has actually been inspected.

## High-value modifiers

| Modifier | Meaning |
|---|---|
| `EXACT` | Copy the current Lean proposition/signature faithfully before analyzing it. |
| `HEAD=<sha>` / `Ref: <branch>` | Pin the audit to a concrete repository state. |
| `repo-only` | Do not use external mathematical sources. |
| `roadmap Q####` | Reconcile the answer with the named earlier design decision. |
| `NOT Minty` | Do not route through monotone-operator/Minty machinery. |
| `NOT Aubin–Lions-only` | Use the repository’s actual Green/local-uniform topology and tail mechanism. |
| `single genuine hard core` | Compress multiple carried predicates to the smallest common analytic theorem. |
| `wireable now` | Separate already-proved atoms from lemmas that still need proof. |
| `minimal DAG` | Avoid broad redesign; give the shortest acyclic dependency chain. |
| `axiom audit` | Track `sorry`/axiom status and avoid treating conditional wrappers as closure. |
| `PATCH` | Modify repository files, rather than only describing a route. |
| `REPORT ONLY` | Make no source-code changes other than the requested report artifact. |

## Preferred output contract

For a substantive theorem audit, the most useful response order is:

1. **Audited repository ref and files.**
2. **Literal theorem signature or exact carried frontier.**
3. **Classification table:** mechanical, proved/wireable, genuine analysis, or statement defect.
4. **Existing reusable definitions and lemmas.**
5. **Minimal Lean DAG.**
6. **Smallest missing theorem signature.**
7. **Single genuine hard core.**
8. **Risks:** stale comments, branch mismatch, circular interfaces, false endpoints, or overly strong uniformity.
9. **GitHub result:** path and commit SHA when a write was requested.

## What to include for best results

Provide the exact repository, branch or SHA, target declarations, relevant files, desired depth, forbidden routes, whether code edits are allowed, and the required delivery path. When a previous roadmap answer controls the architecture, name its Q-number and summarize the binding decision.

## Operational limits

- A theorem is not called discharged merely because an adapter consumes it.
- File-header prose is lower-trust than current declarations and proof terms.
- Results on a non-default branch require direct ref-specific file reads; default-branch search alone is insufficient.
- Compilation, axiom cleanliness, or CI success is reported only after actual verification.
- Work is performed in the current response; there is no background job or deferred delivery.
- When the requested mathematical proof is uncertain, the response should isolate the uncertainty and still provide the strongest source-grounded partial DAG rather than fabricate closure.
