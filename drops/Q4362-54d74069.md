# 1. Exact start-to-end order ledger

For positive modes `k >= 1`, write "order `q`" for a uniform compact-window bound of the form

```text
|a_k(t)| <= C k^(-q).
```

The zero mode is handled separately and is harmless for the final eigenvalue-weighted sum because `λ_0 = 0`.

Assume

```text
u_hat_k(t) = O(k^(-m))
```

uniformly for `t in [c,T']`, with `m >= 1`.

## A. Nemytskii terms

The positive floor `u >= δ > 0`, the uniform upper bound, and the fixed-order Moser/Nemytskii estimate give

```text
(u^γ)_hat_k = O(k^(-m)).
```

Likewise the logistic term

```text
L(u) := u (a - b u^α)
```

has order `m`:

```text
L_hat_k = O(k^(-m)).
```

The scalar factors such as the resolver-source amplitude do not affect the order.

## B. Elliptic resolver

For the generic operator

```text
(μ - ν ∂_xx) R = f,
```

one has

```text
R_hat_k = f_hat_k / (μ + ν λ_k),
λ_k = (kπ)^2.
```

Since the denominator is comparable to `1 + λ_k`, division gains two powers of `k`:

```text
f order m  ->  R order m+2.
```

In the currently compiled `shen_work` resolver, the diagonal denominator is actually

```text
p.μ + λ_k,
```

while `p.ν` is part of the source coefficient. This distinction changes constants but not the `+2` order gain.

A spatial derivative then consumes one of the two gained orders:

```text
R order m+2  ->  R_x order m+1.
```

The denominator factor

```text
D(R) := (1+R)^(-β)
```

preserves the available order because `1+R >= 1`; it is at least as regular as needed for the product. Thus

```text
R_x D(R) has order m+1.
```

## C. Flux bottleneck

The chemotaxis flux is

```text
Q := u * R_x * (1+R)^(-β).
```

Although the resolver factor has order `m+1`, the leading density factor has only order `m`. Product/Moser algebra therefore gives the minimum order:

```text
Q order = min(m, m+1) = m.
```

This is where the tempting but incorrect extra `+1` or `+2` disappears. The elliptic gain makes the resolver side **better than** `u`; the density `u` remains the bottleneck.

## D. Divergence loses exactly one order

The Q4352 identity is

```text
cosineCoeff(∂_x Q, k)
  = (kπ) * sineCoeff(Q, k)
```

up to the already-fixed sign convention. Therefore

```text
sineCoeff(Q,k) = O(k^(-m))
```

implies

```text
cosineCoeff(∂_x Q,k) = O(k^(-(m-1))).
```

Hence

```text
chemDiv order = m-1.
```

The logistic coefficient has the stronger order `m`, so it can be weakened to order `m-1`. Consequently their sum, `bFormSourceCoeffs`, has order

```text
min(m, m-1) = m-1.
```

The complete ledger is therefore:

| object | coefficient-decay order |
|---|---:|
| `u` | `m` |
| `u^γ` | `m` |
| `R[u]` | `m+2` |
| `R_x` | `m+1` |
| `(1+R)^(-β)` | at least the order needed; resolver side is not limiting |
| `R_x (1+R)^(-β)` | `m+1` |
| `Q = u R_x (1+R)^(-β)` | `m` |
| `∂_x Q` | `m-1` |
| logistic term | `m` |
| full B-form source | `m-1` |

Thus there is no missing order in the stated `SourceFromSolutionEnvelopePass` target.

# 2. Where the heat-semigroup `+2` belongs

There are two different `+2` gains and they must not be conflated.

## Elliptic `+2`

This occurs **inside** the nonlinear source construction:

```text
u^γ order m -> R order m+2.
```

It is then partially spent by `R_x` and finally bottlenecked by multiplication with `u`. Its purpose is to ensure that the resolver factor does not reduce the flux below order `m`.

## Parabolic/Duhamel `+2`

This occurs **after** a source envelope has been produced. Schematically,

```text
ladder_pass_gain_envelope :
  WindowSourceEnvelope p -> WindowCoefficientEnvelope (p+2).
```

Combining the two separate passes gives, from the solution-index viewpoint,

```text
WCE m
  -- SourceFromSolutionEnvelopePass (-1) --> WSE (m-1)
  -- ladder_pass_gain_envelope (+2)       --> WCE (m+1).
```

So a complete nonlinear/parabolic cycle has net gain `+1` in the solution order.

Equivalently, from the source-index viewpoint,

```text
WSE p
  -- heat/Duhamel (+2) --> WCE (p+2)
  -- source pass (-1)  --> WSE (p+1).
```

Thus every full heat-then-source cycle raises the source-envelope order by exactly one.

This answers the first question precisely:

- **No**, `ladder_pass_gain_envelope` is not needed to make a single `SourceFromSolutionEnvelopePass` close at its declared output `m-1`.
- **Yes**, it is needed in the overall finite induction when the currently available coefficient order is below `3` and the goal is the final `WindowSourceEnvelope 2`.


Therefore it reaches pass 2 exactly when


Under the pass hypothesis `1 <= m`, natural-number arithmetic gives

```text
2 <= m-1  <=>  3 <= m.
```

The minimal value is consequently

```text

```

The first three cases make the issue transparent:

| input solution order `m` | resolver order | flux order | source order `m-1` | pass-2 ready? |
|---:|---:|---:|---:|---|

| `3` | `5` | `3` | `2` | yes |

So the answer to “does elliptic `+2` make every `m>=1` pass-2 ready?” is **no**. The leading factor `u` and the final divergence force the output to remain exactly `m-1`.
