# Q3952: operator-level IBP for the Neumann conjugate kernel operator

## Executive answer

The cleanest route is to **prove the integration-by-parts identity as an `intervalIntegral` statement on `0..1`**, and only at the very end rewrite the two operator definitions from

```lean
∫ y, ... ∂ intervalMeasure 1
```

to

```lean
∫ y in (0 : ℝ)..1, ...
```

using the standard `Icc`/`Ioc` endpoint-null bridge.  Do not try to prove a new IBP theorem for `MeasureTheory.integral` over `intervalMeasure 1`; localize that nuisance to one conversion lemma.

The sign convention is the expected one.  If

```text
K(y)  = intervalNeumannFullKernel t x y,
dK(y) = deriv (fun y' => intervalNeumannFullKernel t x y') y,
```

then ordinary IBP says

```text
∫₀¹ K(y) Q'(y) dy = K(1)Q(1) - K(0)Q(0) - ∫₀¹ dK(y)Q(y) dy.
```

With `Q 0 = 0` and `Q 1 = 0`, this becomes

```text
∫₀¹ K(y) Q'(y) dy = -∫₀¹ dK(y)Q(y) dy,
```

which is exactly the leading minus in `intervalConjugateKernelOperator`.

## Repo facts used

The relevant definitions are exactly as expected:

```lean
def intervalMeasure (L : ℝ) : Measure ℝ :=
  volume.restrict (intervalSet L)

def intervalSet (L : ℝ) : Set ℝ := Set.Icc 0 L
```

in `ShenWork/PDE/IntervalDomain.lean`, and

```lean
def intervalFullSemigroupOperator (t : ℝ) (f : ℝ → ℝ) (x : ℝ) : ℝ :=
  ∫ y, intervalNeumannFullKernel t x y * f y ∂ intervalMeasure 1
```

in `ShenWork/PDE/IntervalNeumannFullKernel.lean`.  The conjugate operator is

```lean
def intervalConjugateKernelOperator (t : ℝ) (Q : ℝ → ℝ) (x : ℝ) : ℝ :=
  -∫ y, deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y * Q y
      ∂ intervalMeasure 1
```

in `ShenWork/Paper2/IntervalConjugateDuhamelMap.lean`.

Also, the repo already uses the exact measure-conversion pattern in `ShenWork/PDE/IntervalFullKernelSourceIBP.lean`:

```lean
simp only [intervalMeasure, intervalSet]
rw [MeasureTheory.integral_Icc_eq_integral_Ioc,
  ← intervalIntegral.integral_of_le h01]
```

So the operator-level proof should follow that existing style.

## Recommended Lean API

Use an explicit derivative witness first:

```lean
intervalConjugateKernelOperator t Q x = intervalFullSemigroupOperator t Q' x
```

under `HasDerivAt Q (Q' y) y`.  Then specialize `Q' := fun y => deriv Q y` when you actually have pointwise classical differentiability.  This is better for the flux layer, because Lipschitz/AC facts often produce a named a.e. or weak derivative before they produce a convenient theorem about `deriv Q`.

Here is the proof skeleton I would add near the B-form operator file, or in a small downstream bridge file importing it.

```lean
import ShenWork.Paper2.IntervalConjugateDuhamelMap
import Mathlib.MeasureTheory.Integral.IntervalIntegral.IntegrationByParts

open MeasureTheory intervalIntegral
open scoped Topology

noncomputable section

namespace ShenWork.IntervalConjugateDuhamelMap

open ShenWork.IntervalDomain
open ShenWork.IntervalNeumannFullKernel

/-- Convert the concrete restricted measure on `[0,1]` to Mathlib's interval integral.

This is just the `Icc`/`Ioc` endpoint-null bridge; no integrability hypothesis is needed. -/
lemma intervalMeasure_one_integral_eq_intervalIntegral (f : ℝ → ℝ) :
    (∫ y, f y ∂ intervalMeasure 1) = ∫ y in (0 : ℝ)..1, f y := by
  have h01 : (0 : ℝ) ≤ 1 := by norm_num
  simp only [intervalMeasure, intervalSet]
  rw [MeasureTheory.integral_Icc_eq_integral_Ioc,
    ← intervalIntegral.integral_of_le h01]

/-- Operator-level integration by parts for the full Neumann kernel in the source variable.

This version uses an explicit derivative witness `Q'`.  It is the best reusable API:
for a classical source, instantiate `Q' := fun y => deriv Q y`; for a weak/AC source,
instantiate `Q'` with the known a.e. derivative and then separately identify it with
`deriv Q` a.e. if needed. -/
theorem intervalConjugateKernelOperator_eq_intervalFullSemigroupOperator_of_derivative
    {t x : ℝ} (ht : 0 < t) {Q Q' : ℝ → ℝ}
    (hQ_deriv : ∀ y ∈ Set.uIcc (0 : ℝ) 1, HasDerivAt Q (Q' y) y)
    (hQ'_int : IntervalIntegrable Q' MeasureTheory.volume 0 1)
    (hQ0 : Q 0 = 0) (hQ1 : Q 1 = 0) :
    intervalConjugateKernelOperator t Q x =
      intervalFullSemigroupOperator t Q' x := by
  classical

  let K : ℝ → ℝ := fun y : ℝ => intervalNeumannFullKernel t x y
  let dK : ℝ → ℝ :=
    fun y : ℝ => deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y

  have hK_deriv :
      ∀ y ∈ Set.uIcc (0 : ℝ) 1, HasDerivAt K (dK y) y := by
    intro y _hy
    have h := hasDerivAt_intervalNeumannFullKernel_snd ht x y
    simpa [K, dK, h.deriv] using h

  have hdK_int : IntervalIntegrable dK MeasureTheory.volume 0 1 := by
    simpa [dK] using intervalIntegrable_deriv_intervalNeumannFullKernel_snd ht x

  have hIBP :
      (∫ y in (0 : ℝ)..1, K y * Q' y) =
        K 1 * Q 1 - K 0 * Q 0 - ∫ y in (0 : ℝ)..1, dK y * Q y := by
    exact intervalIntegral.integral_mul_deriv_eq_deriv_mul
      hK_deriv hQ_deriv hdK_int hQ'_int

  have hIBP_zero :
      (∫ y in (0 : ℝ)..1, K y * Q' y) =
        -∫ y in (0 : ℝ)..1, dK y * Q y := by
    simpa [hQ0, hQ1] using hIBP

  unfold intervalConjugateKernelOperator
  unfold ShenWork.IntervalNeumannFullKernel.intervalFullSemigroupOperator
  rw [intervalMeasure_one_integral_eq_intervalIntegral
    (f := fun y : ℝ =>
      deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y * Q y)]
  rw [intervalMeasure_one_integral_eq_intervalIntegral
    (f := fun y : ℝ => intervalNeumannFullKernel t x y * Q' y)]
  simpa [K, dK] using hIBP_zero.symm

/-- The displayed `deriv Q` version. -/
theorem intervalConjugateKernelOperator_eq_intervalFullSemigroupOperator_deriv
    {t x : ℝ} (ht : 0 < t) {Q : ℝ → ℝ}
    (hQ_deriv : ∀ y ∈ Set.uIcc (0 : ℝ) 1, HasDerivAt Q (deriv Q y) y)
    (hQ_deriv_int :
      IntervalIntegrable (fun y : ℝ => deriv Q y) MeasureTheory.volume 0 1)
    (hQ0 : Q 0 = 0) (hQ1 : Q 1 = 0) :
    intervalConjugateKernelOperator t Q x =
      intervalFullSemigroupOperator t (deriv Q) x := by
  exact intervalConjugateKernelOperator_eq_intervalFullSemigroupOperator_of_derivative
    (t := t) (x := x) (Q := Q) (Q' := fun y : ℝ => deriv Q y)
    ht hQ_deriv hQ_deriv_int hQ0 hQ1

end ShenWork.IntervalConjugateDuhamelMap
```

## If endpoint differentiability of `Q` is inconvenient

The theorem above uses the repo-friendly lemma `intervalIntegral.integral_mul_deriv_eq_deriv_mul`, whose hypotheses are convenient but strong: it asks for `HasDerivAt` on `Set.uIcc 0 1`.  If your source regularity only gives interior differentiability plus endpoint continuity, use the more explicit Mathlib lemma you mentioned,

```lean
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivAt
```

with hypotheses of the following shape:

```lean
(hQ_cont : ContinuousOn Q (Set.uIcc (0 : ℝ) 1))
(hQ_deriv : ∀ y ∈ Set.Ioo (0 : ℝ) 1, HasDerivAt Q (Q' y) y)
(hQ'_int : IntervalIntegrable Q' MeasureTheory.volume 0 1)
```

For the kernel side there is no real issue, since `hasDerivAt_intervalNeumannFullKernel_snd ht x y` is global in `y`; you can derive the needed continuity of `K` from `HasDerivAt.continuousAt` or use the existing positive-time kernel regularity lemmas.  The rest of the proof is identical: do IBP as an interval integral, kill the boundary terms with `Q 0 = 0` and `Q 1 = 0`, then rewrite the operator definitions.

## Lipschitz vs absolutely continuous vs pointwise `HasDerivAt`

For the shortest Lean proof with current Mathlib/repo lemmas, use one of these classical-differentiability APIs:

```lean
∀ y ∈ Set.uIcc (0 : ℝ) 1, HasDerivAt Q (Q' y) y
```

or, if using the `_of_hasDerivAt` lemma,

```lean
ContinuousOn Q (Set.uIcc (0 : ℝ) 1)
∀ y ∈ Set.Ioo (0 : ℝ) 1, HasDerivAt Q (Q' y) y
IntervalIntegrable Q' volume 0 1
```

Plain `LipschitzOnWith C Q (Set.Icc 0 1)` is not a direct input to these IBP lemmas.  Mathematically it is enough, because Lipschitz implies absolute continuity and a.e. differentiability, but in Lean that leads to a different proof obligation:

1. prove or import the AC/a.e.-derivative IBP statement;
2. use a derivative witness `Qae'` in the semigroup term;
3. prove `Qae' =ᵐ[...] deriv Q` if you want the final RHS literally written with `deriv Q`.

So the practical hierarchy is:

```text
C¹-style / HasDerivAt + IntervalIntegrable derivative
    easiest; use the code above.

AbsolutelyContinuousOn + a.e. derivative witness
    mathematically right for weak fluxes; needs an AC-product IBP bridge.

LipschitzOnWith
    enough only after reducing to the AC/a.e. derivative bridge; not enough by itself
    for `integral_mul_deriv_eq_deriv_mul_of_hasDerivAt`.
```

For your flux application, `truncatedChemFluxLifted_zero_left/right` should provide the two endpoint hypotheses `hQ0` and `hQ1`.  The remaining question is whether the available positive-time bootstrap gives a pointwise classical derivative theorem for the lifted flux.  If it only gives Lipschitz/AC, keep the explicit `Q'` theorem above and postpone the conversion to `deriv Q` until you have an a.e.-equality theorem.

## Answer to the four specific questions

1. **Cleanest Lean approach:** prove the IBP identity with `intervalIntegral` on `0..1`, using `intervalIntegral.integral_mul_deriv_eq_deriv_mul` or the `_of_hasDerivAt` variant, then rewrite the operator integrals with a one-line `intervalMeasure 1` bridge.

2. **Hypotheses on `Q`:** for existing Mathlib IBP, use pointwise `HasDerivAt` plus `IntervalIntegrable` derivative, and endpoint values.  Lipschitz alone is not the right API unless you first develop the absolutely-continuous/a.e.-derivative version.

3. **Avoiding `intervalMeasure`:** yes.  Avoid it during IBP completely.  Convert both operator integrals to `∫ y in 0..1` after unfolding.

4. **Sign convention:** no hidden sign problem.  The leading minus in `intervalConjugateKernelOperator` is exactly the minus from moving `∂ᵧ` off the kernel and onto `Q`.  There is no extra sign from the measure conversion because the interval is oriented as `0..1` with `0 ≤ 1`.
