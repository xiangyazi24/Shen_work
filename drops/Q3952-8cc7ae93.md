# Q3952: operator-level IBP for the Neumann conjugate kernel operator

## Executive answer

The clean Lean route is to prove the integration-by-parts identity as an `intervalIntegral` statement on `0..1`, then rewrite the two operator definitions from `intervalMeasure 1` to interval integrals.  Do **not** try to prove a new IBP theorem directly for `MeasureTheory.integral` against `intervalMeasure 1`; that only adds endpoint bookkeeping.

For

```lean
K  y = intervalNeumannFullKernel t x y
K' y = deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y
```

Mathlib's lemma gives

```lean
∫ y in (0 : ℝ)..1, K y * deriv Q y
  = K 1 * Q 1 - K 0 * Q 0 - ∫ y in (0 : ℝ)..1, K' y * Q y
```

With `Q 0 = 0` and `Q 1 = 0`, this becomes

```lean
∫ y in (0 : ℝ)..1, K y * deriv Q y
  = - ∫ y in (0 : ℝ)..1, K' y * Q y
```

which is exactly

```lean
intervalConjugateKernelOperator t Q x
  = intervalFullSemigroupOperator t (deriv Q) x
```

because the conjugate operator has the leading minus sign.

## 1. The measure bridge to add first

In the repo, `intervalMeasure L` is `volume.restrict (Set.Icc 0 L)`, while `∫ y in 0..1, f y` is represented using `volume.restrict (Set.Ioc 0 1)` when `0 ≤ 1`.  Since Lebesgue measure has no atoms, `Ioc 0 1` and `Icc 0 1` define the same restricted measure.  Isolate this once:

```lean
import ShenWork.Paper2.IntervalConjugateDuhamelMap
import Mathlib.MeasureTheory.Integral.IntervalIntegral.IntegrationByParts

open MeasureTheory Set
open scoped Topology Real Interval

noncomputable section

namespace ShenWork.IntervalConjugateDuhamelMap

open ShenWork.IntervalDomain
open ShenWork.IntervalNeumannFullKernel

/-- Convert the repo's concrete interval measure on `[0,1]` to Mathlib's
`intervalIntegral` convention.  Mathlib interval integrals use `Ioc`; the repo
measure uses `Icc`; these are equal for Lebesgue measure because the missing
endpoint is null. -/
private theorem integral_intervalMeasure_one_eq_intervalIntegral (f : ℝ → ℝ) :
    (∫ y, f y ∂ intervalMeasure 1) = ∫ y in (0 : ℝ)..1, f y := by
  have h01 : (0 : ℝ) ≤ 1 := by norm_num
  unfold intervalMeasure intervalSet
  -- `volume.restrict (Icc 0 1)` -> `volume.restrict (Ioc 0 1)`.
  rw [← MeasureTheory.restrict_Ioc_eq_restrict_Icc
    (μ := volume) (a := (0 : ℝ)) (b := (1 : ℝ))]
  -- `volume.restrict (Ioc 0 1)` -> interval integral over `0..1`.
  rw [← intervalIntegral.integral_of_le h01]
```

If the theorem name is not in scope in the exact file where you place this, the shortest fix is to import a file that already brings atomless singleton lemmas into scope.  In this repo, `IntervalDomain.lean` imports Lebesgue basics, and many interval files already use nearby lemmas such as `MeasureTheory.Ioo_ae_eq_Ioc`; adding `Mathlib.MeasureTheory.Integral.IntervalIntegral.IntegrationByParts` is the important extra import for IBP.

## 2. Core operator theorem

Here is the theorem I would add first.  It deliberately takes the kernel regularity as explicit hypotheses.  Once this compiles, make a second theorem that discharges the `hK_*` inputs from the existing smoothness of `intervalNeumannFullKernel` for `0 < t`.

```lean
import ShenWork.Paper2.IntervalConjugateDuhamelMap
import Mathlib.MeasureTheory.Integral.IntervalIntegral.IntegrationByParts

open MeasureTheory Set
open scoped Topology Real Interval

noncomputable section

namespace ShenWork.IntervalConjugateDuhamelMap

open ShenWork.IntervalDomain
open ShenWork.IntervalNeumannFullKernel

private theorem integral_intervalMeasure_one_eq_intervalIntegral (f : ℝ → ℝ) :
    (∫ y, f y ∂ intervalMeasure 1) = ∫ y in (0 : ℝ)..1, f y := by
  have h01 : (0 : ℝ) ≤ 1 := by norm_num
  unfold intervalMeasure intervalSet
  rw [← MeasureTheory.restrict_Ioc_eq_restrict_Icc
    (μ := volume) (a := (0 : ℝ)) (b := (1 : ℝ))]
  rw [← intervalIntegral.integral_of_le h01]

/-- Operator-level integration by parts for the conjugate Neumann kernel.

This is the clean core statement: the endpoint conditions are on `Q`, and all
analytic differentiability/integrability inputs are explicit.  For the production
flux theorem, discharge `hQ0` and `hQ1` using
`truncatedChemFluxLifted_zero_left/right` and discharge the regularity hypotheses
from the positive-time regularity package. -/
theorem intervalConjugateKernelOperator_eq_intervalFullSemigroupOperator_deriv_core
    (t x : ℝ) (Q : ℝ → ℝ)
    (hQ0 : Q 0 = 0)
    (hQ1 : Q 1 = 0)
    (hQ_cont : ContinuousOn Q (Set.Icc (0 : ℝ) 1))
    (hQ_deriv : ∀ y ∈ Set.Ioo (0 : ℝ) 1,
      HasDerivAt Q (deriv Q y) y)
    (hQ_deriv_int : IntervalIntegrable (deriv Q) volume (0 : ℝ) 1)
    (hK_cont : ContinuousOn
      (fun y : ℝ => intervalNeumannFullKernel t x y)
      (Set.Icc (0 : ℝ) 1))
    (hK_deriv : ∀ y ∈ Set.Ioo (0 : ℝ) 1,
      HasDerivAt
        (fun y' : ℝ => intervalNeumannFullKernel t x y')
        (deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y)
        y)
    (hK_deriv_int : IntervalIntegrable
      (fun y : ℝ => deriv
        (fun y' : ℝ => intervalNeumannFullKernel t x y') y)
      volume (0 : ℝ) 1) :
    intervalConjugateKernelOperator t Q x =
      intervalFullSemigroupOperator t (deriv Q) x := by
  classical
  let K : ℝ → ℝ := fun y : ℝ => intervalNeumannFullKernel t x y
  let K' : ℝ → ℝ := fun y : ℝ =>
    deriv (fun y' : ℝ => intervalNeumannFullKernel t x y') y
  have h01 : (0 : ℝ) ≤ 1 := by norm_num

  have hK_cont_u : ContinuousOn K [[(0 : ℝ), 1]] := by
    simpa [K, Set.uIcc_of_le h01] using hK_cont
  have hQ_cont_u : ContinuousOn Q [[(0 : ℝ), 1]] := by
    simpa [Set.uIcc_of_le h01] using hQ_cont

  have hK_deriv_u :
      ∀ y ∈ Set.Ioo (min (0 : ℝ) 1) (max (0 : ℝ) 1),
        HasDerivAt K (K' y) y := by
    intro y hy
    have hy01 : y ∈ Set.Ioo (0 : ℝ) 1 := by
      simpa [min_eq_left h01, max_eq_right h01] using hy
    simpa [K, K'] using hK_deriv y hy01

  have hQ_deriv_u :
      ∀ y ∈ Set.Ioo (min (0 : ℝ) 1) (max (0 : ℝ) 1),
        HasDerivAt Q (deriv Q y) y := by
    intro y hy
    have hy01 : y ∈ Set.Ioo (0 : ℝ) 1 := by
      simpa [min_eq_left h01, max_eq_right h01] using hy
    exact hQ_deriv y hy01

  have hibp :
      (∫ y in (0 : ℝ)..1, K y * deriv Q y)
        = K 1 * Q 1 - K 0 * Q 0
          - ∫ y in (0 : ℝ)..1, K' y * Q y := by
    simpa [K, K'] using
      (intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivAt
        (a := (0 : ℝ)) (b := (1 : ℝ))
        (u := K) (v := Q) (u' := K') (v' := deriv Q)
        hK_cont_u hQ_cont_u hK_deriv_u hQ_deriv_u
        hK_deriv_int hQ_deriv_int)

  have hsign :
      - (∫ y in (0 : ℝ)..1, K' y * Q y)
        = ∫ y in (0 : ℝ)..1, K y * deriv Q y := by
    rw [hibp]
    simp [hQ0, hQ1]

  calc
    intervalConjugateKernelOperator t Q x
        = - (∫ y in (0 : ℝ)..1, K' y * Q y) := by
            unfold intervalConjugateKernelOperator
            rw [integral_intervalMeasure_one_eq_intervalIntegral]
            simp [K']
    _ = ∫ y in (0 : ℝ)..1, K y * deriv Q y := hsign
    _ = intervalFullSemigroupOperator t (deriv Q) x := by
            unfold intervalFullSemigroupOperator
            rw [integral_intervalMeasure_one_eq_intervalIntegral]
            simp [K]
```

If Lean has trouble rewriting the last two `calc` steps because `K` and `K'` are local `let`s, replace `simp [K]` / `simp [K']` by `rfl` after expanding the `let`, or inline `K` and `K'` in the statement of `hsign`.  The proof idea is unaffected.

## 3. Which hypotheses on `Q`?

For the Mathlib lemma named in the question, the best Lean-facing hypotheses are not merely “Lipschitz”.  Use the same W¹,¹-style package already used elsewhere in the repo:

```lean
(hQ_cont : ContinuousOn Q (Set.Icc (0 : ℝ) 1))
(hQ_deriv : ∀ y ∈ Set.Ioo (0 : ℝ) 1, HasDerivAt Q (deriv Q y) y)
(hQ_deriv_int : IntervalIntegrable (deriv Q) volume (0 : ℝ) 1)
(hQ0 : Q 0 = 0)
(hQ1 : Q 1 = 0)
```

This is exactly the kind of hypothesis shape used by the existing sine-coefficient IBP machinery: continuity on the closed interval, pointwise differentiability on the open interval, and integrability of `deriv Q`.

Mathematically, absolute continuity is the natural minimal condition.  In Lean, however, if you want to use

```lean
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivAt
```

then you must provide pointwise `HasDerivAt` on the interior.  Lipschitz is useful upstream because it implies continuity and absolute continuity, and it gives differentiability almost everywhere, but by itself it does **not** directly satisfy the `HasDerivAt` hypothesis everywhere on `Ioo 0 1`.  If your flux is only known to be Lipschitz, either strengthen the theorem statement to the W¹,¹/pointwise package above, or prove/use an absolutely-continuous IBP theorem stated with an a.e. derivative representative and then separately show `deriv Q` agrees a.e. with that representative.

For the flux application, the endpoint hypotheses should be discharged by the lemmas you already have:

```lean
truncatedChemFluxLifted_zero_left
truncatedChemFluxLifted_zero_right
```

or their primed variants, depending on the branch.

## 4. Can the `intervalMeasure` issue be avoided?

Yes.  The recommended theorem should never reason directly about IBP under `intervalMeasure 1`.  Keep the operator definitions unchanged, but start the proof by rewriting both sides:

```lean
rw [integral_intervalMeasure_one_eq_intervalIntegral]
```

on the relevant integrals.  Then the proof is a standard interval-integral proof.  This is cleaner than trying to restate Mathlib's IBP theorem for `volume.restrict (Set.Icc 0 1)`.

A good long-term cleanup is to add one public lemma near `IntervalDomain.lean`, perhaps named

```lean
intervalMeasure_one_integral_eq_intervalIntegral
```

or, more generally,

```lean
intervalMeasure_integral_eq_intervalIntegral_of_nonneg
```

for `0 ≤ L`.  For the current target, the specialized `L = 1` lemma is enough.

## 5. Sign convention

There is no hidden sign error.  With `u = K` and `v = Q`, Mathlib's IBP lemma says

```lean
∫ K * Q' = K 1 * Q 1 - K 0 * Q 0 - ∫ K' * Q
```

The boundary term vanishes because `Q 0 = Q 1 = 0`, so

```lean
∫ K * Q' = - ∫ K' * Q
```

The definition

```lean
intervalConjugateKernelOperator t Q x
  = -∫ y, K' y * Q y ∂ intervalMeasure 1
```

therefore equals the semigroup applied to `deriv Q`:

```lean
intervalFullSemigroupOperator t (deriv Q) x
  = ∫ y, K y * deriv Q y ∂ intervalMeasure 1
```

The leading minus in `intervalConjugateKernelOperator` is exactly what makes the signs match.  Also note that the boundary term uses `K y * Q y`, not `K'_y` or a Neumann boundary derivative.  You do **not** need a kernel boundary condition for this identity; the endpoint vanishing of `Q` is sufficient.
