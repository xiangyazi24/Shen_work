## Short answer

The smallest honest next step is **not** to push the Task274 bootstrap surface back
into the final route.  The current preferred chi-zero route is already the
Wdata/tower/tight-ledger line:

```lean
ShenWork.IntervalPicardTowerSupply.from_cone_construction
```

and the route-facing aliases:

```lean

ShenWork.IntervalPicardTowerSupply.paper2_theorem_1_1_chiZero_from_coneSupplyNarrow
```

The Task274 hsrc0-bootstrap/no-Hu surface is correct and useful as a local
resolver-source reduction, but it is **not thinner than the existing final
Wdata-tight route**.  In particular, the Task274 route-facing surface still asks
for `S` or `I` and `hpde_u`, while `HresWiring.restartAndFrontierCore_of_wdata_tight`
already builds the restart/core package directly from cone facts + `Wdata` +
`hiter_cont`.

So I recommend either:

1. **Commit A (small, useful): add a final headline alias file** exposing the current
   preferred chi-zero theorem under a clear Paper2 namespace/name, without touching
   `ShenWork.lean`.
2. **Commit B (optional compatibility): add a small Hres-core → Task274-hsrc0 bridge**
   only if another pane needs to compare the Task274 surface against Hres/Wdata.
   Do not make it the main route.

Do **not** integrate the Hu finite-cover work into the main chi-zero route yet; it
is a consumer-side envelope package for the older `HuCoeff` path and is dominated
by Task270/Task274's `limitCoeff`+`hsrc0` path for this branch.

## Hu finite-cover direction

Do not integrate it into the main chi-zero route now.

Current `IntervalHuRestartCoeffFiniteCover.lean` proves a **consumer-side** result:

```lean
huRestartCoeff_henv_of_finiteCover
```

and defines finite-cover variants of `HuCoeff` inputs.  It does not prove the
actual topological finite subcover.  More importantly, it targets the older path
where coefficients are selected from an already-carried
`HasTimeNeighborhoodSpectralAgreement`.  Task270/Task274 already avoid that path
for the χ₀=0 resolver-source package by choosing `bc := limitCoeff` and deriving
the envelope directly from `hsrc0`.

Use finite-cover work only if the project wants to maintain the HuCoeff branch for
PPID/general-χ bookkeeping.  It is not the smallest route toward the current
χ₀=0 headline.
