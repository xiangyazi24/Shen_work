## Findings

### 1. Does the new route still consume `hpde_u` indirectly?

From the theorem shapes described, the intended new route does **not** need `hpde_u` if the proofs really only use:

- `intervalDomainClassicalRegularity D.T D.u (coupledChemicalConcentration p D.u)`,
- `mildChemical_nonneg`,
- `sourceCoeffQuadraticDecay_of_mildSolution_of_closedC2_neumann`, and
- resolver spectral identities such as `intervalNeumannResolverRLap_elliptic_identity` / `resolver_lift_deriv_eq_resolverGrad...`-style lemmas.

That is an honest reduction: `intervalDomainClassicalRegularity` is a regularity bundle, not a PDE assertion. Its definition records spatial/time regularity, Neumann/tendsto data, and joint continuity fields; it does not assert the parabolic PDE. In the committed `IntervalDomain.lean`, `intervalDomainClassicalRegularity` is just a `Prop` made of regularity and continuity conjuncts for `u` and `_v`, including open/closed spatial `C²`, time differentiability/continuity, one-sided Neumann behavior, and joint continuity of the fields. It does not contain `intervalDomain.timeDeriv u = ...`.

The safe route for `chemFluxLifted_hasDerivWithinAt_coupledChemDivSourceLift_open_of_mildRegularity` is:

1. derive `SourceCoeffQuadraticDecay p (D.u s)` from the closed `C²`/Neumann part of `hreg` using `sourceCoeffQuadraticDecay_of_mildSolution_of_closedC2_neumann`;
2. use resolver APIs to identify `deriv (intervalDomainLift (coupledChemicalConcentration p D.u s))` with `resolverGradReal` on `Ioo 0 1`, and the resolver second derivative/Laplacian representative with `p.μ * v - p.ν * u^γ`;
3. differentiate the flux by product/quotient/rpow calculus under `hreg` and nonnegativity of `v`.

This does not require `hpde_u`.

The hidden circularity danger is importing and applying classical-solution helper lemmas whose statements still require `IsPaper2ClassicalSolution`. In particular, avoid using the following as black boxes inside the new regularity bridge:


Some of these ultimately route through `hsol.pde_u` or through an `IsPaper2ClassicalSolution` package that contains `pde_u`. They are fine for classical-consistency theorems, but not for reducing Task308’s `hclassical` input.

The `v`-elliptic side is less dangerous: the resolver concentration is definitionally `intervalNeumannResolverR p (D.u s)`. So the elliptic identity should come from resolver spectral APIs plus source decay, not from `hsol.pde_v`. That distinction matters.

### 2. Is the `hreg + representative` surface an honest non-PDE reduction of Task308?

Yes, with one qualification.

The surface

```lean
hreg : intervalDomainClassicalRegularity D.T D.u
  (coupledChemicalConcentration p D.u)
```

plus an endpoint-insensitive representative

```lean
∃ Gdiv : ℝ → ℝ,
  ContinuousOn Gdiv (Set.Icc (0 : ℝ) 1) ∧
  Set.EqOn (coupledChemDivSourceLift p D.u s) Gdiv (Set.Ioo (0 : ℝ) 1)
```

is the correct non-PDE replacement for `hclassical` in the integrated bridge. The representative is exactly the right endpoint-safe object: `coupledChemDivSourceLift` itself is a literal lift of an ordinary derivative of a zero-extended flux, so endpoint global continuity is generically false. Asking only for a closed-interval continuous representative that agrees on `Ioo 0 1` matches the already landed physical-representative pattern.

The qualification: `hreg` alone is not enough to produce the representative equality if `_v` is arbitrary. It becomes enough only in the specialized resolver situation

```lean
_v = coupledChemicalConcentration p D.u
```

plus the positivity/nonnegativity and source-decay facts needed to identify the resolver’s gradient and second derivative with the product-rule expression. In other words:

- `hreg` gives differentiability/continuity of the factors;
- `D.hpos` / `mildChemical_nonneg` keeps the rpow denominator away from zero;
- `sourceCoeffQuadraticDecay_of_mildSolution_of_closedC2_neumann` plus resolver identities supply the elliptic replacement `v_xx = μ v - ν u^γ` for `v = R(u)`.

Without the resolver identity or source decay, the physical representative

```lean
u_x * v_x / (1+v)^β
+ u * (μ*v - ν*u^γ) / (1+v)^β
- β*u*v_x^2 / (1+v)^(β+1)
```

is not generally the derivative of `u * v_x / (1+v)^β`. For a generic regular pair `(u,v)`, the second term should contain `u * v_xx`, not `u * (μv - νu^γ)`.

So the surface is honest when stated for `D` and `coupledChemicalConcentration p D.u`; it would be false or underpowered as a theorem for arbitrary `u v` with only `intervalDomainClassicalRegularity T u v`.

## Why `hreg` alone is insufficient

There are two insufficiencies if the theorem is stated too generally:

1. For arbitrary `v`, the physical representative with `μ v - ν u^γ` is not the derivative of the flux. You need `v = intervalNeumannResolverR p u` plus source decay to get the elliptic identity.

2. For a closed-interval continuous representative, you need endpoint control of ordinary derivatives. Interior `ContDiffOn` alone gives continuity only on `Ioo 0 1`. The endpoint values and one-sided Neumann/tendsto fields in `hreg` are what make the closed `Icc 0 1` continuity proof possible.

So the smallest theorem should not be stated as:

```lean
intervalDomainClassicalRegularity T u v → ∃ Gdiv, ...
```

It should be stated for the actual resolver pair and the mild data:

```lean
D : GradientMildSolutionData p u₀
hreg : intervalDomainClassicalRegularity D.T D.u (coupledChemicalConcentration p D.u)
```

or, if generic, it must explicitly carry the elliptic replacement/equality hypotheses.
