The **single most likely remaining hole on the Jensen/final-assembly side** is the **closed-endpoint strict positivity upgrade**:

```lean
∀ t, 0 < t → t ≤ T → ∀ x : intervalDomainPoint, 0 < u t x
```

The reducer

really requires pointwise strict positivity for every `intervalDomainPoint`, hence for `x = 0` and `x = 1` as well as the open interior. It is not enough for the matched-divergence weak comparison to prove only `u > 0` a.e. or only on `Ioo 0 1`.

If the weak comparison proves `u ≥ w` in `L²` only, this is still closable, but one additional bridge must be explicit:

1. `∫ (w - u)_+^2 = 0` gives `(w - u)_+ = 0` a.e.

3. the Neumann heat/square barrier `w` is strictly positive on closed `[0,1]`, including endpoints.
4. therefore `u(t,x) ≥ w(t,x) > 0` for every `x : intervalDomainPoint`.

That endpoint/continuity upgrade is the one place I would make Codex produce a named lemma, not leave implicit.

## 1. Jensen assembly: endpoint strict positivity is load-bearing

The current strict-positivity reducer is genuinely closed-domain. Its hypothesis is:

```lean

```

and it returns `TruncatedJensenStrictPosDataFor T u`. This is visible in `IntervalTruncatedStrictPositivityProducerV6.lean`: the theorem takes `hpos` as an input, quantified over `x : intervalDomainPoint`, and later uses it as `have hU : 0 < u t x := hpos t ht htT x`.

So the weak comparison must not stop at any of these weaker conclusions:

```lean

∀ x ∈ Ioo (0:ℝ) 1, 0 < intervalDomainLift (u t) x
∀ x, x ∈ interior → 0 < u t x
```

The target needed by the reducer is the closed-subtype version:

```lean

    {T : ℝ} {u : ℝ → intervalDomainPoint → ℝ} :
    ∀ t, 0 < t → t ≤ T → ∀ x : intervalDomainPoint, 0 < u t x := ...
```

The matched-divergence weak comparison can still produce this, but only if the final bridge is:


The key subtlety is that `hwpos` must be **closed interval positivity** of the Neumann heat barrier. For the Neumann full kernel this should be true: reflecting the heat kernel gives strict positivity even at `x = 0` and `x = 1`. But it must be the lemma actually used. If the existing strict-heat lemma only states positivity on `Ioo 0 1`, it is insufficient for `truncatedJensenStrictPosDataFor_of_strictPos`.

The seed route itself is not circular. In the strictPos reducer, the only strict-positivity input is used at the final target point `u t x`, to choose the per-point discount.

## 2. Do energy and Jensen compose into `hpos`?

They compose only if the workflow is:

```text
energy  →  u ≥ 0
mass/seed + matched weak comparison  →  closed pointwise u > 0
closed pointwise u > 0  →  TruncatedJensenStrictPosDataFor
```

Do **not** rely on this circular shape:

```text
TruncatedJensenStrictPosDataFor → u > 0 → TruncatedJensenStrictPosDataFor
```

`truncatedJensenStrictPosDataFor_of_strictPos` is a reducer from already-proved closed strict positivity to Jensen data. It is not, by itself, a strict-positivity producer. The actual strict positivity must come from the weak comparison or another independent argument.

Once closed strict positivity is available, the final assembly side is type-correct in principle:

- energy supplies the nonnegativity/negative-part field;
- closed strict positivity supplies the Jensen witness field;
- the final classical/mild solution positivity field can be populated from the same closed strict positivity theorem, not from an a.e. statement;
- no extra positivity assumption is needed for truncation removal beyond `u ≥ 0` pointwise, because `ρ₊(u) = u` follows directly from nonnegativity.
