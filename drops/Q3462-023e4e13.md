# Q3462 route audit: producing `H1EnergyDerivativeInitialWindowIntegrableBefore`

## Verdict

No. At main commit `d7857cfa`, I do not see an existing theorem chain that honestly produces

```lean
H1EnergyDerivativeInitialWindowIntegrableBefore u T
```

from the current `IsPaper2ClassicalSolution`, mild/Picard/local-restart, initial H¹ trace, or P3/Moser machinery without carrying essentially the same near-zero derivative integrability input, re-entering the zero-start H²/lap route, or using a downstream boundedness result.

Task82 correctly moved the bottleneck to the scalar level:

```lean
def H1EnergyDerivativeInitialWindowIntegrableBefore
    (u : ℝ → intervalDomainPoint → ℝ) (T : ℝ) : Prop :=
  ∀ {b : ℝ}, 0 ≤ b → b < T →
    IntervalIntegrable (fun r => deriv (H1energy u) r) volume (0 : ℝ) b
```

This is weaker than zero-start `lapL2sq` continuity, but it is still a real near-zero absolute-continuity/integrability statement. Strict positive-time regularity alone does not imply it.

## Existing source audit

### 1. H¹ strict RHS route

The current file

```lean
ShenWork/Paper2/IntervalChiNegH1StrictRHSIntegrability.lean
```

already proves the correct adapters:

```lean
H1IdentityRHSInitialWindowIntegrableBefore_of_deriv_initialWindow
H1IdentityRHSIntegrableBefore_of_componentsStrictBefore_derivInitial
boundedBefore_of_H1supBoundDI_classicalChemRep_strictRHS_derivInitial_before
```

These are pure wiring. They consume `H1EnergyDerivativeInitialWindowIntegrableBefore`; they do not produce it.

### 2. H¹ derivative/RHS integrability file

`IntervalChiNegH1DerivativeIntegrability.lean` has the one-way theorem

```lean
H1_derivInt_of_identityRHS_intervalIntegrable
```

and the packaged version

```lean
H1_derivInt_of_identityRHSIntegrableBefore
```

These produce integrability of `deriv (H1energy u)` from integrability of the explicit H¹ RHS. But in the new route, RHS integrability on zero-start windows is produced *from* derivative initial-window integrability. Using these theorems to produce the new bottleneck would be circular unless there is an independent RHS-integrability producer.

### 3. H¹ initial trace data

The H¹ initial endpoint route gives scalar energy right-continuity, for example:

```lean
H1energy_continuousWithinAt_zero_of_initialTraceEnergy
H1energy_continuousOn_before_of_uxxL1Cont_initialTraceEnergy
```

That is not enough. Continuity of `H1energy u` at zero does not imply that `deriv (H1energy u)` is integrable on `[0,b]`. The missing statement is absolute continuity or an integrable derivative/majorant near zero.

### 4. Strict positive-time scalar/H¹ route

Strict positive-time identities and strict component continuity give integrability on `[a,b]` for `0 < a`. They do not give integrability on `[0,b]`. A function can be continuous on every `[a,b]`, `0 < a`, and fail to be integrable at zero.

### 5. P3/Moser initial-window machinery

The closest analogue is the P3/Moser theorem

```lean
intervalDomain_lpPDECombinedInitialWindowIntegrability_of_moserDerivativeInitial
```

but it is not a producer from classical data. It assumes

```lean
IntegratedMoserEnergyDerivativeInitialWindowIntegrability intervalDomain u T p0
```

and converts that derivative initial-window integrability into a combined PDE initial-window residual. Also, the Moser energy is `integratedMoserEnergy`; it is not the H¹ energy `H1energy`. You cannot reuse it by setting `p0 = 2` or by treating Lp energy as H¹ energy.

### Target 1: integrable derivative proxy

This is the Lean-shortest frontier. It is useful when a construction or integrated H¹ identity naturally produces an integrable function `g` that agrees with `deriv (H1energy u)` on `(0,b]`.

The proxy may be an explicit construction-level RHS or an abstract derivative
field from an integrated H¹ identity. -/
structure H1EnergyDerivativeInitialWindowProxyBefore
    (u : ℝ → intervalDomainPoint → ℝ) (T : ℝ) : Prop where
  g : ℝ → ℝ

    IntervalIntegrable g volume (0 : ℝ) b

theorem H1EnergyDerivativeInitialWindowIntegrableBefore_of_proxy

  exact H1_deriv_intervalIntegrable_of_eq_on_Ioc

This should be very low-risk: it reuses the existing congr lemma in `IntervalChiNegH1DerivativeIntegrability.lean`.

## Optional integrated H¹ energy FTC package

If a construction-level proof naturally gives an absolute-continuity/FTC statement, use the proxy package above rather than making the downstream route depend on a bulky FTC record. A possible construction-facing record is:

```lean
structure H1EnergyInitialWindowFTCBefore
    (u : ℝ → intervalDomainPoint → ℝ) (T : ℝ) : Prop where
  g : ℝ → ℝ
  g_intervalIntegrable : ∀ {b : ℝ}, 0 ≤ b → b < T →
    IntervalIntegrable g volume (0 : ℝ) b
  deriv_eq : ∀ {b : ℝ}, 0 ≤ b → b < T →
    ∀ᵐ r ∂volume.restrict (Set.Ioc (0 : ℝ) b),
      deriv (H1energy u) r = g r
  energy_eq : ∀ {b : ℝ}, 0 ≤ b → b < T →
    H1energy u b - H1energy u 0 = ∫ r in (0 : ℝ)..b, g r
```

For the Task82 bottleneck, only `g_intervalIntegrable` plus the derivative equality is needed. The `energy_eq` field is useful for mathematical auditing but not necessary for the integrability adapter.

## Why current APIs do not produce the bottleneck

- `IsPaper2ClassicalSolution` gives positive-time regularity; it does not give a zero-start absolute-continuity theorem for `H1energy`.
- Mild/Picard local restart APIs are positive-time/restart-based. They do not by themselves supply an integrable majorant for `deriv (H1energy u)` on `[0,b]`.
- Initial H¹ trace gives endpoint convergence/compatibility of `H1energy`, not integrability of its derivative.
- P3/Moser derivative initial-window theorems are about `integratedMoserEnergy`, not `H1energy`.
- Existing H¹ RHS integrability theorems either assume full RHS integrability or closed-window component continuity; neither is an honest producer here.

## Circular or false routes to avoid

1. **Using `H1_derivInt_of_identityRHSIntegrableBefore` to produce the bottleneck.** That theorem needs `H1IdentityRHSIntegrableBefore`, which the Task82 route obtains from the bottleneck. This is circular.

2. **Using `H1IdentityRHSComponentsContinuousBefore`.** It includes zero-start continuity of `lapL2sq`, reintroducing the H²/lap endpoint trace that route C is designed to avoid.

3. **Using `H1LapComponentZeroRightContinuous`, `H1LiftDeriv2ZeroSlabRepresentativeBefore`, or `H1ZeroStartPhysicalRHSDataBefore`.** Those are route-B atoms, not a route-C producer.

4. **Using `IsPaper2BoundedBefore` or the final H¹ bound to prove a near-zero majorant.** That would prove a prerequisite from the conclusion path.

5. **Using `H1SupBoundDIDataBefore` as an integrability theorem.** It is pointwise/existential data for a differential inequality. It does not provide interval-integrability of a fixed RHS split or of `deriv (H1energy u)`.

6. **Using `H1SupBoundSqrtRHSIntegrableBefore` as if produced.** It already contains full RHS interval-integrability, hence it contains the bottleneck in stronger form.

7. **Taking a cutoff limit from strict slabs without a uniform near-zero L¹ bound.** Strict positive-time continuity does not rule out nonintegrable behavior like `1/t` near zero.

8. **Identifying P3/Moser derivative integrability with H¹ derivative integrability.** `integratedMoserEnergy` and `H1energy` are different scalar energies.
