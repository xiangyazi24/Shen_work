## Risks

1. **Header/doc drift.** `IntervalDomainLedgerSweep.lean` still says `hpde_u` is residual in the old audit block. If adding the strict ledger, update only the local docs in that file to avoid misleading future agents. This is not a Lean risk, but it is coordination-important.

2. **Name collision / ambiguity.** There is already `ReducedLimitRegularityInputs`; choose a distinct name such as `StrictReducedLimitRegularityInputs`. Avoid overloading `Reduced...` with a second namespace alias.

3. **Using the wrong hpde theorem.** Do not call `MildLocalChi0.hpde_u_of_inputs_localized_data`, because it expects a full `LimitRegularityInputs`. Use `MildLocalChi0.hpde_u_of_localized_limit_spectral_data`, the field-level theorem.

4. **Continuous subtype vs lifted extension.** The strict ledger should carry `hu₀_cont : Continuous u₀`, exactly like current ledgers. Do not strengthen this to `Continuous (intervalDomainLift u₀)`, which is false for positive endpoint data.

5. **BddOn vs L1ContOn.** Keep `hsrc0` exactly as in the current ledgers:
   ```lean
   ShenWork.IntervalPicardLimitRestartBdd.DuhamelSourceBddOn
     (ShenWork.IntervalPicardLimitBddProducer.patchedSource p u₀ D.u) D.T
   ```
   Do not switch it to the older `DuhamelSourceL1ContOn`; the current localized subtype route consumes `DuhamelSourceBddOn patchedSource`.

6. **Hvpos/Hvsrc remain.** Even after this strict ledger, the local side still carries:
   ```lean
   Hvsrc
   Hvpos
   ```
   Since Task224 is working on `Hvpos`, dm should not start a competing Hvpos interface in this file. `Hvsrc` is the next independent residual after Hvpos, but it is not a safer small step than the strict sweep.
