→ WCE 1
  -- source-from-solution pass, m = 1

There is no negative-index `WSE`. The source pass is not allowed at `m = 0`; it is typed with `1 ≤ m`, and mathematically the chem-div source cannot be controlled from coefficient-boundedness alone. So the **separate WCE0 → WCE1 heat-smoothing base is mandatory**.

## 1. Exact index trace

Use the following notation:

```text
WCE m  := solution cosine coefficients have order m envelope
WSE q  := source cosine coefficients have order q envelope
```

The nonlinear source pass is:

```text
WCE m → WSE (m - 1),     for 1 ≤ m.
```

The parabolic Duhamel/restart heat gain is:

```text
WSE q → WCE (q + 2).
```

Therefore:

### Base 0

```text
WCE 0
```

This is just coefficient boundedness from bounded continuous slices. In the current file this is the role of:

```lean
windowCoefficientEnvelope_zero_of_bound
```

It provides a uniform order-0 coefficient envelope from `S.hbound`/`S.hcont`.

### Base heat smoothing

You cannot run source-from-solution at `m = 0`. Instead, use the separate first heat-smoothing lemma:

```text
WCE 0 + bounded B-form source/flux data → WCE 1.
```

This is the Q4309 base: positive-time heat smoothing plus bounded B-form Duhamel gives one spatial derivative / first coefficient decay. This is the first real bootstrap step and is not the same as the later source-from-solution pass.

### Pass 1

```text
WCE 1 → WSE 0.
```

Interpretation: once the solution has one spatial derivative, the B-form source coefficients are at least uniformly bounded.

For chem-div, this is the first IBP level: the divergence coefficient has a `kπ` prefactor, and the bounded derivative of the flux cancels that growth. This is exactly why WCE1 is the first legal source input.

### Heat gain 1

```text
WSE 0 → WCE 2.
```

A bounded source coefficient gives solution coefficients with two powers from the heat resolvent/Duhamel denominator.

### Pass 2

```text
WCE 2 → WSE 1.
```

Now the source has one power of decay.

### Heat gain 2

```text
WSE 1 → WCE 3.
```

### Pass 3

```text
WCE 3 → WSE 2.
```

This is the first source envelope strong enough for the eigenvalue-weighted restart coefficient summability, because the source side must be summable after the heat ODE is weighted by `λ_k`.

So the honest finite chain is:

```text
WCE0
  → WCE1
  → WSE0
  → WCE2
  → WSE1
  → WCE3
  → WSE2
  → ∑ λ_k |c_k| < ∞.
```

## 2. What does the final summability step consume?

The final step must consume a **source-side summable envelope**, or an equivalent package that contains it.

Mathematically, for restart coefficients

```text
c_k(τ) = e^{-νλ_k τ} a0_k + ∫₀^τ e^{-νλ_k(τ-s)} s_k(s) ds,
```

multiplying by `λ_k` gives

```text
λ_k |c_k(τ)|
  ≤ λ_k e^{-νλ_k τ} |a0_k| + sup_s |s_k(s)|.
```

The first term is summable for any bounded `a0_k`, thanks to exponential damping. The second term is summable only if the source coefficients themselves are ℓ¹. Thus:

```text
WSE0 is not enough.
WSE1 is not enough.
WSE2 is enough, since ∑ k^{-2} converges.
```

So if a lemma is named something like:

```lean
restartCoeff_eigenvalue_weighted_summable_of_pass2_envelope
```

then the correct conceptual input is `WSE 2`, or a package equivalent to `∑ |s_k| < ∞` uniformly on the restart window. It should **not** be read as consuming merely `WCE 2`.

In the current visible code, the actual wired theorem is:

```lean
localRestartCoeff_eigenvalue_summable
    (hτ : 0 < τ)
    (ha₀ : ∀ n, |a₀ n| ≤ M)
    (src : DuhamelSourceTimeC1 a) :
    Summable (fun n : ℕ =>
      unitIntervalCosineEigenvalue n * |localRestartCoeff a₀ a τ n|)
```

That theorem consumes `DuhamelSourceTimeC1`, not a raw `WSE 2` argument. But `DuhamelSourceTimeC1` is precisely the higher package that must include the source regularity/summability needed by this estimate. So the audit conclusion is:

```text
final summability consumes source-side regularity, not solution-side WCE2 alone.
```

If a future proof tries to feed it only `WCE2`, that would be a real bug. The safe feed is either:

```lean
WSE 2
```

or:

```lean
DuhamelSourceTimeC1 (bFormSourceCoeffs p S.u)
```

where `DuhamelSourceTimeC1` has already been built from the source ladder.

## 3. Well-foundedness and ordering

The induction order is well-founded provided the implementation observes this dependency order:

```text
1. WCE0 from boundedness.
2. WCE1 from positive-time heat smoothing / bounded B-form Duhamel.
3. WSE0 from WCE1.
4. WCE2 from WSE0.
5. WSE1 from WCE2.
6. WCE3 from WSE1.
7. WSE2 from WCE3.
8. DuhamelSourceTimeC1 / pass-2 summability from WSE2 plus source time-continuity.
9. eigenvalue-weighted ℓ¹ representation R.
```

Every source pass consumes a WCE that has already been produced:

```text
WCE1 is produced before WSE0.
WCE2 is produced before WSE1.
WCE3 is produced before WSE2.
```

No pass asks for its own output, and no pass asks for a negative source envelope.

The only nontrivial base is step 2. It is not optional. It is exactly the heat-smoothing base from bounded data. Without it, the attempted chain would start:

```text
WCE0 → WSE(-1)
```

which is both ill-typed and analytically false.

## 4. Potential failure modes to avoid

### Failure mode A: claiming `WCE0 → WSE0`

This is not generally valid for the chem-div source. Bounded `u`/bounded coefficients alone do not bound the divergence source coefficients. The B-form/conjugate formulation gives a bounded flux leg, but the source coefficient after taking divergence has the `kπ` factor. You need at least one spatial derivative / one IBP level, i.e. `WCE1`.

### Failure mode B: stopping at `WCE2`

`WCE2` is not the final source summability needed for `∑ λ_k |c_k|`. You still need:

```text
WCE2 → WSE1 → WCE3 → WSE2.
```

### Failure mode C: treating `DuhamelSourceTimeC1` as free

The current `restartSliceCoeff_eigenvalueSummable` theorem hides the source envelope inside the `DuhamelSourceTimeC1` hypothesis. That is fine, but then the source ladder must really construct `DuhamelSourceTimeC1`. Do not count the summability theorem as discharging the source ladder by itself.

### Failure mode D: ignoring the positive floor

For high-order source-from-solution passes with real exponents `α, γ ≥ 1`, the Nemytskii bounds for `u^α` and `u^γ` at high derivative order need a positive floor if negative powers such as `u^{γ-j}` appear. The current comments correctly identify the need for the strict-positive/compact floor. Without that floor, the high-order source pass can fail for non-integer exponents.

## 5. Final answer

The route is **gap-free in index bookkeeping** if implemented as:

```text
WCE0
  → WCE1                         -- separate heat-smoothing base
  → WSE0                         -- source pass m=1
  → WCE2                         -- heat +2
  → WSE1                         -- source pass m=2
  → WCE3                         -- heat +2
  → WSE2                         -- source pass m=3
  → DuhamelSourceTimeC1/pass2     -- source-side ℓ¹ package
  → ∑ λ_k |c_k| < ∞.
```

There is **no fatal ordering issue** and no negative-index envelope, but there are two hard requirements:

1. The base `WCE0 → WCE1` must be a separate heat-smoothing lemma from bounded data. It cannot be replaced by the source pass.
2. The final restart summability step must be fed by `WSE2` or by a package equivalent to source ℓ¹ summability, such as the current `DuhamelSourceTimeC1`; it cannot be fed by `WCE2` alone.

With those conditions, the chain is a well-founded finite induction and reaches the eigenvalue-weighted ℓ¹ representation `R`.
