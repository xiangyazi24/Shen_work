# Help

This drop confirms the `--help` request and records the expected workflow for future GitHub/Lean proof requests.

## Git-drop contract

For requests with a git-drop block, I will:

1. Write the complete response to the specified path under `drops/`.
2. Ensure the first line is exactly the requested sentinel line.
3. Use the requested branch, normally `chatgpt-scratch`.
4. Create the file if it is absent, or overwrite/update it if it already exists.
5. Commit the change to `xiangyazi24/Shen_work` and report the commit SHA.

I will not use sandbox files or `/mnt/data` for these deliveries.

## What to include in a proof-target request

A good request should include:

* the repository and branch to inspect;
* the intended theorem name and expected statement, if known;
* likely files/imports;
* constraints such as `no sorry`, `no axiom`, `source-side only`, or `avoid classical-solution hypotheses`;
* whether the desired answer should be a compiling theorem, a diagnosis that the theorem is currently blocked, or both.

## Expected response format for Lean audit questions

For a proof-target audit, I will usually write:

* verdict: whether the proposed theorem is provable now;
* smallest non-circular theorem statement;
* exact file and imports;
* compiling Lean proof when available;
* obstruction explanation when not available;
* next theorem/lemma needed to unblock the target.

Lean code will be fenced as:

```lean
import ...

open ...

namespace ...

-- theorem here

end ...
```

## Failure mode

If the GitHub connector is unavailable or read-only, the required response is:

```text
GIT-DROP FAILED: connector unavailable
```

and I should not fall back to a sandbox artifact.
