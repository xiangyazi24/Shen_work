# Q3785 audit: Task306–308 integrated bridge vs. true `hpde_u`

## Verdict

The new Task306–308 integrated mixed-source Duhamel equality is useful, but it does **not** honestly reduce the `hpde_u` obligation by itself. It is a classical-consistency / integrated-RHS bridge:

```lean
gradientMildChemotaxisDuhamelTerm p u t x
  + gradientMildLogisticDuhamelTerm p u t x
  =
∫ s in (0 : ℝ)..t,
  ((-p.χ₀) *
      unitIntervalSineHeatValue (t - s)
        (sineCoeffs (coupledChemDivSourceLift p u s)) x
    + unitIntervalCosineHeatValue (t - s)
        (coupledLogisticSourceCoeffs p u s) x)
```

That equality rewrites the **value** of the two Duhamel source terms at a fixed `(t,x)`. The current `hpde_u` producers, by contrast, need a local-in-time restart representation of `u`, coefficient time-`C¹`, a coefficient-level source split, and summability/inversion packages. Those are time-differentiation inputs, not consequences of the integrated equality.

The decisive technical mismatch is the Duhamel endpoint. The integrated theorem works by an a.e. in `s` slice bridge and explicitly throws away `s = t` because `t - s = 0`. That is harmless for equality of integrals, but it is **not** harmless for differentiating in `t`: the endpoint term is exactly where the source coefficient `a(t,n)` enters the spectral Duhamel ODE. So an a.e. integrated bridge cannot produce the time derivative field.

### The existing true `hpde_u` producer

`ShenWork/Paper2/IntervalDomainPdeUGeneralChi.lean` has:

```lean

```

Its `Hsource` input is substantially stronger. For every interior `t₀`, it asks for local restart data

```lean
∃ (a₀ : ℕ → ℝ) (M : ℝ) (_ : 0 ≤ M) (_ : ∀ n, |a₀ n| ≤ M)
  (a : ℝ → ℕ → ℝ) (_ : DuhamelSourceTimeC1 a)
  (offset : ℝ) (_ : 0 < t₀ - offset)
  (_ : LogisticCosineFourierData p u t₀)

    (coupledChemicalConcentration p u t₀)),

    u s y = ∑' n, localRestartCoeff a₀ a (s - offset) n
      * cosineMode n y.1) ∧
  (∀ n, a (t₀ - offset) n
    = coupledLogisticSourceCoeffs p u t₀ n
      - p.χ₀ * coupledChemDivSourceCoeffs p u t₀ n) ∧
```

The proof then uses `bForm_timeDeriv_eq_of_local_restart` to get the actual time derivative series, `laplacian_eq_of_rep` for the spatial second derivative, and the logistic/chem-div cosine inversion packages to identify the two source series with the physical terms. This is the right architecture for `hpde_u`.

### The source-form PDE path says the same thing

`ShenWork/Wiener/EWA/SourcePdeU.lean` proves:

```lean
theorem fullSourceCoeff_pde_u
```

from a time-derivative series, a laplacian series, chem/log inversions, and summability. The coefficient-level engine is

```lean
theorem fullSourceCoeff_spectral_ode
```

in `ShenWork/Wiener/EWA/SourceSpectralODE.lean`:

That is the kind of coefficient/time-differentiation fact `hpde_u` needs. The Task306–308 integrated bridge has no analogue of this `HasDerivAt`/coefficient ODE data.

## Hidden circularity risks

Do **not** use `hclassical : IsPaper2ClassicalSolution ...` to prove the missing `hpde_u` field for the same bootstrap target, unless `hclassical` is an externally supplied classical solution. In the local-existence bootstrap, `IsPaper2ClassicalSolution` already includes the PDE content that `hpde_u` is meant to provide. The Task306–308 theorem assumes `hclassical`, so it is safe as a consistency bridge, but dangerous as an upstream PDE producer.

Do **not** treat the mixed sine heat integral as a Neumann cosine restart source. The general-χ `hpde_u` route wants the source coefficient family

```lean
fun s n => coupledLogisticSourceCoeffs p u s n
  - p.χ₀ * coupledChemDivSourceCoeffs p u s n
```

inside `localRestartCoeff`. The integrated theorem’s chem leg is a sine heat value evaluated at `x`, not a local Neumann cosine restart representation of `u` in a neighborhood of `t₀`.

Do **not** use per-time half-step slice data as a time-derivative input. `GradientMildHalfStepRestartData` gives an equality for `D.u t` at that one target time. `hpde_u` needs `∀ᶠ s in 𝓝 t₀, ...`, with a fixed restart anchor and a source family differentiable in the time variable. This is exactly why `HasTimeNeighborhoodSpectralAgreement` exists in `IntervalMildTimeDerivContinuity.lean`.

Do **not** try to revive closed-endpoint gradient/source equalities. The newer physical-representative bridge correctly uses a continuous representative that agrees with the literal chem-div lift on `Ioo 0 1`; endpoint-global statements are still the old trap.

## How this relates to a `GradientMildSolutionData D`

For a target `D : GradientMildSolutionData p u₀`, instantiate the theorem with

```lean
u := D.u
T := D.T
```

but only after you have actual B-form global cosine data for `D.u`:

```lean
hB_global : ∀ t, 0 < t → t ≤ D.T,
  Set.EqOn (intervalDomainLift (D.u t))
    (fun x => ∑' n, localRestartCoeff aInit aB t n * cosineMode n x)
    (Set.Icc (0 : ℝ) 1)

hB_split : ∀ s n,
  aB s n = coupledLogisticSourceCoeffs p D.u s n
    - p.χ₀ * coupledChemDivSourceCoeffs p D.u s n
```

The Task306–308 integrated equality can help justify that this B-form/source representation matches the original gradient mild RHS as a value identity, but it does not itself produce `hB_global`, `hsrcB`, `hB_split`, or the local restart/eigenvalue data. So the correct dependency direction is:

```text
B-form global cosine data + source time-C¹ + source split
  ⟶ Hsource_of_bForm_global_generalChi
  ⟶ hpde_u_of_generalChi_sourceSpectralData
```

not:

```text
integrated mixed gradient-source equality
  ⟶ hpde_u
```

## Bottom line

Task306–308 is good evidence that the gradient mild and mixed spectral source RHSs are compatible for values. It is not a time-differentiation theorem. True `hpde_u` progress should land the B-form/global-cosine-to-`Hsource` producer above, or the equivalent source-form full-coefficient producer using `fullSourceCoeff_timeDeriv_eq`, `fullSourceCoeff_laplacian_eq`, and the chem/log inversion packages.
