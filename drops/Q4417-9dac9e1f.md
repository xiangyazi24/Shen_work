# 2. Exact parity calculation

Let `U` and `V` be global smooth cosine-series representatives of the solution slice and resolver slice. Their endpoint symmetries are

```plain text
U(-x)   = U(x),          U(2-x) = U(x),
V(-x)   = V(x),          V(2-x) = V(x).
```

Differentiating gives

```plain text
V'(-x)   = -V'(x),
V'(2-x)  = -V'(x).
```

Let

```plain text
A(x) := U(x)·(1+V(x))^(-β).
```

Since products and scalar post-compositions preserve double-even parity,

```plain text
A(-x)=A(x),
A(2-x)=A(x).
```

The full flux is

```plain text
Q(x) = A(x)V'(x).
```

Hence

```plain text
Q(-x)  = -Q(x),
Q(2-x) = -Q(x).
```

In particular,

```plain text
Q(0)=0,
Q(1)=0.
```

Differentiating an odd-about-the-endpoint function produces an even-about-the-endpoint function, so

```plain text
S(x):=Q'(x)
```

satisfies

```plain text
S(-x)=S(x),
S(2-x)=S(x).
```

Therefore `S` has Neumann compatibility:

```plain text
S'(0)=0,
S'(1)=0.
```

Equivalently,

```plain text
Q''(0)=0,
Q''(1)=0.
```

These are exactly the endpoint conditions needed to integrate the source cosine coefficient by parts twice.

## Why the question's initial parity premise is false

The scalar factor

```plain text
u·(1+R)^(-β)
```

is even. But the **flux** includes the additional factor `Rₓ`, which is odd. Therefore the full flux is odd, not even.

The implication

```plain text
Q even ⇒ Qₓ odd
```

is mathematically correct, but its premise does not describe the chemotaxis flux used in this repository.

# 4. What `IntervalWeakH2Neumann` does—and does not—mean

Its definition is not a global parity predicate. It stores:

```lean
structure IntervalWeakH2Neumann (f : ℝ → ℝ) where
  secondDeriv : ℝ → ℝ
  second_intervalIntegrable : ...
  second_abs_integral_bound : ...
  weak_cosine_laplacian : ∀ k,
    ∫ cos(kπx) * secondDeriv x
      = -(kπ)^2 * ∫ cos(kπx) * f x
```

Thus it is a weak cosine integration-by-parts certificate. A doubly-even smooth representative is one sufficient way to build the certificate because it supplies the Neumann endpoint conditions, but the structure itself does not assert that the literal function `f : ℝ → ℝ` is globally even.

This matters because `coupledChemDivSourceLift` is a zero-extended function. It need not be globally even or globally smooth. The weak certificate can still be transferred from a global even representative because all occurrences of `f` in the certificate are inside integrals over `[0,1]`.

The repository supplies

```lean

```

for true pointwise agreement on the closed interval. For chemDiv, however, the safest existing theorem is `chemDivSource_weakH2_of_cosineRep`, because it transfers by `Ioo`/a.e. agreement. This avoids relying on endpoint equality between:

```plain text
deriv of the honest global cosine representative,
```

and

```plain text
deriv expressions formed using the zero-extension lift.
```

Endpoint values are measure-zero for the cosine integrals, but zero-extension derivatives at endpoints can be implementation artifacts. Do not replace the existing a.e. transfer with an `Icc` equality unless that endpoint equality has actually been proved.

# 5. Cosine coefficients are still the right coefficients

There are two separate facts that are sometimes conflated:

1. Cosine modes are the Neumann eigenbasis on `[0,1]`.
2. A globally odd function integrated over a symmetric domain has zero pairing with an even function.

The cosine coefficient here is integrated only over `[0,1]`, not over `[-1,1]`. Therefore even if a function has an odd extension about `0`, its cosine coefficients on `[0,1]` need not vanish.

For example,

```plain text
Q(x)=sin(πx)
```

is odd about both Neumann endpoints, but

```plain text
∫₀¹ Q(x) dx = 2/π ≠ 0.
```

So the argument “the function is odd, therefore its cosine coefficients vanish” is invalid on the half interval. More importantly, the actual source `S=Q'` is even anyway.

The zero mode of the **source** vanishes for a different reason:

```plain text
∫₀¹ S = ∫₀¹ Q' = Q(1)-Q(0)=0,
```

using the flux endpoint values.

# 6. Should the weak-H² certificate instead be built for the flux?

Not with the current `IntervalWeakH2Neumann` interface.

The flux has odd/Dirichlet parity:

```plain text
Q(0)=Q(1)=0,
```

but generally does not have the Neumann condition

```plain text
Q'(0)=Q'(1)=0.
```

Therefore applying

```lean
intervalWeakH2Neumann_of_doublyEven
```

or any other Neumann/even adapter directly to `Q` is wrong.

The committed divergence identity

```lean
ShenWork.Paper2.ChemFluxCosineDivergence.cosineCoeff_deriv_eq_kpi_sineCoeff
```

is a valid alternative route:

```plain text
cosCoeff(S,k) = kπ · sineCoeff(Q,k).
```

But to conclude

```plain text
cosCoeff(S,k)=O(k^-2),
```

one then needs

```plain text
sineCoeff(Q,k)=O(k^-3).
```

That requires the same substantive regularity as the direct source route:

```plain text
Q''' ∈ L¹,
Q(0)=Q(1)=0,
Q''(0)=Q''(1)=0.
```

The last boundary pair follows from the odd parity of `Q`. This would naturally be packaged in a separate odd/Dirichlet flux certificate, not in `IntervalWeakH2Neumann Q`.

Indeed, three integrations by parts give schematically

```plain text
∫ Q sin(kπx)
  = -1/(kπ)^3 ∫ Q''' cos(kπx),
```

with the `Q` and `Q''` boundary terms killed by odd parity. Multiplying by the divergence factor `kπ` then gives `O(k^-2)` for the source.

So the two sound routes are equivalent in required regularity:

```plain text
Route A, already implemented:
  Q ∈ C³ and odd
  → S=Q' ∈ C² and even-Neumann
  → IntervalWeakH2Neumann S
  → cosCoeff(S)=O(k^-2).

Route B, alternative:
  Q ∈ C³ and odd-Dirichlet
  → sineCoeff(Q)=O(k^-3)
  → cosCoeff(Q')=kπ·sineCoeff(Q)=O(k^-2).
```

Route A is shorter in the current repository because its exact constructor already exists.
