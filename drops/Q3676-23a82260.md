# Audit: χ₀ = 0 headline route after `2fbed9de`

## 1. Which route should Codex focus on?

Focus on **B: `LimitRegularityInputs` in `IntervalDomainMildLocalChi0.lean`**, not the older `PicardIterateConvergenceData` route in `IntervalPicardLimitLogisticSource.lean`.

This is a dependency reason, not a taste preference.

`PicardIterateConvergenceData` only feeds the restart package:

```lean
PicardIterateConvergenceData D
  → gradientMildHalfStepRestartData_of_iterateConvergence D I
  → GradientMildHalfStepRestartData D
```

But its decisive spectral realization is already a raw field:

```lean
hSpectralAgree : ∀ t, 0 < t → t < D.T →
  Set.EqOn (intervalDomainLift (D.u t))
    (fun x => ∑' n,
      restartDuhamelCoeff
        (gradientMildHalfStepInitialCoeff D t)
        (fun σ n => cosineCoeffs (logisticSourceFun p.a p.b p.α
          (profile t σ)) n)
        (t / 2) n * cosineMode n x)
    (Set.Icc 0 1)
```

So proving `PicardIterateConvergenceData` still leaves the main restart identity as an opaque, high-level assumption, and it does not produce the reduced classical frontier core.

By contrast, `LimitRegularityInputs` is the current headline-facing ledger. It directly feeds:

```lean
restartData_of_inputs : LimitRegularityInputs p u₀ D → GradientMildHalfStepRestartData D
frontierCore_of_inputs : LimitRegularityInputs p u₀ D → GradientMildClassicalFrontierCoreData p D
hMildLocal_chi0_zero_of_inputs : ... → IntervalDomainGradientMildHalfStepRestartFrontierCoreLocalData p
```

The `frontierCore_of_inputs` proof shows the dependency split exactly: `Hrestart` is built from `restartData_of_inputs`, `Hv` is built from `Hvsrc` via `hasResolverDirectSpectralData_of_clamped_perT0`, and the remaining frontier fields are supplied to `gradientMildClassicalRegularityFrontierData_of_spectral` as `I.Hu` and `I.Hvpos`, with `I.hpde_u` carried directly. Thus reducing any `LimitRegularityInputs` field immediately shrinks the current χ₀ = 0 local-existence hypothesis. Reducing `PicardIterateConvergenceData` does not, unless it is first threaded back into this ledger.

A secondary point: the current `paper2_theorem_1_1_chiZero_of_inputs` still takes a separate `hPLF : PicardLimitRestartFrontier p` for the quantitative side. Since `LimitRegularityInputs` contains enough data to build both `GradientMildHalfStepRestartData D` and `GradientMildClassicalFrontierCoreData p D`, a pure wrapper could derive `PicardLimitRestartFrontier` from the ledger. That wrapper is not the main target here because it would not reduce an analytic field, but it confirms that B is the dependency-dominating route.

## 2. Smallest plausible existing producer among `LimitRegularityInputs` fields

The best small target is **`hpde_u`**.

It already has a source-side producer split into two proven APIs:

```lean
ShenWork.Paper2.PdeUWiring.hasSpectralPdeAgreement_of_localized_data
ShenWork.IntervalDomainPdeUProducer.mildSolution_pde_u_of_spectral
```

The first builds `HasSpectralPdeAgreement p D.T D.u` from exactly the localized ledger ingredients: `hχ0`, `hα/ha/hb`, `hu₀_cont`, `hu₀_bound`, `hfix`, `hsrc0`, `bc/hbsum/hagree/hpost/hubt`, `hG1t/hG2t`, the logistic K1 fields `adott/hderivt/hadotcontt/hMdott`, and the `constExtend` form of `hLc`. The second turns that spectral agreement into the exact parabolic PDE field `hpde_u`.

This is not just a repackaging of `hpde_u`: it replaces the field by strictly smaller, already-present spectral/restart/K1 inputs. In fact, `IntervalDomainThm11ChiZeroCoreProvider.lean` already uses this pattern inline in its provider body.

There are two immediate siblings:

* `Hu` can be reduced by the same localized data using `TimeNhdSubtype.Hu_of_restart_localized_of_subtypeCont`.
* `hLc` itself can be derived from `D.hcont` by unfolding `intervalLogisticSource` and composing continuity with `rpow_const`; the proof pattern is already inline in `IntervalDomainThm11ChiZeroCoreProvider.lean`:

```lean
have hcu := D.hcont s hs (hsT.trans htT.le)
unfold ShenWork.IntervalDomainExistence.intervalLogisticSource
exact hcu.mul
  (continuous_const.sub
    (continuous_const.mul (hcu.rpow_const (fun _ => Or.inr p.hα.le))))
```

Still, `hpde_u` is the more valuable first reduction because it removes a frontier-core field currently passed directly in `frontierCore_of_inputs`.

## 4. What remains hard after this reduction?

After removing `hpde_u` (and likely `Hu`/`hLc` by the same localized-data pattern), the next hard frontier is the **source/restart representation package feeding `hsrc0`, `bc`, `hbsum`, `hagree`, the K2 bounds, and the K1 coefficient-time data**, plus the resolver-side fields `Hvsrc` and `Hvpos`.

Evidence from the code:

* `restartData_of_inputs` consumes `hsrc0`, `bc`, `hbsum`, `hagree`, `hpost`, `hubt`, `hG1t`, `hG2t`, and the K1 quadruple directly to build `GradientMildHalfStepRestartData`. There is no way to get restart data without these fields in the current localized packaging API.
* Route A does not eliminate this: `PicardIterateConvergenceData` still carries `hSpectralAgree` as a raw field. It is therefore not a smaller producer of the restart realization.
* `IntervalDomainThm11ChiZeroCoreProvider.lean` documents the structural obstruction around `hsrc0F`: trying to build the bounded source package from `bc/hbsum/hagree` is circular because those representations are themselves obtained from a `DuhamelSourceBddOn` package. The code comments isolate the real missing input as a direct per-window envelope / iterate-side bootstrap from `D`, not another wrapper.
* `Hvpos` is explicitly described in `IntervalRegularityFrontierWiring.lean` as coming from the elliptic strong maximum principle for `-Δv + μv = νu^γ > 0` with Neumann boundary conditions. That is a genuine maximum-principle theorem, not a ledger repackaging.
* `Hvsrc` was retyped to a per-`t₀` clamped witness because the global `DuhamelSourceTimeC1` form is false at `s = D.T`; closing it requires the power-source analogue of the logistic K1/decay machinery for `ν·u^γ` on compact interior windows.

So the next real Lean effort after the small reducer should be either:

1. a direct producer for the `hsrc0`/representation/K2/K1 package from the Picard-iterate residual data, avoiding the `hsrc0F ← hagreeF ← hsrc0F` loop; or
2. the resolver-source clamped `Hvsrc` power-source K1/decay theorem; or
3. the elliptic strong-positivity theorem for `Hvpos`.

Among these, `hpde_u` is the near-term 20–80 line reduction. The representation/source package and resolver positivity are the genuinely hard frontiers.
