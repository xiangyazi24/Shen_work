## Executive recommendation

Adding only

```lean

```

would be correct but mostly cosmetic: it composes Task96’s strict classical continuity package with Task95’s spatial-Young lowering. It would not expose anything beyond the strict-initial route layer.

If a compact exit wrapper is added, it should expose either scalar regularity or bounded-before while keeping the three real H¹ physical frontiers explicit in the signature:

```lean
H1PhysicalRHSIdentityBefore p u v T
H1PhysicalRHSSqrtBoundsBefore p u v T V₁ V₂ M L

```

My recommendation is to add **at most two wrappers** in a small new file, e.g.

```text

```

The two wrappers should be:

1. a strict-initial route normalizer from `spatialYoungData`;
2. a bounded-before exit wrapper using bundled `H1InitialEndpointData`.

A scalar-regularity wrapper is also natural, but if only two wrappers are allowed, the bounded-before wrapper better shows the actual “exit” and the strict-route wrapper keeps the inner normalization reusable.

## 2. Is the strict-route wrapper alone worth it?

Only modestly.

A strict-route-only wrapper is useful as a convenience normalizer because it collapses:

```lean
H1PhysicalRHSComponentSquareSpatialYoungDataBefore
  → H1PhysicalRHSComponentSquareZeroDataBefore
  → H1PhysicalRHSYoungScalarZeroMajorantsBefore
  → H1PhysicalRHSStrictInitialRouteBefore
```

and combines it with Task96’s classical strict-continuity producer. But it still stops before scalar regularity, RHS integrability, and bounded-before.

If the goal is a strategic exit marker, a scalar-regularity or bounded-before wrapper is better. It makes the exact remaining frontiers visible in one theorem statement, including ambient endpoint/bounded-before requirements. The bounded-before version is noisier, but it is the true exit.

## 3. False-green risk

Yes, there is a false-green risk if the wrapper name is too optimistic.

Avoid names like:

```lean
boundedBefore_of_physical_classical
H1PhysicalRHSStrictInitialRouteBefore_of_classical
H1ScalarRegularityBefore_of_physical_classical
```

Those names would obscure that the route still requires:

```lean
H1PhysicalRHSIdentityBefore p u v T
H1PhysicalRHSSqrtBoundsBefore p u v T V₁ V₂ M L
H1PhysicalRHSComponentSquareSpatialYoungDataBefore p u v T
```

Use names that include `spatialYoungData` and `initialEndpointData` when those hypotheses remain. The theorem signature should keep `hId`, `hBounds`, and `hSpatial` as ordinary visible arguments, not bury them inside an opaque “route assumptions” record.

The wrapper should also avoid the older closed-window package:

```lean
H1PhysicalRHSRouteBefore
```

because that route carries closed-window component continuity and can suggest zero-start `lapL2sq` continuity has been solved. The strict/initial route is the right one.

## 5. What to audit instead if wrappers are not added

If the team does not want more pure wiring, skip the strict-route wrapper. The next useful audit should target producers for the still-open structures:

```lean
H1PhysicalRHSIdentityBefore p u v T
H1PhysicalRHSSqrtBoundsBefore p u v T V₁ V₂ M L
H1PhysicalRHSComponentSquareSpatialYoungDataBefore p u v T
H1InitialEndpointData u₀ u T
```

Among these, the most source-facing unresolved package after Task95/96 is:

```lean
H1PhysicalRHSComponentSquareSpatialYoungDataBefore p u v T
```

Task95 proves the Young inequalities once this package is supplied, but the package still carries zero-window time integrability, a.e. spatial integrability, and assembled RHS measurability. It is not proved from the classical solution.

## Bottom line

Adding a strict-route-only wrapper is safe but cosmetic. Adding a bounded-before wrapper with `spatialYoungData` and `initialEndpointData` in the name is useful as a strategic exit marker, provided it keeps `hId`, `hBounds`, and `hSpatial` explicit. It must not be described as making the H¹ route unconditional.
