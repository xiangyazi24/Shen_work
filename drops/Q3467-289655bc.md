# Q3467 route audit: next non-circular H¹ route-C adapter

## Verdict

Candidate A is sound and useful as a small Task87 adapter, but it is not an analytic producer by itself.

It proves the missing scalar derivative initial-window proxy from an **independently supplied** assembled RHS initial-window integrability hypothesis:

```lean
hId : ∀ τ, τ ∈ Set.Ioo (0 : ℝ) T →
  H1EnergyIdentity p u τ (taxisX τ) (uvxx τ) (reactX τ)

hInit : H1IdentityRHSInitialWindowIntegrableBefore p u T
  taxisX uvxx reactX
```

using

```lean
g := H1IdentityRHSValue p u taxisX uvxx reactX
```

This is route-C compatible: no `H1IdentityRHSIntegrableBefore`, no bounded-before, no zero-start `lapL2sq` continuity, no `H1LapComponentZeroRightContinuous`, and no `H1ZeroStartPhysicalRHSDataBefore`.

The circularity risk is only in how the theorem is used. It is non-circular if `hInit` comes from a direct assembled-RHS initial-window estimate or RHS majorant. It is circular if `hInit` was itself obtained from `H1EnergyDerivativeInitialWindowIntegrableBefore` via Task82 and then fed back through Candidate A.

Candidate B is lower value for the present bottleneck. An FTC package built from already assuming derivative initial-window integrability may be useful for documentation or later integrated-energy statements, but it consumes the current bottleneck rather than producing it.

## Lean pitfalls

1. **Use `Ioc`, not `Ioo`, for the proxy field.**

   `H1EnergyDerivativeInitialWindowProxyBefore` asks for

   ```lean
   ∀ r, r ∈ Set.Ioc (0 : ℝ) b → deriv (H1energy u) r = g r
   ```

   From `hr : r ∈ Ioc 0 b`, use:

   ```lean
   have hr0 : 0 < r := hr.1
   have hrT : r < T := lt_of_le_of_lt hr.2 hbT
   ```

   Then feed `⟨hr0, hrT⟩` to `hId`.

2. **Do not use `H1IdentityRHSIntegrableBefore`.**

   Candidate A should consume only

   ```lean
   H1IdentityRHSInitialWindowIntegrableBefore p u T taxisX uvxx reactX
   ```

   Full `H1IdentityRHSIntegrableBefore` already contains all windows and risks hiding the circular route.

3. **Unfold only `H1EnergyIdentity`, then simp with `H1IdentityRHSValue`.**

   This is the same pattern already used in Task82:

   ```lean
   have hEnergy := hId r ⟨hr0, hrT⟩
   unfold H1EnergyIdentity at hEnergy
   simpa [H1IdentityRHSValue] using hEnergy.deriv
   ```

4. **Endpoint value at `0` is irrelevant.**

   The proxy equality is only on `Ioc 0 b`. No derivative identity at `0` is required.

5. **No positivity/strict-component hypotheses are needed.**

   The theorem only uses the identity and integrability of the assembled RHS. Strict component continuity belongs to the later Task82 adapter into full `H1IdentityRHSIntegrableBefore`.

## Better source-facing companion for Task87

If Task87 has room for one more small adapter, add an assembled-RHS majorant frontier parallel to Task83’s derivative majorant. This is useful if physical estimates naturally bound the RHS itself.

Suggested shape:

```lean
/-- Near-zero L¹ majorant for the assembled H¹ identity RHS.  This is weaker
than zero-start component continuity and avoids route-B lap/H² data. -/
def H1IdentityRHSInitialWindowMajorantBefore
    (p : CM2Params) (u : ℝ → intervalDomainPoint → ℝ) (T : ℝ)
    (taxisX uvxx reactX : ℝ → ℝ) : Prop :=
  ∃ G : ℝ → ℝ,
    (∀ {b : ℝ}, 0 ≤ b → b < T →
      IntervalIntegrable G volume (0 : ℝ) b) ∧
    (∀ {b : ℝ}, 0 ≤ b → b < T →
      AEStronglyMeasurable
        (H1IdentityRHSValue p u taxisX uvxx reactX)
        (volume.restrict (Set.Ioc (0 : ℝ) b))) ∧
    (∀ {b : ℝ}, 0 ≤ b → b < T →
      ∀ᵐ r ∂volume.restrict (Set.Ioc (0 : ℝ) b),
        ‖H1IdentityRHSValue p u taxisX uvxx reactX r‖ ≤ ‖G r‖)
```

Then prove:

```lean

    {p : CM2Params} {T : ℝ}
    {u : ℝ → intervalDomainPoint → ℝ}
    {taxisX uvxx reactX : ℝ → ℝ}
    (hMaj : H1IdentityRHSInitialWindowMajorantBefore p u T
      taxisX uvxx reactX) :
    H1IdentityRHSInitialWindowIntegrableBefore p u T taxisX uvxx reactX
```

The proof should copy the pattern from


in `IntervalChiNegH1InitialDerivativeProducer.lean`, replacing `deriv (H1energy u)` with `H1IdentityRHSValue p u taxisX uvxx reactX`.

Then Candidate A turns that `hInit` into the derivative proxy. This is a good source-facing route-C target because it asks only for a near-zero L¹ estimate of the assembled RHS, not componentwise zero-start continuity.

## False or circular routes to avoid

1. Do not prove Candidate A using `H1IdentityRHSIntegrableBefore`; use only `H1IdentityRHSInitialWindowIntegrableBefore`.

2. Do not produce `hInit` from `H1IdentityRHSInitialWindowIntegrableBefore_of_deriv_initialWindow` and then use Candidate A to produce the derivative proxy. That is a loop.

3. Do not use `H1_derivInt_of_identityRHSIntegrableBefore`; it consumes full RHS integrability and is downstream of the current bottleneck.

4. Do not use `IsPaper2BoundedBefore`, `boundedBefore_of_...`, or the final H¹ bound to prove `hInit`, a proxy, or a majorant.

5. Do not use `H1LapComponentZeroRightContinuous`, `H1ZeroStartPhysicalRHSDataBefore`, or `H1LiftDeriv2ZeroSlabRepresentativeBefore` for route C. Those are route-B endpoint-H²/lap assumptions.

6. Do not use strict positive-time component continuity alone to infer initial-window integrability. A positive-time continuous function can still be nonintegrable at zero.
