# Q3795 route audit: next shortest core step after Task319

## Verdict

Yes: the candidate On direct conjugate/B-form `hpde_u` provider is useful, and it is **not merely cosmetic** under the current rule.  It is a composition theorem, but it removes two real assumptions from the Task316 direct provider:

1. `DuhamelSourceTimeC1 (bFormSourceCoeffs p (conjugatePicardLimit p u₀ T))` is lowered to
   `DuhamelSourceTimeC1On ... 0 T`.
2. `Continuous (intervalDomainLift u₀)` is lowered to subtype continuity
   `Continuous u₀`.

This is exactly the bank-shaped interface produced by Task319 and consumed by Task317.  It should be added as a small core provider, not as a chi-headline theorem.  The next true producer after it is still source/Fourier data: especially `hsrcBDirect` from logistic + chem-div `DuhamelSourceTimeC1On`, and the `hchemIoo` representative/Fourier construction.  But landing this provider first is the shortest non-circular proof step because it exposes the lower-assumption route as a reusable theorem.

## Why this is an acceptable composition theorem

The theorem is short, but it is not a headline alias.  It exposes a strictly lower-assumption `hpde_u` route:

* Task316 still asks for global `DuhamelSourceTimeC1` and `Continuous (intervalDomainLift u₀)`.
* Task319 proves the needed `hB_global` from `DuhamelSourceTimeC1On ... 0 T` and `Continuous u₀`.
* Task317 consumes exactly `DuhamelSourceTimeC1On ... 0 T` plus the global B-form representation.

So the theorem removes real assumptions from the direct provider API.  It also makes the banked path more direct: downstream code can call one `hpde_u` provider rather than manually composing `conjugatePicardLimit_hB_global_of_open_sourceBridgeRepresentativeSubtypeLogisticDataOn` and `hpde_u_of_bForm_global_generalChiOn`.

## Existing inputs this theorem can use immediately

* `hfix`: from `conjugateMildSolutionData_of_data D` / `ConjugateMildExistenceData`, not from a PDE theorem.
* `hu₀_cont`: subtype-continuity of the initial datum, matching Task319.  This avoids the false zero-extension continuity requirement.
* `hu₀_bound`: already a bank-shaped input (`MInit` / `haInit`) in the existing B-form banked modules.
* `hsrcB_on`: the committed bank shape is `DuhamelSourceTimeC1On (bFormSourceCoeffs p (conjugatePicardLimit p u₀ T)) 0 T`; Task317 consumes this directly.
* `hB_global`: produced by Task319's
  `conjugatePicardLimit_hB_global_of_open_sourceBridgeRepresentativeSubtypeLogisticDataOn`.
* `hsource_split`: definitionally `rfl` for `bFormSourceCoeffs`.
* `hlogData`: can still use `logisticCosineFourierData_constExtend` when the logistic const-extension continuity/Fourier fields are honestly available.
* `hchemData`: must be the endpoint-safe `ChemDivCosineFourierDataIoo`, not the old closed-`Icc` const-extension chem-div package.

## Assumptions still not lowered by this theorem

This theorem does **not** prove the source-bridge and Fourier side conditions.  Those remain the next genuine producers:

1. `hsrcB_on` producer:
   a theorem combining logistic and chem-div `DuhamelSourceTimeC1On` packages via
   `bFormSource_duhamelSourceTimeC1On`.  If the logistic source package is not already public, that is likely the highest-value next producer.

   Suggested target shape:

   ```lean
   theorem conjugatePicardLimit_bFormSource_duhamelSourceTimeC1On
       {p : CM2Params} {u₀ : intervalDomainPoint → ℝ} {T : ℝ}
       (hlog_on : DuhamelSourceTimeC1On
         (coupledLogisticSourceCoeffs p (conjugatePicardLimit p u₀ T)) 0 T)
       (hchem_on : DuhamelSourceTimeC1On
         (coupledChemDivSourceCoeffs p (conjugatePicardLimit p u₀ T)) 0 T) :
       DuhamelSourceTimeC1On
         (bFormSourceCoeffs p (conjugatePicardLimit p u₀ T)) 0 T :=
     ShenWork.IntervalBFormSpectral.bFormSource_duhamelSourceTimeC1On hlog_on hchem_on
   ```

   This exact theorem is only worthwhile if it also packages real upstream producers for `hlog_on` and `hchem_on`; otherwise it is just a projection wrapper.

2. `hchemIoo` representative producer:
   use `ChemDivCosineFourierDataIoo` / `chemDivCosineFourierDataIoo_of_repr` with a continuous representative agreeing only on `(0,1)`.  Do **not** revive the false endpoint const-extension continuity of the chem-div source.

3. Source-bridge side conditions:
   `hB_int`, `hlog_int`, `hchem_cont`, `hlog_cont`, `hlog_bound`, `hchem_bound`, `hQderiv`, and `hdiv_rep` are still exposed.  Some of these should be derivable from the same source/Fourier packages, but this On provider does not discharge them.

4. `hdiv_rep` specifically:
   if `hchemData` is already available, the representative part can likely be projected safely from
   `ChemDivCosineFourierDataIoo`:

   ```lean
   theorem hdiv_rep_of_chemDivCosineFourierDataIoo
       {p : CM2Params} {u₀ : intervalDomainPoint → ℝ} {T : ℝ}
       (hchemData : ∀ t, 0 < t → t < T →
         ChemDivCosineFourierDataIoo p
           ((conjugatePicardLimit p u₀ T) t)
           (coupledChemicalConcentration p (conjugatePicardLimit p u₀ T) t)) :
       ∀ s, 0 < s → s < T →
         ∃ Gdiv : ℝ → ℝ,

           Set.EqOn
             (coupledChemDivSourceLift p (conjugatePicardLimit p u₀ T) s)

     intro s hs hsT
     refine ⟨(hchemData s hs hsT).representative, ?_, ?_⟩
     · exact (hchemData s hs hsT).continuous_representative.continuousOn
     · intro y hy
       exact ((hchemData s hs hsT).representative_eq_chemDiv hy).symm
   ```

   This is safe and endpoint-insensitive, but it is mostly a field projection; it should not be prioritized over `hsrcB_on` or the actual `hchemIoo` construction unless it unlocks a nearby proof.
