## 1. Next true non-circular frontier after `ResolverSourceWindowJointInputs`

Do **not** add one monolithic global producer package. Split the package.

The current `ResolverSourceWindowJointInputs` still mixes four logically different layers:

1. **u-side cosine representation** of `D.u σ`:
   `bc`, `hbsum`, `hagree`.
2. **joint continuity** of the lifted solution:
   `hliftCont`.
3. **spatial K2 compact bounds**:
   `hG1`, `hG2`.
4. **power-source K1 coefficient time regularity** for `ν * (D.u)^γ`:
   `adotPow`, `hderivPow`, `hadotPowCont`, `hMdotPow`.

These should be split, because the first two are essentially already provided by `HasTimeNeighborhoodSpectralAgreement`, while the last two are genuine analytic estimates. One global producer would hide which part is source-free and which part is the real analytic wall, and it risks circularly using downstream regularity/core outputs to prove upstream source data.

The next true frontier should therefore be a smaller package, roughly:


## 2. Field availability from existing infrastructure

### Already available from `GradientMildSolutionData`

`GradientMildSolutionData` already gives:

* positivity of `D.u t` on positive times (`D.hpos`), used in `resolverSourceWindowInputs_of_jointInputs` to fill `hpos`;
* pointwise/subtype upper bounds (`D.hbound`), used to fill `Msup` and `hub`;
* continuous spatial slices (`D.hcont`);
* mild-map equality (`D.hmild`) and measurability.

It does **not** give the global cosine representation, joint time-space continuity, spatial derivative bounds, or power-source K1 coefficient regularity by itself.

### Already available from `HasTimeNeighborhoodSpectralAgreement`

`Hu : HasTimeNeighborhoodSpectralAgreement D.T D.u` is already a field in the PPID joint-input frontier. It should provide two current `ResolverSourceWindowJointInputs` fields/layers:

1. `hliftCont` directly through:


   This theorem returns exactly:

2. `bc/hbsum/hagree` should be derivable, though not yet packaged, by freezing the local restart data from:

   For each interior `σ`, the structure gives `a₀`, a bound `M`, a source family `a`, a `DuhamelSourceTimeC1 a`, an `offset`, `0 < σ - offset`, and eventually-near-`σ` agreement:

This is non-circular: it uses only the u-side spectral restart representation, not classical/core regularity and not resolver source data.

### Requires genuinely new analytic lemmas

The fields that are not just packaging are:

1. `hG1` and `hG2`, the compact-in-time spatial first/second derivative bounds for `D.u`.

   These should come from a positive-time smoothing/K2 estimate for the Picard-limit solution, preferably from restart cosine representation plus summability, or from a direct mild smoothing estimate. They must not be taken from `GradientMildClassicalFrontierCoreData` or `IsPaper2ClassicalSolution`, since those are downstream consumers.

2. `adotPow`, `hderivPow`, `hadotPowCont`, `hMdotPow`, the power-source K1 package for

   There is a chi-zero-flavored pattern in `ResolverPowerK1.powerK1_quadruple_of_subtypeCont`, used in the older `IntervalDomainThm11ChiZeroCoreProvider` route, but the non-χ₀ PPID path needs a faithful general producer from u-side time regularity plus positive lower bound / spatial K2 bounds. This is real analysis: chain rule for `r ↦ u(r,x)^γ`, continuity of the derivative field, and dominated differentiation under the cosine integral.

## 3. Circularity traps to avoid

1. **Do not use `GradientMildClassicalFrontierCoreData`, `RegularityBootstrap`, or `IsPaper2ClassicalSolution` to produce `ResolverSourceWindowJointInputs`.**

   Those are downstream of the same resolver-source data. Using them would close a loop.

2. **Do not use `HasResolverDirectSpectralData` to prove the resolver-source witness.**

   `HasResolverDirectSpectralData` is the consumer of the clamped resolver-source witness via:

   ```lean
   hasResolverDirectSpectralData_of_clamped_perT0
   ```

3. **Do not prove spatial K2 from `ResolverSourceWindowJointInputs` itself.**

   The source-window package consumes K2 to build decay for `ν * u^γ`; it cannot be used to supply K2.

4. **Do not rely on global `DuhamelSourceTimeC1` for the canonical resolver source on the whole horizon.**

   The route intentionally uses clamped/window-local witnesses because global endpoint behavior is too strong or false for the intended data. Stay window-local.

5. **Do not use the χ₀=0 gradient-map simplification for the non-χ₀ PPID route.**

   The non-χ₀ route must carry the chemotaxis term through B-form or direct source identities. Any proof using `intervalGradientDuhamelMap_eq_of_chi0_zero` is only a chi-zero proof.

6. **Avoid one giant “producer” theorem.**

   A monolithic `∀ D, ResolverSourceWindowJointInputs p D` would blur representation, K2, and K1 obligations and make it hard to detect circular dependencies. Split the source-free fields from the genuinely analytic fields.
