# Hostile audit: strict-negative EWA floor dependence at `52cf8838`

## Verdict

A genuinely floor-robust, uniform-in-`M` EWA fixed-point estimate is **not plausible from the current equations/interfaces**. The current clean fixed-point route degenerates as the datum floor tends to `0`.

This is not just a missing wrapper. The current prescribed-time fixed-point theorem literally requires:

```lean
(hfloor₀ : ∀ y, δ₀ ≤ u₀ y)
(hKlt : |p.χ₀| * C₀ * CleanFPConst.L_Q p normBound δ₀ * Real.sqrt T
    + CleanFPConst.L_G p normBound δ₀ * T < 1)
(hsmall : |p.χ₀| * C₀ * CleanFPConst.M_Q p normBound δ₀ * Real.sqrt T
    + CleanFPConst.M_G p normBound δ₀ * T ≤ δ₀ / 2)
```

and `SourceCleanFPFloorAudit.lean` verifies that the constants used in those inequalities evaluate the negative-power estimates at `floor/2`. Concretely:

```lean
CleanFPConst.Mdv_eq_floorHalf
CleanFPConst.M_Q_eq_floorHalf
CleanFPConst.M_G_eq_floorHalf
CleanFPConst.L_Q_eq_floorHalf
CleanFPConst.L_G_eq_floorHalf
CleanFPConst.prescribed_smallness_rhs_eq_floorHalf
```

The definitions behind those audit lemmas are in `SourceCleanFPConstants.lean`; the load-bearing constants use `negNormConst ... (floor / 2) ...` and `negLipConst ... (floor / 2) ...`. In `FnegLipschitz.lean`, these constants contain powers of `(1 / δ)`:

```lean
def negNormConst (s δ Md : ℝ) : ℝ :=
  (1 / Real.Gamma s) * (8 * (1 + 1 / Real.pi) ^ 2) *
    ((1 / δ) ^ s * Real.Gamma s
      + 2 * Md * (1 / δ) ^ (s + 1) * Real.Gamma (s + 1)
      + Md ^ 2 * (1 / δ) ^ (s + 2) * Real.Gamma (s + 2))

def negLipConst (s δ Md : ℝ) : ℝ :=
  (1 / Real.Gamma s) * (8 * (1 + 1 / Real.pi) ^ 2) *
    ((1 / δ) ^ (s + 1) * Real.Gamma (s + 1)
      + 2 * Md * (1 / δ) ^ (s + 2) * Real.Gamma (s + 2)
      + Md ^ 2 * (1 / δ) ^ (s + 3) * Real.Gamma (s + 3))
```

So as `floor → 0`, the argument `δ = floor/2` drives these constants upward, while the self-map target simultaneously shrinks to `floor/2`. A fixed positive `T` cannot be obtained from this mechanism uniformly across floors tending to zero. `exists_uniform_EWA_lifespan` is only a bookkeeping theorem once fixed upper bounds on `L_Q`, `L_G`, `M_Q`, `M_G` and a fixed positive radius `ρ` are supplied; it is not a source of floor-robustness.

Even though `DatumWienerLifting` does not have the direct `C3NeumannDatum.hagree` field used in Task236, it does have:

```lean

W.hsumc  : Summable (fun k => |cosineCoeffs W.u₀ k|)
```

Existing cosine inversion APIs should turn this into the same floor obstruction for constant data. Therefore `fm ≤ W.floor ≤ W.u₀(1/2) = a`. Choosing constant PPID data `a < fm`, `0 < a ≤ 1`, contradicts `DatumWienerData`.

## Comments on the other options

### Option 2: restricted uniformly-floored subclass

This is mathematically honest but should be explicitly marked **non-headline**. It would prove a theorem for data satisfying a fixed lower bound `η > 0` plus a uniform Wiener norm bound. That does not imply `Theorem_1_1` as stated, because `Theorem_1_1` quantifies over all PPID and Task235/236 show no common positive `η` exists even in the bounded `M=1` class.

A restricted theorem is useful documentation, but it should not be sold as headline progress.

### Option 3: min-persistence/floor propagation

The repo already has min-persistence/boundary machinery, especially:

```lean
BoundaryMinPersistenceWindowBound p
boundaryMinPersistenceWindowBound_chiNonpos
classicalMinPersistence_of_boundary_window_regime
```

and the general-χ boundary producer in `IntervalDomainBoundaryChemDivLimit.lean` now supplies the boundary min-persistence residual from classical solutions.

But this does **not** replace the common datum floor in the current EWA local factory. Min-persistence propagates lower bounds for an already-built classical solution or feeds continuation/quantitative restart after local existence exists. The EWA fixed-point construction needs a uniform lower floor at the center before it can build the strict-negative local object. So this interface helps continuation, not the initial all-PPID EWA local construction.

### Option 4: constants blow up as floor → 0

The existing `SourceCleanFPFloorAudit.lean` already exposes the exact half-floor dependency. A formal asymptotic theorem such as

```lean
Tendsto (fun δ => negNormConst s δ Md) (𝓝[>] 0) atTop
```

would be mathematically honest, but it is probably more work than the `DatumWienerData` obstruction because it requires real-analysis lemmas for `Real.Gamma`, positive powers, and sums of divergent positive terms. It is useful later, but not the smallest next commit.
