# Verdict: NO, not as a ready-to-use Mathlib IBP theorem

I do **not** see a current Mathlib theorem giving the integration-by-parts identity in the exact AC/a.e.-derivative form you want.

The shipped interval-integral IBP lemmas are the `intervalIntegral.integral_mul_deriv...` family, but they require pointwise derivative hypotheses, not merely absolute continuity plus a.e. derivative.

The relevant Mathlib names are:

```lean
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDeriv_right
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivAt
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivWithinAt
intervalIntegral.integral_mul_deriv_eq_deriv_mul
```

Their shape is, schematically:

```lean
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDerivAt
  (hu : ContinuousOn u (Set.uIcc a b))
  (hv : ContinuousOn v (Set.uIcc a b))
  (huu' : ∀ x ∈ Set.Ioo (min a b) (max a b), HasDerivAt u (u' x) x)
  (hvv' : ∀ x ∈ Set.Ioo (min a b) (max a b), HasDerivAt v (v' x) x)
  (hu' : IntervalIntegrable u' volume a b)
  (hv' : IntervalIntegrable v' volume a b) :
  ∫ x in a..b, u x * v' x =
    u b * v b - u a * v a - ∫ x in a..b, u' x * v x
```

The right-derivative variant weakens two-sided differentiability to right-derivatives on the interior:

```lean
intervalIntegral.integral_mul_deriv_eq_deriv_mul_of_hasDeriv_right
  ...
  (huu' : ∀ x ∈ Set.Ioo (min a b) (max a b),
    HasDerivWithinAt u (u' x) (Set.Ioi x) x)
  (hvv' : ∀ x ∈ Set.Ioo (min a b) (max a b),
    HasDerivWithinAt v (v' x) (Set.Ioi x) x)
  ...
```

but this is still a pointwise derivative assumption at every interior point. It is not an a.e.-derivative/absolute-continuity theorem.

## What this means for your `positivePart` flux

For

```lean
Q y = positivePart (lift w y) * g y / (1 + R y)^β
```

`positivePart ∘ lift w` can fail to be differentiable at interior zeros where `deriv (lift w) y ≠ 0`. So the existing repo lemma

```lean
intervalConjugateKernelOperator_eq_semigroup_deriv
```

which asks for

```lean
∀ y ∈ Set.Ioo 0 1, HasDerivAt Q (deriv Q y) y
```

cannot be satisfied from `lift w ∈ C¹` alone. The `not_bad` condition

```lean
lift w y = 0 → deriv (lift w) y = 0
```

is exactly what removes the `positivePart` corner and recovers pointwise differentiability.

## About absolute continuity / Lipschitz

Mathematically, yes: an AC/Lipschitz integration-by-parts theorem would be the right way to avoid the `not_bad` hypothesis. If you had a theorem of the form

```lean
AbsolutelyContinuousOn Q (Set.Icc 0 1)
AbsolutelyContinuousOn K (Set.Icc 0 1)
HasDerivAt Q (Q' y) y for a.e. y
HasDerivAt K (K' y) y for a.e. y
```

then

```lean
∫ y in 0..1, K y * Q' y
  = K 1 * Q 1 - K 0 * Q 0 - ∫ y in 0..1, K' y * Q y
```

would be the mathematically correct route.

But the Mathlib file `Mathlib.MeasureTheory.Integral.IntervalIntegral.IntegrationByParts` does not expose that AC/a.e. variant. The documented IBP lemmas are derived from FTC-2 and use `HasDerivAt` / `HasDerivWithinAt` hypotheses pointwise on the relevant interval.

So there is no one-line Mathlib lemma I would expect to close your goal from continuity + boundedness + Lipschitz composition alone.

## Narrow answer

**NO**: for this proof, do not expect Mathlib to provide an AC/a.e.-derivative integration-by-parts theorem that bypasses everywhere differentiability of `Q`.

You have two realistic options:

1. Keep using the existing `intervalConjugateKernelOperator_eq_semigroup_deriv` path and add/carry the needed per-iterate regularity hypothesis, in practice the `not_bad` condition or a positivity/strong-maximum-principle input.

2. Develop a custom AC/weak-derivative IBP theorem for your file. That is new infrastructure; it is not a small application of an existing Mathlib lemma.

Therefore the theorem, as stated with only DT continuity/boundedness and no `C¹`/`not_bad`/AC-IBP infrastructure, is effectively under-hypothesized for the existing formal route.
