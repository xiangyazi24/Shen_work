## Bottom line

You **can** assemble the *per-time Holder part* of `ChemFluxCthetaSourceOn` from

```lean
hdecay_s : SourceCoeffQuadraticDecay p (u s)
```

plus uniform `u` bound/Holder and resolver nonnegativity: this is exactly what the newly added

```lean
chemFluxLifted_holder_Icc_of_sourceDecay_components
```

already does for a fixed slice `w`.

However, you **cannot honestly assemble the full `ChemFluxCthetaSourceOn p u T θ CQ HQ` from merely per-time `SourceCoeffQuadraticDecay p (u s)`**, because `ChemFluxCthetaSourceOn` needs a **single fixed** `CQ` and a **single fixed** `HQ` for all `0 < s ≤ T`, while the existing source-decay/resolver APIs return existential constants slice-by-slice. A per-time hypothesis

```lean
∀ s, 0 < s → s ≤ T → SourceCoeffQuadraticDecay p (u s)
```

allows the source-decay constant `hdecay.C` to vary arbitrarily with `s`; it gives no Lean handle for a uniform bound on the existential `G`, `Hg`, `Hv`, or the resulting flux `HQ`.

So the next non-circular theorem should be **conditional on uniform component constants**, or else you need a stronger uniform source-decay/component record. The smallest faithful next theorem is a pure assembler for `ChemFluxCthetaSourceOn` from uniform source-side fields; the next analytic task after that is to produce those uniform fields from the mild solution data.

## Why per-time source decay is not enough for `ChemFluxCthetaSourceOn`

Suppose the intended hypothesis is only:

```lean
hdecay : ∀ s, 0 < s → s ≤ T → SourceCoeffQuadraticDecay p (u s)
```

For each `s`, `chemFluxLifted_holder_Icc_of_sourceDecay_components (hdecay s hs0 hsT) ...` produces

```lean
∃ HQs, 0 ≤ HQs ∧ flux_holder_at_s HQs
```

But `ChemFluxCthetaSourceOn` needs one fixed `HQ`:

```lean
flux_holder : ∀ s, 0 < s → s ≤ T → ... ≤ HQ * |a - b| ^ θ
```

There is no theorem in the current API converting a family of existential per-slice `HQs` into a single bound. Nor is there a theorem showing that the `HQs` are bounded in terms of only `T`, `θ`, `U`, `Hu`, and fixed parameters. The source-decay record stores `hdecay.C`, and `hdecay.C` may vary with `s` without any upper bound.

Even adding a uniform bound on `hdecay.C` would not immediately feed the current per-slice existential wrappers, because `resolverGradReal_bounded_of_sourceDecay`, `resolverGradReal_holder_Icc_of_sourceDecay`, and `intervalNeumannResolverR_lift_holder_Icc_of_sourceDecay` currently expose only existential compactness/MVT constants, not quantitative constants monotone in `hdecay.C`. A uniform `Csrc` is the right analytic data, but an additional quantitative resolver wrapper would still be needed if the goal is to derive `G/Hg/Hv/HQ` rather than assume them directly.

## What a stronger uniform source-decay record would have to provide

A source-decay-only record should not just say `SourceCoeffQuadraticDecay` per time. At minimum it needs a uniform constant:

```lean
structure UniformSourceCoeffQuadraticDecayOn
    (p : CM2Params) (u : ℝ → intervalDomainPoint → ℝ) (T Csrc : ℝ) : Prop where
  Csrc_nonneg : 0 ≤ Csrc
  decay : ∀ s : ℝ, 0 < s → s ≤ T → ∀ k : ℕ, 1 ≤ k →
    |(intervalNeumannResolverSourceCoeff p (u s) k).re| ≤
      Csrc / ((k : ℝ) * Real.pi) ^ 2
```

But this record alone is not immediately enough with the current theorem surface, because current resolver APIs expose existential `G`, `Hg`, `Hv` through compactness rather than explicit functions of `Csrc`. To make this record useful, add quantitative variants, for example:

```lean
resolverGradReal_bound_Icc_of_uniform_sourceDecay
resolverGradReal_holder_Icc_of_uniform_sourceDecay
intervalNeumannResolverR_lift_holder_Icc_of_uniform_sourceDecay
```

with explicit constants depending on `p`, `θ`, and `Csrc`, or packaged as uniform component outputs.

Without those quantitative wrappers, the practical next step remains `ChemFluxUniformComponentsOn` plus `ChemFluxCthetaSourceOn_of_uniformComponents`.

## Final recommendation

Do **not** try to prove:

```lean

```

from only

```lean

```

plus `u` bound/Holder and resolver nonnegativity. That statement is underpowered: it has no uniform `HQ` source.
