## 1. Is Task268 non-circular and a real reduction?

Yes. The proposed Task268 is non-circular and genuinely reduces the headline-facing residual, provided it is implemented as a new additive surface that consumes the already-carried `Hu` and the already-carried envelope `henv`.

It is not merely cosmetic because it removes two independent carried obligations that are already derivable:

* `hliftCont` is derivable from the u-side spectral agreement already present in the PPID surface:

  ```lean
  RegularityFrontierAssembly.jointSolutionClosed_u_of_spectralAgreement Hu
  ```

* `hG1/hG2` are derivable from the compact-window eigenvalue envelope:

  ```lean
  ResolverSourceWindowInput.hG1_of_envelope
  ResolverSourceWindowInput.hG2_of_envelope
  ```

So the new primitive package should carry neither `hliftCont` nor `hG1/hG2`.

This is source-side safe: it uses only `HasTimeNeighborhoodSpectralAgreement` and coefficient envelope data. It does not use `GradientMildClassicalFrontierCoreData`, `IsPaper2ClassicalSolution`, `HasResolverDirectSpectralData`, or any downstream classical/core object.

## 4. After EnvelopeOnly and Task268, what remains?

Assuming concurrent EnvelopeOnly removes explicit `hbsum` from the envelope surface using singleton-window `henv`, and Task268 removes `hliftCont` while retaining envelope-based spatial K2 discharge, the real remaining producer frontier should be:

1. **u-side representation**:
   * `bc`
   * `hagree`
   * plus enough envelope data to imply summability (`henv`, possibly no explicit `hbsum`)

2. **compact-window coefficient envelope**:
   * `henv : ∀ a b, ... ∃ E, Summable E ∧ ...`

   This is now the substitute for explicit `hG1/hG2`.

3. **power-source K1 coefficient time regularity**:
   * `adotPow`
   * `hderivPow`
   * `hadotPowCont`
   * `hMdotPow`

4. **still outside resolver-source package but headline-facing in the PPID route**:
   * `PicardIterateConvergenceData D` (or a later producer of the logistic half-step source data)
   * `HasTimeNeighborhoodSpectralAgreement D.T D.u`
   * `hpde_u`

The next analytical split after Task268 and EnvelopeOnly should be to remove the u-side representation fields from the resolver-source package by deriving `bc/hagree` from the already-carried `Hu : HasTimeNeighborhoodSpectralAgreement D.T D.u`. That is non-circular but somewhat more involved: choose local restart data from `Hu.exists_data σ hσ hσT`, set `bc σ n := localRestartCoeff a₀ a (σ - offset) n`, prove `hagree` by extracting the eventual agreement at `σ`, and prove summability/envelope support via `restartDuhamelCoeff_eigenvalue_summable` or via the EnvelopeOnly `henv` if it is already available.

The genuine hard producer after these source-free field eliminations is:

```text
compact-window eigenvalue envelope henv
+ power-source K1 time-regularity for ν * (D.u)^γ
```

These must be proved from upstream mild/Picard data, not from `GradientMildClassicalFrontierCoreData`, `IsPaper2ClassicalSolution`, or resolver direct spectral data.

## Circularity warnings

Task268 itself is safe if it follows the conversion above. Avoid these traps:

* Do not derive `Hu` from a frontier core that already consumed resolver-source data.
* Do not derive `henv` from `hG1/hG2` if those `hG1/hG2` came from `ResolverSourceWindowJointInputs` or downstream regularity; Task264 direction is envelope → K2, not K2 → envelope.
* Do not use `HasResolverDirectSpectralData` to prove any resolver-source input; it is the downstream consumer.
* Do not import or mention EnvelopeOnly modules in Task268. Let EnvelopeOnly and EnvelopeNoJoint be orthogonal. A later tiny combined `EnvelopeOnlyNoJoint` wrapper can be added after both land.

## Verdict

Task268 is worth committing. It is an honest downstream-facing residual reduction, not just cosmetic, as long as it introduces PPID and chi-zero wrappers that consume the new surface. It should be implemented as new additive modules with `EnvelopeNoJoint` names and no root import changes.
