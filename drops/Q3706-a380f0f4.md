# Q3706 audit: next honest producer for `ResolverSourceWindowData`

## Executive answer

A. I do **not** see an already-landed non-`χ₀ = 0` producer

```lean
ResolverSourceWitnessFrontier.ResolverSourceWindowData p D
```

for an arbitrary PPID Picard-limit datum `D.u`.

The current Task246 files are honest frontier reductions, not producers.  In
`IntervalResolverSourceWitnessFrontier.lean`, `ResolverSourceWindowData` is the
new producer-facing residual, and the landed theorems are only

```lean
ResolverSourceWitnessFrontier.resolverSourceWitness_of_windowData
ResolverSourceWitnessFrontier.resolverDirectSpectralData_of_windowData
```

namely

```lean

    {D : GradientMildSolutionData p u₀}
    (H : ResolverSourceWindowData p D) :
    ResolverSourceWitness p D

    {p : CM2Params} {u₀ : intervalDomainPoint → ℝ}

    (H : ResolverSourceWindowData p D) :
    ShenWork.IntervalResolverDirectTimeRegularity.HasResolverDirectSpectralData
      D.T (ShenWork.IntervalMildToClassical.mildChemicalConcentration p D.u) p
```

These consume `ResolverSourceWindowData`; they do not build it.

Likewise, `IntervalDomainPPIDResolverSourceFrontier.lean` replaces the raw source
witness by the window field in the PPID wrappers, but its theorem hypotheses still
quantify over a producer:

```lean
hWindow : ∀ (u₀ : intervalDomainPoint → ℝ),
  PositiveInitialDatum intervalDomain u₀ →
  ∀ (D : GradientMildSolutionData p u₀),
    D.u = picardLimit p u₀ D.T →
      PerDatumWindowSourceSpectralFrontier p D
```

or the iterate version.  This is a good reduction surface, not a solved producer.

The closest existing proof is the `χ₀ = 0`-localized internal construction of
`Hvsrc` inside `Thm11ChiZeroCoreProvider.reducedLimitRegularityInputs_of_picard`.
That code builds a clamped per-`t₀` resolver-source witness using
`ResolverPowerDecay.powerSource_window_uniform_decay`,
`ResolverPowerK1.lift_window_uniformPositive_of_subtypeCont`, and
`ResolverPowerK1.powerK1_quadruple_of_subtypeCont`.  But that path is explicitly
`χ₀ = 0` because both `ResolverPowerK1` producer theorems take

```lean
(hχ0 : p.χ₀ = 0)
```

and route through the χ0 local-restart/gradient-Duhamel machinery.

So the answer to A is **no** for arbitrary PPID / `χ₀ ≤ 0` Picard limits.

## What the next interface should be

The next smallest honest reduction is useful **only if it is not just a synonym
for `ResolverSourceWindowData`**.

A structure that merely re-bundles the fields

* window endpoints `c' c d d'`,
* `bc`,
* `C`,
* `adot`, `Mdot`,
* the final decay and K1 coefficient properties,

would be a renaming of `ResolverSourceWindowData` and would not reduce anything.

The useful interface is a more primitive, time-global/per-compact input package
from which the existing power-decay theorem can actually derive the `C` in
`ResolverSourceWindowData`.  It should carry:

1. one global cosine representation family `bc : ℝ → ℕ → ℝ` for `D.u` on `(0,D.T)`;
2. summability and `[0,1]` agreement for that `bc`;
3. strict positivity;
4. a per-compact positive lower bound `m` for the lift;
5. a per-compact upper bound and K2 derivative/Hessian bounds;
6. the **power-source** coefficient time-`C¹` data for
   `fun x => p.ν * intervalDomainLift (D.u σ) x ^ p.γ`.

That is not a cosmetic rename: it replaces the per-`t₀` existential `C/adot/Mdot`
with global producer-side fields and uses
`ResolverPowerDecay.powerSource_window_uniform_decay` to manufacture `C` on each
clamp window.  The only analytic residual left in the interface is the true one:
proving the representation/K2/lower-bound/power-K1 fields for the actual PPID
Picard limit.

Suggested structure shape:

```lean
/-- Primitive producer-side inputs for the resolver-source window field.
This is intentionally stronger/different than `ResolverSourceWindowData`:
it carries global/per-compact representation, K2, lower-bound, and power-K1 data;
the per-window decay constant `C` is produced by
`ResolverPowerDecay.powerSource_window_uniform_decay`. -/
structure ResolverSourceWindowInputs
    (p : CM2Params) {u₀ : intervalDomainPoint → ℝ}
    (D : GradientMildSolutionData p u₀) : Prop where
  bc : ℝ → ℕ → ℝ

  hbsum : ∀ σ, 0 < σ → σ < D.T →
    Summable (fun n => unitIntervalCosineEigenvalue n * |bc σ n|)

  hagree : ∀ σ, 0 < σ → σ < D.T →
    Set.EqOn (intervalDomainLift (D.u σ))
      (fun x => ∑' n, bc σ n * cosineMode n x)
      (Set.Icc (0 : ℝ) 1)

  hpos : ∀ σ, 0 < σ → σ < D.T →
    ∀ x ∈ Set.Icc (0 : ℝ) 1, 0 < intervalDomainLift (D.u σ) x

  Msup : ℝ
  hub : ∀ σ, 0 < σ → σ < D.T →
    ∀ x ∈ Set.Icc (0 : ℝ) 1, intervalDomainLift (D.u σ) x ≤ Msup

  /-- Window-uniform positive lower bound.  This is the non-χ0-specific replacement
  for `lift_window_uniformPositive_of_subtypeCont`. -/
  hlower : ∀ a b, 0 < a → b < D.T → a ≤ b →
    ∃ m : ℝ, 0 < m ∧
      ∀ σ ∈ Set.Icc a b, ∀ x ∈ Set.Icc (0 : ℝ) 1,
        m ≤ intervalDomainLift (D.u σ) x

  hG1 : ∀ a b, 0 < a → b < D.T →
    ∃ G1, ∀ σ ∈ Set.Icc a b,
      ∀ x ∈ Set.Icc (0 : ℝ) 1,
        |deriv (intervalDomainLift (D.u σ)) x| ≤ G1

  hG2 : ∀ a b, 0 < a → b < D.T →
    ∃ G2, ∀ σ ∈ Set.Icc a b,
      ∀ x ∈ Set.Icc (0 : ℝ) 1,
        |deriv (deriv (intervalDomainLift (D.u σ))) x| ≤ G2

  adotPow : ℝ → ℕ → ℝ

  hderivPow : ∀ σ, 0 < σ → σ < D.T → ∀ n,
    HasDerivAt
      (fun r => cosineCoeffs
        (fun x => p.ν * intervalDomainLift (D.u r) x ^ p.γ) n)
      (adotPow σ n) σ

  hadotPowCont : ∀ n,
    ContinuousOn (fun σ => adotPow σ n) (Set.Ioo 0 D.T)

  hMdotPow : ∀ a b, 0 < a → b < D.T →
    ∃ Mdot, ∀ σ ∈ Set.Icc a b, ∀ n,
      |adotPow σ n| ≤ Mdot
```

Main theorem target:

```lean
/-- Primitive producer-side resolver-source inputs produce the Task246 window data. -/
theorem resolverSourceWindowData_of_inputs
    {p : CM2Params} {u₀ : intervalDomainPoint → ℝ}
    {D : GradientMildSolutionData p u₀}
    (H : ResolverSourceWindowInputs p D) :
    ShenWork.Paper2.ResolverSourceWitnessFrontier.ResolverSourceWindowData p D
```

Proof skeleton:

* Fix `t₀`, `0 < t₀`, `t₀ < D.T`.
* Use the same clamp window as the existing χ0-internal `Hvsrc` construction:

```lean
c' := t₀ / 4
c  := t₀ / 2
d  := (t₀ + D.T) / 2
d' := (t₀ + 3 * D.T) / 4
```

* Prove

```lean
c' < c ∧ c < t₀ ∧ t₀ < d ∧ d < d'
0 < c'
d' < D.T
c' ≤ d'
Set.Icc c' d' ⊆ Set.Ioo 0 D.T
```

* Restrict `H.hbsum`, `H.hagree`, `H.hpos`, `H.hub` to the clamp window.
* Obtain `m` from `H.hlower c' d'`.
* Obtain `G1`, `G2` from `H.hG1 c' d'` and `H.hG2 c' d'`.
* Apply

```lean
ShenWork.Paper2.ResolverPowerDecay.powerSource_window_uniform_decay
  (ν := p.ν) (γ := p.γ) (M := H.Msup) (m := m)
  p.hν.le p.hγ hm
  (w := D.u) (c' := c') (d' := d')
  hcd' H.bc hbsumW hagreeW hlbW hubW
  (G1 := G1) (G2 := G2) hG1W hG2W
```

  to produce the `C`, nonnegativity, quadratic decay, and zeroth coefficient bound.
* Obtain `Mdot` from `H.hMdotPow c' d'`.
* Restrict `H.hderivPow`, `H.hadotPowCont`, and `H.hMdotPow` to `[c',d']`.
* Return the existential fields required by `ResolverSourceWindowData`.

This should be a real residual reduction.  It is not a final analytic producer for
PPID, because `hlower` and `hderivPow`/`hadotPowCont`/`hMdotPow` remain genuine
producer-side inputs.  But it removes the clamped per-`t₀` boilerplate and ensures
all downstream Task246 wrappers consume a standardized primitive ledger rather than
the raw existential window shape.

## Which existing producers are general vs χ0-localized

### General in `p` / not intrinsically χ0

* `ResolverSourceWitnessFrontier.resolverSourceWitness_of_windowData`:
  consumes `ResolverSourceWindowData p D` and produces the raw per-`t₀` witness.
  No `hχ0`.

* `ResolverSourceWitnessFrontier.resolverDirectSpectralData_of_windowData`:
  consumes `ResolverSourceWindowData p D` and produces resolver direct spectral data.
  No `hχ0`.

* `PPIDThresholdReachability.sourceSpectralFrontier_of_windowSourceSpectralFrontier`,
  `picardLimitRestartFrontier_of_windowSourceSpectralFrontier`, and the
  `theorem_1_1_intervalDomain_of_ppid_windowSourceSpectralFrontier_*` wrappers:
  general wrappers, but they carry the window source frontier as a hypothesis.

* `ResolverPowerDecay.powerSource_window_uniform_decay`:
  general analytic decay reducer for `ν·u^γ`; it only needs representation,
  positivity/lower/upper bounds, and K2 bounds on a window.  It has no `χ₀ = 0`
  assumption.

* `ResolverSourceClampedWitness.clampedResolverSource_duhamelSourceTimeC1`:
  general clamped time-`C¹` packaging once all window decay and K1 coefficient inputs
  are supplied.

* `IntervalTimeNhdChiNeg.HasTimeNeighborhoodSpectralAgreement_of_conjugateMild`:
  χ-general/B-form for the **u-side time-neighborhood spectral agreement** of the
  conjugate Picard limit.  It is not a resolver-source-window producer and should
  not be confused with one.

### χ0-localized only

* `ResolverPowerK1.powerK1_quadruple_of_subtypeCont`:
  explicitly takes `hχ0 : p.χ₀ = 0`.  It builds the power-source K1 quadruple by
  routing through `LocalRestartWeak` and the χ0 gradient-Duhamel ledger.

* `ResolverPowerK1.lift_window_uniformPositive_of_subtypeCont`:
  explicitly takes `hχ0 : p.χ₀ = 0`.  It gets joint continuity/minimum positivity
  from the same χ0 local restart machinery.

* `Thm11ChiZeroCoreProvider.reducedLimitRegularityInputs_of_picard` and
  `tightLimitRegularityInputs_of_picard`:
  χ0-only providers.  They contain the useful internal `Hvsrc` construction, but
  they are not non-χ0 PPID source-window producers.

* `MildLocalChi0.LimitRegularityInputs` and the related `hpde_u_of_inputs_localized_data`,
  `Hu_of_inputs_localized_data`, `restartData_of_inputs`, `frontierCore_of_inputs`:
  χ0-only ledger/wiring.

## Circularity warnings

Do **not** use any of the following as inputs to produce `ResolverSourceWindowData`
for the PPID route:

* `GradientMildClassicalFrontierCoreData p D`.  This is downstream of resolver
  direct spectral data; using it to prove the resolver source window would be
  circular.

* `IsPaper2ClassicalSolution`.  This packages classical PDE conclusions and traces
  that the restart/core frontier is intended to produce.

* `hpde_u` or the PDE equality field as an input to `ResolverSourceWindowData`.
  In Task246's PPID surface, `hpde_u` is a sibling field of the source-window
  frontier, not a primitive producer for it.

* `Hu : HasTimeNeighborhoodSpectralAgreement D.T D.u` as if it implied the resolver
  source window.  `Hu` is u-side time-neighborhood spectral agreement; the resolver
  source window is about the time-`C¹` cosine coefficients of
  `ν * (D.u σ)^γ` used to build `mildChemicalConcentration`.  Existing χ-neg
  B-form Hu work does not discharge this source package.

The honest PPID/χ≤0 gap is therefore not the clamped packaging.  It is the missing
non-χ0 producer for the primitive source-window inputs: global/conjugate representation
for `D.u`, compact K2 bounds, compact positive lower bounds, and power-source K1
coefficient time regularity without routing through `hχ0`.
