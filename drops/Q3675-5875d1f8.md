# Q3675 audit: next real frontier toward `PicardLimitRestartFrontier`

## Verdict

The smallest next Lean theorem that makes **real** progress is not another C1/eta or Wiener wrapper.  It is a small source-coefficient congruence lemma that removes the algebraic mismatch between:

* the true limit restart identity, which uses the genuine source
  ```lean
  logisticLifted p (D.u (t / 2 + σ))
  ```
* and `PicardIterateConvergenceData.hSpectralAgree`, which is phrased with an abstract globally-C² profile:
  ```lean
  logisticSourceFun p.a p.b p.α (profile t σ)
  ```

Once this congruence is banked, the next missing premise for `hSpectralAgree` is exposed cleanly: prove that the chosen profile agrees with the true lifted mild slice on `[0,1]` at the shifted time `t/2 + σ`.

This is better than trying to construct all of `PicardIterateConvergenceData` at once.  That structure currently bundles several hard analytic convergence fields; the proposed lemma is a 20–50 line theorem, has low compile risk, and directly prepares the hSpectralAgree discharge.

## Existing APIs determining the frontier

### Existing theorem that already proves the true restart identity

`IntervalPicardLimitRestart.lean` already proves the genuine limit restart identity:

```lean
theorem picardLimitRestart_cosineIdentity
    (p : CM2Params) (hχ0 : p.χ₀ = 0)
    (u₀ : intervalDomainPoint → ℝ) (u : ℝ → intervalDomainPoint → ℝ)
    (hfix : ∀ t, 0 < t → ∀ x : ℝ, (hx : x ∈ Set.Icc (0:ℝ) 1) →
      intervalDomainLift (u t) x = intervalGradientDuhamelMap p u₀ u t ⟨x, hx⟩)
    (hu₀_cont : Continuous (intervalDomainLift u₀))
    {M₀ : ℝ} (hu₀_bound : ∀ k, |cosineCoeffs (intervalDomainLift u₀) k| ≤ M₀)
    (hsrc0 : DuhamelSourceTimeC1
      (fun s k => cosineCoeffs (logisticLifted p (u s)) k))
    {t : ℝ} (ht : 0 < t)
    (hL_cont : ∀ s, 0 < s → s ≤ t → Continuous (logisticLifted p (u s))) :
    Set.EqOn (intervalDomainLift (u t))
      (fun x => ∑' k : ℕ,
        restartDuhamelCoeff
          (cosineCoeffs (intervalDomainLift (u (t/2))))
          (fun σ k => cosineCoeffs (logisticLifted p (u (t/2 + σ))) k)
          (t/2) k * cosineMode k x)
      (Set.Icc (0:ℝ) 1)
```

This proves the right spectral agreement for the true source.  The remaining mismatch is replacing the true source coefficients by the abstract-profile coefficients in `PicardIterateConvergenceData.hSpectralAgree`.

## Is `hSpectralAgree` just field extraction?

No, not as a theorem-producing task.

Inside `PicardLimitHasLogisticSourceRegularity_of_iterateConvergence`, it is only field forwarding:

```lean
hSpectralAgree := I.hSpectralAgree
```

But producing the `I.hSpectralAgree` field is not a mere extraction/rewriting theorem.  It depends on:

1. the true restart identity `picardLimitRestart_cosineIdentity`;
2. the fixed-point/mild equation input `hfix` used by `limit_lift_eq_cosineSeries`;
3. source time-C¹ data for true `logisticLifted` coefficients;
4. per-slice logistic continuity;
5. a proof that the abstract `profile t σ` used by `PicardIterateConvergenceData` agrees with the true lifted slice on `[0,1]`, at least on the Duhamel integration interval `0 ≤ σ ≤ t/2`.

The genuinely hard analytic part is not the final `hSpectralAgree` rewriting once these ingredients are present.  The hard part is constructing the profile and the `profile = true lift on [0,1]` agreement, plus the source-coefficient convergence data in `SingleRestartConvergenceData`.

## Recommended theorem: coefficient congruence for profile logistic source

Target file:

```text
ShenWork/Paper2/IntervalPicardLimitLogisticSource.lean
```

Reason: this is exactly where `PicardLimitHasLogisticSourceRegularity`, `SingleRestartConvergenceData`, `PicardIterateConvergenceData`, and `hSpectralAgree` live.  The theorem only needs APIs already imported there (`logisticSourceFun`, `logisticLifted`, `cosineCoeffs`, `cosineCoeffs_congr_on_Icc` via earlier imports).  If Codex wants zero risk to existing file compile time, place it in a new file `IntervalPicardLimitLogisticSourceAgree.lean` importing `IntervalPicardLimitLogisticSource`; but semantically it belongs in `IntervalPicardLimitLogisticSource.lean`.

Exact theorem statement:

```lean
import ShenWork.Paper2.IntervalPicardLimitLogisticSource

open MeasureTheory Filter Topology
open ShenWork.IntervalDomain (intervalDomainLift intervalDomainPoint)
open ShenWork.IntervalNeumannFullKernel (cosineCoeffs)
open ShenWork.IntervalGradientDuhamelMap (logisticLifted)
open ShenWork.IntervalMildPicardRegularity (logisticSourceFun)

namespace ShenWork.IntervalPicardLimitLogisticSource

noncomputable section

/-- If a global profile agrees with the true lifted interval slice on `[0,1]`,
then its logistic-source cosine coefficients agree with the coefficients of
`logisticLifted`.  This is the algebraic bridge needed to turn the true restart
identity into the `profile`-based `hSpectralAgree` field. -/
theorem cosineCoeffs_logisticSourceFun_eq_logisticLifted_of_eqOn_Icc
    (p : CM2Params) {g : ℝ → ℝ} {w : intervalDomainPoint → ℝ}
    (hgw : Set.EqOn g (intervalDomainLift w) (Set.Icc (0 : ℝ) 1))
    (k : ℕ) :
    cosineCoeffs (logisticSourceFun p.a p.b p.α g) k =
      cosineCoeffs (logisticLifted p w) k := by
  refine ShenWork.Paper2.cosineCoeffs_congr_on_Icc ?_ k
  intro x hx
  have hgx : g x = intervalDomainLift w x := hgw hx
  -- On `[0,1]`, `logisticLifted` unfolds to the interval logistic source,
  -- and `intervalDomainLift w x = w ⟨x,hx⟩`.
  simp [logisticSourceFun, logisticLifted,
    ShenWork.IntervalDomainExistence.intervalLogisticSource,
    intervalDomainLift, hx, hgx]

end

end ShenWork.IntervalPicardLimitLogisticSource
```

Likely compile risks:

* Namespace/simp only: `cosineCoeffs_congr_on_Icc` is in `ShenWork.Paper2`, not the local namespace.
* The final `simp` may need a `rw [hgx]` before `simp`, or a small `change` if Lean does not unfold `intervalLogisticSource` aggressively enough.
* No analytic risk: no integrability, summability, or dominated-convergence work is involved.

The consumer `duhamelSourceTimeC1_of_singleRestartConvergence` simply applies

to these fields.  Therefore `restartData` is where the nonlinear source-coefficient convergence, derivative convergence, envelope passing, and continuity are concentrated.  It is not a 20–50 line extraction theorem.

## Dependency direction

The direction should be:

1. prove profile/logistic-source coefficient congruence;
2. prove `hSpectralAgree_of_profile_eqOn_Icc` from `picardLimitRestart_cosineIdentity`;
3. use that to define a reduced convergence-data structure that no longer carries `hSpectralAgree` as a field, or a constructor:

```lean
PicardIterateConvergenceData_of_profileEqOn_and_sourceConvergence
```

4. only then assemble `gradientMildHalfStepRestartData_of_iterateConvergence` and move toward `PicardLimitRestartFrontier`.

Do not reverse the dependency by trying to get `hSpectralAgree` from `HasRestartCosineRepresentations`; that is circular, since `HasRestartCosineRepresentations` is downstream of `GradientMildHalfStepRestartData`, and the restart data needs `hSpectralAgree`.
