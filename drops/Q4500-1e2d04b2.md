# 1. Roadmap P3.1--P3.7 versus the five final fields

This distinction matters for execution planning:

- **P3.4 is analytically load-bearing upstream**, because the stable half of Theorem 2.2 needs the local exponential-decay engine. But after it has been converted into `Paper3Theorem22BranchData`, it disappears as a separate assembly input.
- **P3.7 is not required by the type of the mainline umbrella.** Threshold-ordering results may be used upstream to prove explicit parameter implications or sharpen the constants inside Theorems 2.2--2.5, but the final `paper3_mainlineTargets_of_data` theorem does not consume a P3.7 argument.

If the desired project capstone is “mainline plus every explicit threshold-ordering corollary,” that should be a new outer conjunction built above `Paper3MainlineTargets`; it is not the existing definition.

# 3. Is there inter-floor dependency in the final wiring?

**No.** The exact body of `paper3_mainlineTargets_of_data` applies five assemblers independently:

```plain text
hData.propositions → P3 Proposition targets
hData.persistence  → Theorem 2.1 targets
hData.theorem22    → Theorem 2.2
hData.compactness  → Lemmas 3.1--3.5 and 7.1
hData.stability    → Theorems 2.3--2.5
```

Notably, the theorem does **not** do anything like:

```lean
paper3_stability23To25Targets_of_branchData
  hData.persistence hData.compactness hData.stability
```

It passes only `hData.stability` to the stability assembler. Likewise, persistence is not passed to Theorem 2.2, and compactness is not passed to the top-level global-stability assembly.

This means:

- any mathematical use of persistence inside a proof of global stability must be internalized upstream when constructing `Paper3Stability23To25BranchData`;
- any mathematical use of compactness, Lyapunov decay, zero-dissipation rigidity, or local sectorial decay must likewise already have been consumed by the corresponding branch-data producer;
- the final umbrella does not enforce or reconstruct those dependencies.

The global-stability statements themselves are phrased against positive global bounded solutions and the appropriate equilibrium/mass/parameter conditions. They do not take `Theorem_2_1` as an explicit hypothesis. Hence P3.2 and P3.6 are parallel conjuncts at the statement layer even if a chosen analytic proof of P3.6 reuses persistence estimates.

There **is** parameter sharing:

```plain text
same D, p, S, K, N, C1, C3
```

must be used in every field. That is the only coherence enforced by `Paper3MainlineData`. A producer built for a different `Paper3Constants`, spectrum, norm package, or compactness package will not elaborate without an explicit identification theorem.

# 7. What can still go wrong at final integration?

The final proof term is trivial, but five interface mismatches can still block elaboration:

1. **Different constants packages.** All P3.2--P3.6 fields must use definitionally the same `C3 : Paper3Constants D p`; inequalities proved for an ad hoc constants record need an equality/transport lemma.
2. **Different stability norms.** Theorem 2.2, compactness Lemma 3.3, and Theorems 2.3--2.5 must agree on the same `N : StabilityNorms D`.
3. **Different spectral data.** Theorem 2.2 must use the intended `S`; for `[0,1]`, this should normally be `unitIntervalNeumannSpectrum`, not an unrelated abstract witness.
4. **Different compactness package.** The P3.5 raw data must target exactly the `K` in the final theorem.
5. **Mistaking P3.7 for a missing record field.** No P3.7 argument belongs in the existing constructor. Export those results separately or define a larger capstone.

There is no hidden need to install `Fact` instances. The explicit theorem above is sufficient. The repository also provides `paper3_mainlineTargets_of_dataFact` and corresponding `...Fact` wrappers when instance-driven assembly is convenient.
