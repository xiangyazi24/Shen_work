# Q3753 audit: direct PLF / strict-negative headline route

I checked GitHub `main`; it is identical to `ecf4a4605d018cfc02026c3acd04223f621faab2`.

The short answer for the implementation direction is: do **not** add another wrapper or alias. The direct headline is not blocked by theorem-surface plumbing now; it is blocked by producer content. The existing named residual to discharge is the per-datum core behind `ConeQuantBridge.PicardLimitRestartFrontier p`.

## What `PicardLimitRestartFrontier` really requires

In `ShenWork/Paper2/IntervalDomainConeQuantBridge.lean`, the target is:

```lean
def ConeQuantBridge.PicardLimitRestartFrontier (p : CM2Params) : Prop :=
  ∀ (u₀ : intervalDomainPoint → ℝ),
    PositiveInitialDatum intervalDomain u₀ →
  ∀ (D : GradientMildSolutionData p u₀),
    D.u = picardLimit p u₀ D.T →
    ∃ _R : GradientMildHalfStepRestartData D,
      GradientMildClassicalFrontierCoreData p D
```

In `ShenWork/Paper2/IntervalMildToLocalExistence.lean`, the core record is exactly:

```lean
structure GradientMildClassicalFrontierCoreData
    (p : CM2Params) {u0 : intervalDomainPoint → ℝ}
    (D : GradientMildSolutionData p u0) : Prop where
  hpde_u :
    ∀ t x, 0 < t → t < D.T → x ∈ intervalDomain.inside →
      intervalDomain.timeDeriv D.u t x =
        intervalDomain.laplacian (D.u t) x
          - p.χ₀ * intervalDomain.chemotaxisDiv p (D.u t)
              (mildChemicalConcentration p D.u t) x
          + D.u t x * (p.a - p.b * (D.u t x) ^ p.α)
  hregularityFrontier :
    GradientMildClassicalRegularityFrontierData p D
```

So a direct proof of PLF must produce both a half-step restart package and the reduced classical frontier core for the canonical Picard limit.

## Why the direct headline/PLF is not currently provable from committed code alone

The existing code does not contain unconditional producers for the four fields of `PerDatumRestartCoreSpectralFrontier` for an arbitrary canonical Picard-limit `D`.

### Field 1: `GradientMildHalfStepLogisticSourceData D`

`ShenWork/Paper2/IntervalPicardLimitLogisticSource.lean` proves:

```lean
def gradientMildHalfStepLogisticSourceData_of_iterateConvergence
    {p : CM2Params} {u₀ : intervalDomainPoint → ℝ}
    (D : GradientMildSolutionData p u₀)
    (I : PicardIterateConvergenceData D) :
    GradientMildHalfStepLogisticSourceData D
```

but there is no committed constructor:

```lean
PicardIterateConvergenceData D
```

from just:

```lean
D.u = picardLimit p u₀ D.T
```

`PicardIterateConvergenceData` itself is a large analytic residual: it contains globally C² profiles, positivity, Neumann endpoint derivatives, per-restart iterate coefficient convergence data, decay, derivative bounds, and half-step spectral agreement.

### Field 2: `HasTimeNeighborhoodSpectralAgreement D.T D.u`

`ShenWork/Paper2/IntervalPicardLimitTimeNhdLocalized.lean` has `Hu_of_restart_localized`, but its statement includes:

```lean
(hχ0 : p.χ₀ = 0)
```

and a large localized ledger: `hfix`, source L1 continuity, localized cosine reps/summability, positivity/sup bounds, per-compact gradient/Hessian bounds, unshifted K1 data, and source continuity. That is not an unconditional strict-negative producer.

### Field 3: `HasResolverDirectSpectralData D.T (mildChemicalConcentration p D.u) p`

`IntervalMildRegularityFrontierAssembly.lean` proves a packaging theorem from a `DuhamelSourceTimeC1` resolver-source coefficient package, but that package is not produced unconditionally. The source-side route eventually runs into the power-source K1 / decay / source witness ledger. The chi-zero no-K1 shortcuts are not typed for strict negative.

### Field 4: `hpde_u`

The pointwise PDE identity is also not stored in `GradientMildSolutionData`; it is produced in the source/spectral frontiers after the relevant restart/source/regularity data are available. There is no committed unconditional theorem deriving this `hpde_u` from `D.u = picardLimit p u₀ D.T` alone in the strict-negative setting.

Therefore the direct headline is currently impossible without a major producer residual. The precise existing residual is:

```lean
∀ (u₀ : intervalDomainPoint → ℝ),
  PositiveInitialDatum intervalDomain u₀ →
∀ (D : GradientMildSolutionData p u₀),
  D.u = picardLimit p u₀ D.T →
    PPIDThresholdReachability.PerDatumRestartCoreSpectralFrontier p D
```

Equivalently, one may target `ConeQuantBridge.PicardLimitRestartFrontier p` directly, but the per-datum restart/core frontier above is the already-existing decomposed residual.

Since wrappers/aliases/surfaces are forbidden, the first useful theorem should be a producer for the first missing field in the existing core package: `GradientMildHalfStepLogisticSourceData D`. The committed infrastructure already turns `PicardIterateConvergenceData D` into that field, so the first real theorem should construct `PicardIterateConvergenceData D` for canonical Picard limits.

## Why not target the full PLF theorem first?

A theorem like:

```lean

    (p : CM2Params) :
    ConeQuantBridge.PicardLimitRestartFrontier p
```

or even a strict-negative version:

```lean
theorem picardLimitRestartFrontier_chiNeg
    (p : CM2Params) (hχ : p.χ₀ < 0)
    (hα : 1 ≤ p.α) (hγ : 1 ≤ p.γ) :
```

would immediately require solving all four producer families above. There is no committed strict-negative source of `Hu`, `HasResolverDirectSpectralData`, `hpde_u`, or `PicardIterateConvergenceData` from `hD` alone. Starting with the full PLF theorem would hide the real analytic blockers in one giant proof.

## Bridges to avoid

Avoid proving or using:

```lean
PositiveInitialDatum intervalDomain u₀ → PaperPositiveInitialDatum intervalDomain u₀
```

That implication is false on the interval.

Avoid all-PPID common-floor/Wiener routes; the repo already has obstructions such as:

```lean
ShenWork.EWA.not_datumWienerData
ShenWork.EWA.not_uniformFlooredC3NeumannData
```

Avoid using chi-zero producers for strict negative. In particular, `Hu_of_restart_localized` and `ResolverPowerK1.powerK1_quadruple_of_subtypeCont` carry `hχ0 : p.χ₀ = 0`; they cannot discharge strict-negative PLF fields.

Avoid adding more public aliases around the already-existing consumers. The routing layer is saturated; the missing work is producer-level.
