## 1. Verdict

**Not directly usable.**

The new C1/source-IBP package in `IntervalSemigroupC1ApproxIdentity.lean` is sound for the **full Neumann semigroup**

```lean
intervalFullSemigroupOperator
```

and for the full-kernel homogeneous initial leg

```lean

```

But `IntervalCoupledClassicalBallEstimates.lean`'s current C1 hmap chain is built around the existing coupled Duhamel scaffold

```lean

```

which expands through `intervalFullDuhamelOperator`, and that operator is defined using

```lean

```

not `intervalFullSemigroupOperator`.

The relevant definition in `IntervalDomainExistence.lean` is:


## 3. Is there a bridge already?

No equivalent bridge currently exists.

The bridge that would be needed to use the new full-kernel C1 package on the current hmap chain would be an operator-level equivalence, at least:

```lean
-- This is the kind of bridge one would need, but it is not available
-- and should not be assumed.
theorem intervalSemigroupOperator_eq_intervalFullSemigroupOperator_on_Icc
    {t : ℝ} (ht : 0 < t) {f : ℝ → ℝ} {x : ℝ}
    (hx : x ∈ Set.Icc (0 : ℝ) 1) :
    intervalSemigroupOperator 1 t f x =
      intervalFullSemigroupOperator t f x
```

and for hmap one would also need the corresponding Duhamel and derivative forms:

```lean
intervalCoupledDuhamelOperator p R u₀ u t x
  = intervalFullKernelCoupledDuhamelOperator p R u₀ u t x

and

deriv (intervalDomainLift (fun y => intervalCoupledDuhamelOperator ... y)) x
  = deriv (intervalDomainLift (fun y => intervalFullKernelCoupledDuhamelOperator ... y)) x
```

But this bridge is not merely absent; the raw equality is not mathematically faithful for the present definitions.  The helper kernel is a normalized two-term reflected kernel restricted to `[0,1]`; the full kernel is the full periodized Neumann kernel.  These are different operators.

The repo already documents this mismatch in `IntervalFullKernelDuhamelGradEq.lean`:

```lean
-- file comment summary
-- `hGradEq` is FALSE at `x = 1` when the Duhamel operator is built on the
-- zeroth-reflection semigroup `intervalSemigroupOperator` (Neumann at `0` only).
-- Resolution: rebuild on the full Neumann kernel.
```

That file defines the full-kernel replacement:

```lean
noncomputable def intervalFullKernelCoupledDuhamelOperator (p : CM2Params)
    (R : (intervalDomainPoint → ℝ) → intervalDomainPoint → ℝ)
    (u₀ : intervalDomainPoint → ℝ) (u : ℝ → intervalDomainPoint → ℝ)
    (t : ℝ) (x : intervalDomainPoint) : ℝ :=
  intervalFullSemigroupOperator t (intervalDomainLift u₀) x.1
    + ∫ s in Set.Icc 0 t, intervalFullSemigroupOperator (t - s)
        (intervalDomainLift (intervalCoupledSource p (u s) (R (u s)))) x.1
```

and proves:

```lean
theorem intervalFullKernel_hGradEq
    {p : CM2Params} {R : (intervalDomainPoint → ℝ) → intervalDomainPoint → ℝ}
    {u₀ : intervalDomainPoint → ℝ} {u : ℝ → intervalDomainPoint → ℝ} {T : ℝ}
    (τ x : ℝ) (hτ : τ ∈ Set.Ioo (0 : ℝ) T) (hx : x ∈ Set.Icc (0 : ℝ) 1) :
    deriv
      (intervalDomainLift
        (fun y : intervalDomainPoint =>
          intervalFullKernelCoupledDuhamelOperator p R u₀ u τ y)) x =
    deriv (fun z : ℝ =>
      intervalFullSemigroupOperator τ (intervalDomainLift u₀) z +
      ∫ s in (0 : ℝ)..τ,
        intervalFullSemigroupOperator (τ - s)
          (intervalDomainLift (intervalCoupledSource p (u s) (R (u s)))) z) x
```

This is the faithful full-kernel path.  It is not a bridge back to the old helper operator.

## 6. Circularity and faithfulness warning

A naive identification of

```lean
intervalSemigroupOperator 1
```

with

```lean
intervalFullSemigroupOperator
```

would be a faithfulness bug.  It changes the semigroup in the fixed-point map.  The helper operator is a normalized zeroth-reflection kernel; the full operator is the full periodized Neumann heat kernel.  The repo already records that endpoint derivative behavior differs at `x = 1`.  Any proof that treats them as the same without an explicit theorem would silently replace the PDE scaffold and could make a false Neumann boundary argument look valid.

Bottom line: the new package is sound, but for the full-kernel route.  It does not reduce the current helper-operator `hmap` chain unless the hmap chain is refactored to `intervalFullKernelCoupledDuhamelOperator` or a new, correct helper-operator C1 package is proved separately.
